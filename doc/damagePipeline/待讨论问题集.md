# Part C: 保留观察 🟡

## C1. 双层条件树（Predicate + Condition）的复杂度

**现状**：实现层是 `UDamagePredicate`（AND/OR/Single + bReverse）+ `UDamageCondition`（原子）双层。

**观察点**：
- 双层结构需要 `DamagePredicate_Single` 包装每个原子 Condition，稍显啰嗦
- 模型规范历史描述是单层（`UDamageConditionNode` 同时承担 AND/OR 和叶子）
- 实现层选择双层是因为参照了 AIToken 插件的模式，职责更清晰

**决策**：保留双层，不回滚。v4.6 → v4.7 演进期已将此决策反哺回模型规范——双层方案是 v4.7 的官方形态。详见 [`_archive/已解决决策_v4.3-v4.7.md#D3`](./_archive/已解决决策_v4.3-v4.7.md)。

---

*（C3 UDamageContext 命名问题随 A1 消解一起解决——保持 `UDamageContext` 名字不变。理由：既然不引入 `DamageEvent` 概念，"Context" 作为"Game 用来装任意 Effect 的上下文容器"语义准确，不需要改为 `UDamageData` 或其他名字。详见 [`_archive/已解决决策_v4.3-v4.7.md#D14`](./_archive/已解决决策_v4.3-v4.7.md)）*
