## 一、Condition 体系 🔴

Condition 是当前待讨论问题最集中的主题，涉及本质边界、实现方案、Usability 三个层面。

### 1.1 表达力边界（本质复杂度问题）

**本质问题**：Condition 的表达力边界应该划在哪里？

这不是"支不支持比较运算符"的偶然复杂度问题，而是 DPU 和 Condition 职责分界的本质问题：
- Condition 只做布尔门控 → 数值比较是 DPU 内部事务 → Condition 复杂度被严格控制，但 DPU 粒度可能被迫变粗
- Condition 支持数值比较 → 某些 DPU 可以拆得更细 → 但 Condition 开始承担"计算"职责，边界模糊
- 极端情况：Condition 越来越强 → 变成"另一个 DPU" → R4（装配关系）的意义被稀释

**E/A 检验**：删掉边界约束 → Condition 无限膨胀 → R4 失去意义 → DPU 自洽性被破坏。换任何实现都要回答"门控能做多少事"。**本质复杂度。**

#### 候选方案对比

本质上都在回答同一个问题："如何安全地引用一个动态命名空间中的标识符，并以此驱动执行门控和依赖推导"。

| # | 方案 | UE 先例 | 优点 | 缺点 | 是否驱动 DAG |
|---|------|---------|------|------|-------------|
| **A** | **GameplayTag 两层门控** ⭐ | GAS RequiredTags/BlockedTags + CanActivateAbility | Tag 预注册，编辑器下拉不可能拼错；GAS 已验证的成熟模式；声明式/程序式职责分界清晰；自然驱动 DAG 推导 | 所有产销标识都必须预注册为 Tag；纯 AND 逻辑（无 OR），复杂条件需拆成多个 Tag 或下沉到 CanActive | Tag 门控：是；CanActive：否 |
| **B** | **FBlackboardKeySelector 模式** | 行为树 Blackboard Key 选择器 | 从当前 PipelineAsset 已注册的 Produces 中选，上下文感知；编辑期即报错 | 需要自定义 Editor 控件；Selector 绑定的是 PipelineAsset 而非全局注册表，切换 Asset 后需要重新校验 | 可以，通过 Selector 引用的 Key 推导 |
| **C** | **StateTree Condition 对象** | StateTree 条件/任务结构体 | 每个 Condition 类型是独立 UObject，有自己的 UPROPERTY，编辑器表单化；类型安全 | 每种条件模式都需要写一个类；组合逻辑（AND/OR/NOT）需要额外的组合器类 | 可以，通过 Condition 对象显式声明 Consumes |


**当前倾向：方案 A（GameplayTag 两层门控）**。理由：实现成本最低（UE 已有 Tag 基础设施），GAS 已验证模式可靠，声明式/程序式分层清晰回答了表达力边界问题。

### 1.2 方案 A 详细设计：两层 Condition（参照 GAS GameplayAbility）

**GAS 同构分析**：

GAS 的 GameplayAbility 用两层门控解决了同类问题：
- **RequiredTags / BlockedTags**：纯 Tag 存在性检查，声明式配置
- **CanActivateAbility()**：C++/蓝图 override，处理数值判断等个性化逻辑

GAS 的实践证明：**纯 Tag 存在性检查（RequiredTags 是纯 AND，BlockedTags 也是纯 AND，连 OR 都没有）已经足够支撑一个完整的 Ability 系统。**

**核心思路：Tag 取代 Produces 中的 FName，成为 DPU 间产销关系的连接件**

- **Produces 声明 = 这个 DPU 产出哪些 Tag**
- **Tag 门控 = 这个 DPU 依赖哪些 Tag（Require / Block）**
- 系统从这两个声明自动匹配：Tag X 的生产者 DPU_A → Tag X 的消费者 DPU_B → A 在 B 之前
- DC 内部的存储方式（Key 用什么类型）是实现层的事，和产销关系建模无关

**两层映射到 DPU 框架**：

| 层 | 职责 | 参与 DAG 排序 | 表达方式 | 出错可能性 |
|---|------|-------------|---------|-----------|
| **Tag 门控（声明式）** | 声明 DPU 间的产销依赖 | **是** | 编辑器下拉选 Tag | 几乎为零 |
| **CanActive（程序式）🔴** | 个性化的执行前置判断 | **当前：否** | C++ / 蓝图 override | 由程序员负责，CanActive函数的归属问题🔴 |

求值顺序：先 Tag 门控（快，纯查表），再 CanActive（可能涉及计算）。任意一层返回 false，DPU 跳过。

**只狼示例**：

- DPU_Mixup：Produces = `[Guard]`，无 Tag 门控，无 CanActive → 永远执行，产出 Guard Tag
- DPU_Guard：Produces = `[GuardSuccess, IsJustGuard]`，RequireTag = `Guard`，BlockTag = `IsLightningInAir` → 系统自动推导：Mixup 在 Guard 之前，LightningInAir 在 Guard 之前
- DPU_Poison：BlockTag = `IsJustGuard` → 系统自动推导：Guard 在 Poison 之前。CanActive：`if (DC->GetFloat("poizonAttackPower") == 0) return false` → 读的是 DC 初始字段（攻击方属性），不产生 DPU 间依赖

### 1.3 Usability：字符串表达式的安全性问题

**现状**：StringConditionExpression 以纯字符串编写（如 `"Guard && !IsLightningInAir"`）。字段名拼错 → R3 缺失=false → 静默失败。没有自动补全、没有编辑期校验。

**此问题被 1.2 的两层方案自然解决**：Tag 门控层用编辑器下拉选 Tag，拼写错误从源头消除。StringConditionExpression 降级为 C++ 调试/高级用途。

### 1.4 待定子问题：CanActive 是否参与 DAG 推导？

如果 CanActive 里读了某个 DPU 的产出但系统不知道，排序可能把它排到那个 DPU 前面。

当前倾向：**不参与**。理由是 CanActive 读的通常是 DC 初始字段（无 DPU 生产者），不涉及 DPU 间依赖。但如果实践中发现 CanActive 确实需要读其他 DPU 的产出，则需要重新审视。

## 二、Produces 独占性 🔴

**本质复杂度问题**：独占性是静态约束还是运行时约束？

**现状**：模型规范要求"同一个 DC 字段只能有一个生产者 DPU"。但只狼管线中：

```
DPU_Collapse      produces: [IsCollapse]
DPU_CollapseGuard produces: [IsCollapse]  ← 同一字段，两个生产者
```

两者 Condition 互斥，运行时不会冲突。但静态声明层面违反了独占性。

**需要决策**：
- 放松为"运行时独占"（同一次执行中只会有一个实际写入）→ 需要 Condition 互斥保证，但模型层没有声明互斥的机制
- 引入"互斥组"概念 → 两个 DPU 显式声明互斥关系
- 维持静态独占 → 需要修改只狼管线设计（拆分字段名）

**E/A 检验**：删掉独占性约束 → 同一字段多个生产者 → 拓扑排序出现歧义 → DAG 不唯一 → 表达能力受损。从 DAG 数学性质长出来。**本质复杂度。**

## 三、DC 缺失语义精度 🟡

**本质复杂度问题**：R3 在数值字段上的边界定义。

Bool 缺失→false 语义清晰。Float/Int 缺失→0，无法区分"值为零"和"字段不存在"。

```cpp
float HP = DC->GetFloat("CurrentHP");  // 缺失返回 0
// HP=0 和字段不存在，结果一样
```

**当前判断**：R3 的"缺失视为 false"暗示 DPU 不应关心字段是否存在。`Contains()` 方法可用于需要区分的场景。当前测试未触发此问题，但值得在规范中显式说明边界。

**E/A 检验**：R3 是本质约束，"缺失=0.0"是 R3 在数值类型上的延伸。删掉 → 数值字段语义不可预测。**本质复杂度。**

## 四、Pipeline 执行架构 🔴

以下问题均围绕 PipelineManager 和 PipelineAsset 的职责划分，解决核心问题后其余自然消解。

### 4.1 统一 PipelineAsset 为唯一执行入口

**现状**：`UPipelineManager` 暴露两条路径（`ExecutePipeline(TArray<DPU*>)` 和 `ExecuteFromAsset(PipelineAsset)`），代码允许绕过 Asset 直接执行。DPUDefinition 可以以裸数组形式消费。

**问题**：设计文档明确说"PipelineAsset 是唯一数据交换格式"，但代码违反了这条约束。使用者心智模型变成"一堆 Definition 给 Manager 执行"，而非"Pipeline 自己知道怎么执行"。

**讨论结论**：统一为 PipelineAsset 作为唯一入口。区别只在 Asset 来源（编辑器预制 vs 运行时动态组装），不在调用路径。DPUDefinition 始终通过 PipelineAsset 消费。

### 4.2 拓扑排序烘焙/缓存

**现状**：每次执行都重新跑拓扑排序，即使 DPU 集合没变。

**讨论结论**：
- PipelineAsset 持有烘焙后的 `SortedDPUs`
- 提供 `Build()` 方法触发排序并缓存
- 编辑器下 Build 可序列化保存；运行时 Build 缓存在内存
- `Execute(DC)` 直接按缓存顺序执行，不再每次重排
- DPU 集合未变时，重复 Execute 复用缓存

### 4.3 PipelineAsset 验证覆盖

**现状**：TestActor 直接创建 DPU 数组绕过了 Asset，"PipelineAsset 是唯一数据交换格式"未被验证。

**建议**：随 4.1 重构一起解决——TestActor 改为通过 PipelineAsset 执行。

**E/A 检验**：4.1/4.2/4.3 全部是实现层的架构选择。删掉 PipelineManager 换成 PipelineAsset 直接执行 → 功能完全不变。**偶然复杂度。**

## 五、未实现的模块 🔴

### 5.1 表现选取阶段

模型规范已完整定义 Channel、Priority、CombinedPresentation 和选取算法。代码中 `DPUDefinition.BasePriority` 声明了但未使用，没有 Channel 数据结构，没有选取逻辑。**当前最大的模型-代码差距。**

建议：Phase 1.5 补上。

### 5.2 运行时 Produces 验证

DPU 可以写入未声明的字段，代码完全没有检查。影响：拓扑排序基于错误的依赖关系，Mermaid DAG 看不到真实数据流。

建议：至少在 Debug/Editor 模式下加验证——执行前后 diff DC，检查 DPU 是否写了未声明的字段。

**E/A 检验**：两者都是工作量/防御性编程问题，不影响模型的表达能力。**偶然复杂度。**

## 六、有倾向但待讨论确认的问题 🟡

### 6.1 命名：DPU / DC / Condition

**当前倾向**：

**DPU / DC**：D = Damage 没问题。框架处理的问题域就是伤害管线，命名体现聚焦于 Damage 是正确的。

**Condition**：不需要改。够准确、够中性、不和问题域冲突（Guard 在只狼中是格挡机制，如果 Condition 也叫 Guard 就冲突）。如果要改，Gate 是最好的替代。

**行业对标**：

| 引擎/框架 | 概念 | 命名 |
|-----------|------|------|
| 行为树 | 条件门控 | Decorator / Conditional |
| 编程语言 | 模式匹配守卫 | **Guard** |
| 电路设计 | 使能端 | **Enable** |
| 管线架构 | 阀门 | **Valve** / Gate |
| 本框架 | 读DC输出bool | **Condition** |

**待讨论**：命名问题虽然不影响表达能力（偶然复杂度），但影响对外沟通效率和使用者的第一印象。需要在更多人接触框架后收集反馈再最终确认。

### 6.2 DPU 职责边界

**当前倾向**：不是问题。DPU 承载的是一个完整的机制表达——从判定到数值到表现（往哪个 Channel 输出反馈信息）都是这个机制自身的事。与模型文档定义一致。

**待讨论**：当框架进入正式生产期，表现信息和机制逻辑的变更节奏是否会产生实际矛盾（美术改动画方案是否需要打开 DPU 逻辑代码）。当前阶段无实证，保留观察。
