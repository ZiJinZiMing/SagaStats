## §1 领域原语

| 术语 | 类型 | 定义（一句话） |
|---|---|---|
| **DamageRule** | `UDamageRule`（DataAsset） | 一条伤害规则，由 Condition（生效判定）+ OperationClass（计算逻辑）构成 |
| **DamageOperation** | `UDamageOperationBase` | Rule 的计算单元，`Execute(Context, OutEffect&)` 填字段 |
| **DamageEffect** | `USTRUCT` 子类（如 `FGuardEffect`） | Rule 的结构化产出；也可以是无字段的"信号"类型 |
| **DamageCondition** | `UDamageCondition`（抽象基类） | 条件原子的公共契约；具体子类必须继承 `_Effect` 或 `_Context` 之一 |
| **DamageCondition_Effect** | `UDamageCondition_Effect` | 基于 Effect 的判定；绑定 `EffectType`（贡献 R5 产销依赖）；Evaluate 签名 `(Context, InEffect)` |
| **DamageCondition_Context** | `UDamageCondition_Context` | 基于 Context 的判定；无 EffectType（不贡献产销依赖）；Evaluate 签名 `(Context)`；**编译期禁止读任何 Effect** |
| **DamagePredicate** | `UDamagePredicate` | 条件组合容器：Single / AND / OR + `bReverse`；递归求值 |
| **DamageContext** | `UDamageContext` | 运行时共享上下文，存 `TMap<TObjectPtr<UScriptStruct>, FInstancedStruct>`；Effect 读写 API 为 protected |
| **DamagePipeline** | `UDamagePipeline`（DataAsset） | Rule 列表 + Build（拓扑排序）+ Execute（依序执行） |
| **DamagePipelineResults** | `UBlueprintFunctionLibrary` | Game 侧读写 DC 的唯一外部 API 入口（`WriteEffect` / `ReadEffect` / `HasEffect` / `GetAllEffects`） |
| ~~DamageContextLibrary~~ | （v4.7 已删除） | 旧的 v4.6 通用蓝图读写桥，是绕过 R5 的漏洞来源；v4.7 删除；Game 侧改用 `UDamagePipelineResults` |

## §2 语法

| 术语 | 定义 |
|---|---|
| **Produces 声明** | Rule 通过 `OperationClass` 的 CDO 反查 `GetProducesEffectType()` —— Operation 子类声明自己产出的 Effect 类型 |
| **ConsumedEffectType** | Condition 子类在类级别声明消费哪种 Effect，通过 `EffectType` 字段暴露；`UDamagePredicate::GetDependencyEffectTypes()` 递归收集所有叶子 Condition 的类型 |
| **Condition 树** | Predicate 容器 + Condition 叶子的递归树形结构，支持 `Single / AND / OR + NOT` |
| **OperationClass 引用** | Rule 通过 `TSubclassOf<UDamageOperationBase>` 引用 Operation 类，运行时框架 NewObject |
| **bReverse** | 只在 `UDamagePredicate` 上，不在 `UDamageCondition` 上——取反逻辑统一在容器层 |

## §3 语义（速查）

### 四条设计思想

只列名称，详细展开在 [`README.md`](./README.md) §5：

- **机制优先**：体验由机制驱动，不是由数值驱动
- **数值兜底**：数值不驱动体验，但保证数学基础正确
- **反馈清晰可感**：每个机制结果必须被玩家感知到
- **涌现自治**：多机制组合由 Condition 树在 DC 上自然产生

### R1-R6 本质约束

只列词条，详细解释在 [`架构设计.md`](./架构设计.md) §5.2，速查卡片在 [`快速入门.md`](./快速入门.md) §3：

- **R1**：DC 是唯一通信介质
- **R2**：Operation 自洽（Execute 纯函数）
- **R3**：Condition 可读 DC 全部内容，Effect 缺失视为 false
- **R4**：Condition 与 Rule 是装配关系
- **R5**：执行顺序由 EffectType 产销关系自动决定
- **R6**：禁止循环依赖

### Essential vs Accidental Complexity

| 类型 | 定义 | 信号 |
|---|---|---|
| **Essential**（本质复杂度） | 问题域固有，无法消除 | 删掉它，丢功能 |
| **Accidental**（偶然复杂度） | 表达方式引入的额外复杂度 | 换种方式表达，它就消失了 |

**三步检验程序**：

| 步骤 | 问题 | 判断 |
|---|---|---|
| 1. 删除测试 | "删掉它，丢什么？" | 丢了 = 本质，没丢 = 偶然 |
| 2. 来源追溯 | "长出来的还是借来的？" | 借来的要额外验证 |
| 3. 换表达验证 | "换种方式表达，它还在吗？" | 还在 = 本质，不在 = 偶然 |

## §4 类型系统

| 术语 | 定义 |
|---|---|
| **UScriptStruct\*** | DSL 的类型标识——作为 DC 的 key，作为 Pin 的 `PinSubCategoryObject`，作为 Condition/Operation 声明自己处理哪种 Effect 的锚点 |
| **FInstancedStruct** | UE 的多态结构体容器，DC 用它承载任意 Effect 实例 |
| **PinSubCategoryObject** | Graph 编辑器 Pin 的类型字段，存 `UScriptStruct*`；配色 / 通道分配 / 连线匹配都依赖此字段 |
| **类型即 Key** | 不存在独立的 "Effect Key" 概念——`DamageEffects[FXxxEffect::StaticStruct()]` 直接用类型查 |

## §5 执行引擎

| 术语 | 定义 | 代码锚点 |
|---|---|---|
| **Build** | `UDamagePipeline::Build()` — 拓扑排序 + 产销校验 + 缓存 SortedRules | `DamagePipeline.cpp:204` |
| **Kahn BFS** | 稳定拓扑排序算法：提取入度 0 的节点进队，出队后移除下游入边 | `DamagePipeline.cpp:204` 内部 |
| **StableTopologicalSort** | Pipeline 私有实现方法；BFS 队列按原始索引稳定排序，保证同入度节点的执行顺序确定 | 同上 |
| **SortIndex** | 每条 Rule 在拓扑序里的序号（0-based）；Graph Editor 显示为 `1. RuleName` | `DamagePipelineGraphNode.h` |
| **bIsBaked** | Pipeline 的烘焙标志；为 true 表示 SortedRules 有效可直接用 | `DamagePipeline.h` |
| **Execute** | `UDamagePipeline::Execute(Context)` — 按 SortedRules 顺序遍历：Predicate 求值 → 预创建 OutEffect → Operation 填字段 → 写 DC | `DamagePipeline.cpp:303-366` |
| **FRuleExecutionEntry** | Execute 返回的日志条目：RuleName + 是否命中 + EffectType | `DamagePipeline.h` |

## §7 验证机制

| 术语 | 定义 |
|---|---|
| **拓扑防环** | Build 用 Kahn 算法，若 `Sorted.Num() < Rules.Num()` 则 Pipeline 有环，Build 失败 |
| **EffectType 校验** | Build 前置检查：每条 Rule 的 Operation CDO 必须声明 `GetProducesEffectType()` 非空 |
| **缺失容错** | Condition 消费的 Effect 在 DC 里不存在时，`GetEffect<T>()` 返回 nullptr，Condition 自行决定语义（默认 false） |

## §8 L1-L4 工具成熟度阶梯

引自 `全量资料/L3工具的本质_领域DSL_认知成长记录.md`：

| 级别 | 特征 | 典型表现 |
|---|---|---|
| **L0 硬编码** | 所有逻辑写在 C++ 中 | 修改需重编译 |
| **L1 数据驱动** | 配置文件/表格控制参数 | JSON/XML 配置 |
| **L2 可视化编辑** | 节点图 / GUI 编辑器 | Blueprint、行为树编辑器 |
| **L3 领域 DSL**（DamagePipeline 的位置） | 完整语义的专用语言，领域原语 + 类型安全 + 验证 + 组合 + 错误检查 | 本项目 |
| **L4 自动化生态** | DSL + 代码生成 + CI/CD | 完整工具流水线 |

**L3 vs L2 的关键区别**：L2 是"功能包装"——把 C++ 代码包装成节点；L3 是"领域建模"——抽象出领域原语，使用者用领域语言思考。

## §9 已废弃术语对照

| 已废弃 | 当前 v4.7 | 废弃原因 |
|---|---|---|
| DPU | DamageRule | 工具层身份（DPU = Damage Process Unit）替代内容域；v4.5 → v4.6 重构 |
| Fact | DamageEffect | 同上，强调"规则的结构化产出" |
| Logic | DamageOperation | 同上，强调计算语义 |
| FMixupResult | FMixupEffect | 对齐 Effect 命名 |
| UPipelineAsset | UDamagePipeline | 同上 |
| UDPUDefinition | UDamageRule | 同上 |
| UDamageRuleDefinition | UDamageRule | 同上 |
| 固定管线 / 动态管线 | 无官方术语 | 独创分类，不在模型规范里；用法上是"静态声明的 Produces 集合 / 运行时 Build 后重复 Execute" |
| Pipeline Manager | 无（DamagePipeline 自洽） | 外部 Manager 被 `DamagePipeline::Build/Execute` 内聚替代 |
| FactMethodRegistry | 子类即域方法 | 全局注册表被子类化替代，编译时类型安全 |
| UDamageConditionNode | UDamageCondition（原子）+ UDamagePredicate（容器） | 单层 → 双层分离，职责更清晰 |
| FactKey / Effect Key | UScriptStruct\* 本身 | 类型即 key，不需要额外 Key 抽象 |
| MethodName 下拉 | Condition 子类 | 子类支持实例参数、类型安全、蓝图可扩展 |
| RuleName / PipelineName 字段 | 资产 `GetFName()` / `GetName()` | 冗余字段移除；`FRuleExecutionEntry::RuleName` 作为日志接口保留 |
