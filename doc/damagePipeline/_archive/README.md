## 为什么归档

`doc/damagePipeline/待讨论问题集.md` 在 v4.3 → v4.7 演进期承担了两种角色：

1. **活跃讨论区**（Part A/B/C）——当前未决的模型级问题和待实施项
2. **已解决决策日志**（Part D）——13 条已落地为 v4.7 规范的历史决策

两种内容混在一份文档里带来三个问题：

- Part D 占 60% 篇幅，淹没了真正需要对齐的活跃讨论
- Part D 多处引用已被删除的 `全量资料/模块化伤害管线_实现层变更记录_v4.6.md`——坏链
- "待讨论"名字与"已解决"内容语义冲突

v4.7 终态下，把这两种内容分开：活跃讨论留在主目录，历史决策归档到此。

## 归档原则

进入归档的条件（同时满足）：

1. 决策已落地为当前 v4.7 代码或规范的一部分
2. 决策的动机/取舍/反方案在当前文档中没有完整描述
3. 未来新人/AI 协作者可能追问"为什么是这个设计"

不满足的不归档（直接删除即可）：

- 纯编码细节（commit 信息已记录）
- 已被后续决策完全覆盖的中间版本（只留最终形态）
- 可以从当前代码/文档 100% 推导出动机的显而易见的选择
