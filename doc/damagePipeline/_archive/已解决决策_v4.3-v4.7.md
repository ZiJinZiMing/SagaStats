# 归档内容

## D1. ✅ DPU → DamageRule 术语重构

**决策**：代码完成 v4.5 → v4.6 的术语全面重构。DPU/Fact/Logic/PipelineAsset → DamageRule/DamageEffect/DamageOperation/DamagePipeline。CoreRedirects 已完整覆盖。

## D2. ✅ Execute 签名改为 OutEffect 直传

**决策**：`Execute(DC, OutEffect&)`，框架预创建 OutEffect 传入，Operation 只填字段，框架负责写 DC。R2 自洽性恢复（Operation 不需要知道自己的身份标识）。

## D3. ✅ Predicate/Condition 双层分离

**决策**：参照 AIToken 模式，职责分离。Predicate 负责组合逻辑（AND/OR + bReverse），Condition 负责原子检查。

## D4. ✅ UScriptStruct\* 作为 DC 键

**决策**：类型即 key，编译时类型安全，DamageRule:Effect 1:1 关系。

## D5. ✅ 移除 EventContext/标量 API/ContextCheck

**决策**：`SetBool/GetFloat/SetInt` 等散碎接口移除，攻击上下文改用 typed struct + SetEffectByType。`UDamageCondition_ContextCheck` 删除，替换为具体的 Condition 子类（如 `UDamageCondition_IsLightning`、`UDamageCondition_IsInAir`）。

## D6. ✅ 移除 RuleName / PipelineName / Description 冗余字段

**决策**：DamageRule 和 DamagePipeline 的身份直接使用资产/对象的 `GetFName()`/`GetName()`，不再存冗余字段。`FRuleExecutionEntry::RuleName` 保留作为日志接口。

## D7. ✅ DamagePipeline Graph Editor（只读 DAG 可视化）

**决策**：`FDamagePipelineAssetEditor` 提供双面板编辑器（Details + Graph），Build 按钮在 Toolbar。Graph 是 Build 后的只读可视化，参照 AIGraph 模式，节点从左到右按拓扑层级自动布局（与 Mermaid `graph LR` 一致）。

## D8. ✅ 蓝图 CustomThunk API 独立为 Library

**决策**：`UDamageContextLibrary` 作为 `UBlueprintFunctionLibrary`，提供 `SetEffect`/`GetEffect` 通配结构体接口。**不能**在 UDamageContext 成员函数上直接使用 CustomThunk + CustomStructureParam，会因缺少 K2Node 配套导致 Stack 解析错乱崩溃。

## D9. ✅ FactMethodRegistry 删除

**决策**：实现层选择"子类即域方法"后，全局注册表成为偶然复杂度，整套删除。

## D10. ✅ 独立 Fact Key 概念删除

**决策**：经三步检验确认为偶然复杂度。现以 UScriptStruct\* 为 DC 索引键。

## D11. ✅ DamageEffect 缺失语义的精度（消解 A2）

**原问题**：R3 规定"Effect 缺失视为 false"，数值型字段上"缺失"和"零/默认"无法区分，是否需要更精细的约定？

**消解理由**（2026-04-13）：

R3 只是"默认行为约定"，不是"强制规则"。Condition 子类有完整的表达能力：

1. **类型级缺失可精确感知**：
   - `TMap<UScriptStruct*, FInstancedStruct>` 的 key 是类型本身，不是内存布局
   - 即使 FStructA 和 FStructB 字段完全相同，它们有不同的 UScriptStruct\*，DC 查找可正确区分
   - `GetEffect<T>()` 返回 nullptr 或 `ConsumedEffect.IsValid() == false` 都是明确信号
   - 蓝图 CustomThunk `execGetEffect` 走 Valid/NotValid 执行引脚，双重保险

2. **Condition 作者保有语义决定权**：
   ```cpp
   bool UDamageCondition_HasZeusBonus::Evaluate_Implementation(
       const UDamageContext* Context, const FInstancedStruct& ConsumedEffect) const
   {
       const FZeusBoonContext* Zeus = ConsumedEffect.GetPtr<FZeusBoonContext>();
       if (!Zeus) return false; // 或 true，由 Condition 作者决定
       return Zeus->Level >= 3;
   }
   ```

3. **字段级缺失是 Operation 的约定责任**：
   - 如果 Operation 作者需要区分"值为零"和"未设置"，让结构体字段带 `bHasValue` 标志位或用 `TOptional<T>`
   - 这是 Operation 内部设计问题，不是管线模型问题

**结论**：问题域层面不存在 A2，只有 Condition 作者的实现选择。模型规范不需要额外约定，只需说明"R3 是默认约定，Condition 子类可自定义"。

## D12. ✅ 子类即域方法 vs MethodName 下拉（消解 C2）

**原担忧**：子类方式在 Hades 级复杂管线下可能"爆炸"。

**消解理由**（2026-04-13）：

**担忧本身是错的**：
- N 个域方法 = N 个下拉项 = N 个子类，数量线性相等
- 子类数量爆炸是把"类的数量"和"文件数量"混淆了
- 每个子类是域方法的壳子（几行代码），不是传统 OOP 的"一个类"

**子类方式实际上比 MethodName 下拉更优**：

1. **能配置实例参数**（MethodName 做不到）
   ```cpp
   UCLASS(meta=(DisplayName="HasZeusBonusLevel"))
   class UDamageCondition_HasZeusBonusLevel : public UDamageCondition
   {
       UPROPERTY(EditAnywhere) int32 MinLevel = 3;       // 实例参数
       UPROPERTY(EditAnywhere) EZeusSkill RequiredSkill; // 多个参数自由组合
   };
   ```
   MethodName 下拉做不到实例参数化——要么硬编码成 N 个方法，要么引入通用参数机制（JSON/反射，偶然复杂度爆炸）。

2. **类型安全**
   - 子类的 `ConsumedEffectType` 在类级别硬编码（`HideInDetailPanel`）
   - Evaluate 内直接 cast 到具体类型，编译器检查
   - 重命名子类编译器报错；重命名 MethodName 字符串运行时静默失效

3. **蓝图可扩展性**
   - 蓝图用户可通过创建 UDamageCondition 子类（蓝图）扩展新域方法
   - MethodName 下拉需要全局注册表（像之前删除的 FactMethodRegistry）

4. **发现性**
   - Class Defaults 面板直接看到参数
   - MethodName 需要文档说明方法含义和前提

**结论**：子类即域方法不是"实现层的妥协"，而是"实现层优于模型层"的改进。此项应反哺回模型规范。

## D13. ✅ DamageOperation BP 通配返回值不可行（接受标准范式）

**问题**：DamageOperation 的 BlueprintNativeEvent `Execute(Context, OutEffect&)` 在 Blueprint 实现时，用户必须用 `Set Instanced Struct Value(OutEffect, MyStruct)` 把具体结构体塞进 OutEffect。能否消掉这个中间节点，让 BP 直接"返回"具体结构体由框架自动包装？

**调研结论**（2026-04-13）：**标准引擎机制无法做到**。

**根本原因**：

`CustomStructureParam` + `CustomThunk` 是引擎支持通配结构体引脚的唯一机制，但**只处理"被调用的函数"**，不处理"被重写的事件函数"：

| 场景 | 节点类 | 是否处理 CustomStructureParam |
|---|---|---|
| 调用静态库函数（如 `MakeInstancedStruct`） | `UK2Node_CallFunction` | ✅ 通过 `FCustomStructureParamHelper::UpdateCustomStructurePins` |
| 重写 BlueprintNativeEvent 函数体 | `UK2Node_Event::CreatePinsForFunctionEntryExit` | ❌ 不调用 Helper，忽略 meta |

即使在 BlueprintNativeEvent 参数上打 `CustomStructureParam` 也无效——引擎根本不读这个 meta。

**引擎为什么这么设计**：

引擎刻意分离了两种职责：
- **被调用函数**（CallFunction）：支持动态类型绑定，因为调用点在 BP 图上，拓扑已知
- **被重写函数**（Event）：拒绝动态类型绑定，因为 override 的语义契约要求子类遵守父类签名

允许 BlueprintNativeEvent 参数类型被子类"改变"会破坏 LSP（Liskov 替换原则）——父类承诺调用 `Execute(Context, OutEffect)`，但子类声称它接受一个具体 struct，C++ 侧调用父类签名时不知道怎么构造参数。

`UK2Node_GetDataTableRow` 实现"通配返回值"的方式是：**它不是 override，是 call**。通配 K2Node 只在调用侧出现，不在被重写侧出现。

**唯一替代路径**：写自定义 K2Node（参考 `UK2Node_GetDataTableRow` 约 600 行编辑器代码），在 Blueprint 编译期注入 `Set Instanced Struct Value` 调用。

**决策**：保持现状，接受 "Make X → Set Instanced Struct Value" 标准范式。

**理由**：
- 这是 UE 任何写 `FInstancedStruct&` 输出参数的函数都用的模式，和引擎一致
- 只狼 13 个 + Hades 30-50 个 DamageRule 规模下，不值得为消减 1 个节点投入 600 行编辑器代码 + 后续维护
- K2Node API 在引擎主版本间可能变动，维护成本持续

**调研涉及的关键文件**（避免后续重复调研）：
- `Engine/Source/Runtime/Engine/Classes/Kismet/BlueprintInstancedStructLibrary.h` — Make/Set/Get InstancedStruct UFUNCTION 声明
- `Engine/Source/Runtime/Engine/Private/BlueprintInstancedStructLibrary.cpp` — 三个 CustomThunk `DEFINE_FUNCTION`
- `Engine/Source/Editor/BlueprintGraph/Private/K2Node_CallFunction.cpp:93-176` — `FCustomStructureParamHelper`
- `Engine/Source/Editor/BlueprintGraph/Private/K2Node_Event.cpp:347-370` — 事件节点 pin 创建（不经过 Helper）
- `Engine/Source/Editor/BlueprintGraph/Private/K2Node_GetDataTableRow.cpp` — 自定义 K2Node 通配返回值范本

## D14. ✅ 攻击上下文的概念定位（消解 A1）

**原问题**：在当前模型里，"被 DamageCondition 消费的 USTRUCT" 有两种语义——Rule 产出的 DamageEffect vs 调用方预填的攻击上下文。两者用同一个名字（DamageEffect）包装造成概念混淆，Build 校验无法区分"合法缺失（攻击上下文缺失视为 false）"和"非法缺失（漏配 Rule）"。

**消解理由**（2026-04-17）：

**这不是 DSL 的问题，是使用场景的问题**：

- DamagePipeline 作为领域 DSL，其职责是提供**可扩展的容器机制**
- "攻击上下文长什么样 / 装哪些字段 / 由谁填充"这些是**Game 项目的使用惯例**，不是 DSL 本体需要定义的概念
- `UDamageContext` 已加 `UCLASS(BlueprintType, Blueprintable)` 标记（DamageContext.h:24），Game 项目可以自由蓝图继承扩展

**"区分合法/非法缺失"的伪需求**：

- 这个需求假设 DSL 需要知道"哪些 Effect 是外部输入、哪些是 Rule 产出"——但 Fowler 的 DSL 设计原则里有一条反例："受限表达力是核心优势"
- DSL 不应该试图校验使用者自己的应用约定
- Game 项目如果需要对"必填的攻击上下文"做校验，这是 Game 侧的单元测试 / Actor 初始化检查的职责，不是 DSL Build 时的职责

**L3 DSL 的职责边界划分**：

| 层级 | 职责 |
|---|---|
| DamagePipeline DSL（L3 本体） | 提供 Condition/Predicate/Operation 原语、Build 拓扑、Execute 循环、DC 存储、缺失容错（R3） |
| Game 使用层（应用层） | 决定 DC 里装什么攻击上下文、何时填充、如何跟 Actor 状态对接、扩展 `UDamageContext` 子类添加便利方法 |

**"攻击上下文 vs Rule 产出"在代码里如何自然区分**：

- 由**使用惯例**区分——Game 代码在 `Pipeline->Execute(Ctx)` **之前** 调 `Ctx->SetEffect<FSekiroAttackContext>(...)` 预填攻击上下文
- `SortedRules` 中没有任何 Rule produces `FSekiroAttackContext` → 它天然是"外部输入"
- Condition 消费 `FSekiroAttackContext` 时，R3 缺失容错给出 false——符合直觉（没预填就是 false）

**结论**：
- "攻击上下文是 DSL 里的独立概念"这个判断是**过度设计**
- `UDamageContext` 加 `Blueprintable` 已经提供了足够的可扩展性
- 不引入 `DamageEvent` / `EventContext` / `RequiredEventContexts` 等额外概念——这些会把 DSL 的表达力从"受限领域语言"扩散到"通用应用框架"

## D15. ✅ Build 时合法/非法缺失区分（消解 B2）

**原问题**：Build() 阶段需要区分"合法缺失（攻击上下文）"和"非法缺失（漏配 Rule）"。

**消解理由**（2026-04-17，随 D14 一起消解）：

B2 的这部分需求建立在 D14 的伪需求之上——既然 D14 已确定"攻击上下文不是 DSL 概念"，Build 时就不需要也不应该区分这两种情况。

**B2 实际已完成的部分**：
- ✅ Rule 的 Operation 未配置 ProducesEffectType → Build 失败（DamagePipeline.cpp:204 内部）

**B2 不再需要做的部分**：

**结论**：B2 的实际交付已完成，不再有未实施项。
