## §2 五分钟跑通一个 DamageRule（最小示例）

我们做一条最小可运行的 Rule：**收到伤害 → 记录 "受到了 X 点伤害"**。不依赖其他 Rule，不产出给下游——最简单的形态。

### Step 1：定义 Effect 类型

```cpp
// MinimalHurtEffect.h
USTRUCT(BlueprintType)
struct FMinimalHurtEffect
{
    GENERATED_BODY()

    UPROPERTY(BlueprintReadOnly)
    float DamageAmount = 0.f;
};
```

**或蓝图方式**：在 Content Browser 右键 → Miscellaneous → **Damage Effect Struct**（本插件提供的自定义 UDS 工厂），命名 `DE_MinimalHurt`，加一个 Float 字段 `DamageAmount`。

### Step 2：创建 DamageOperation

Content Browser → 右键 → **Damage Pipeline / Damage Operation**（本插件的工厂菜单项）→ 父类 `UDamageOperationBase` → 命名 `BP_Op_MinimalHurt`。

**在 Class Defaults 中**：`Produces Effect Type` → 选 `DE_MinimalHurt`（或 C++ 的 `FMinimalHurtEffect`）

**重写 Execute 事件**：

```
Execute (Context, OutEffect)
  │
  ├─ 从调用方预填的 AttackContext 里读 DamageAmount
  │     Get Effect (Context, FSekiroAttackContext)
  │     → Valid 引脚：继续
  │     → NotValid 引脚：设 DamageAmount = 0
  │
  ├─ Set Instanced Struct Value (OutEffect, DE_MinimalHurt)
  │     DamageAmount = 上一步读到的值
  │
  └─ (无须调 SetEffect——框架会把 OutEffect 自动写入 DC)
```

### Step 3：创建 DamageRule

Content Browser → 右键 → **Damage Pipeline / Damage Rule** → 命名 `DR_MinimalHurt`。

- `Operation Class` = `BP_Op_MinimalHurt`
- `Condition` = 留空（无条件触发）

### Step 4：创建 DamagePipeline 资产

Content Browser → 右键 → **Damage Pipeline / Damage Pipeline** → 命名 `DP_MyPipeline`。

- `Damage Rules` → 加一条 `DR_MinimalHurt`

**双击打开**——左侧 Details / 右侧 Graph。

### Step 5：点 Toolbar 的 Build 按钮（或按 F7）

Graph 里出现一个节点 `1. DR_MinimalHurt`，输出 Pin 是 `DE_MinimalHurt`。**状态变成 Baked（绿色）**。

### Step 6：运行时 Execute

在任意 Actor 蓝图里：

```
任意触发点（BeginPlay / 碰撞事件 / AnimNotify）
  │
  ├─ Construct Object (UDamageContext) → 存为 Ctx
  │
  ├─ Set Effect (Ctx, FSekiroAttackContext)
  │     DamageAmount = 100
  │
  ├─ DP_MyPipeline → Execute (Ctx) → 返回 ExecutionLog
  │
  └─ Get Effect (Ctx, DE_MinimalHurt)
        → Valid：Print "Took X damage"
        → NotValid：Print "Rule did not fire"
```

**完成。** 这就是一条最小可运行的 Rule。

## §3 R1-R6 六条硬约束速查卡

六条约束是 DSL 的**语义契约**。速查即可，深度解释在 [`架构设计.md §5.2`](./架构设计.md#5-2-r1-r6-本质约束深度解释)。

| # | 约束 | 违反时的典型错误 |
|---|---|---|
| **R1** | DC 是唯一通信介质 | Operation A 直接调 Operation B——崩 |
| **R2** | Operation 自洽，不知道自己的身份 | Operation 里查询"我产出的是什么类型"——Execute 不是为此设计 |
| **R3** | Condition 只读 DC，缺失视为 false | 想在 Condition 里写 DC——禁止 |
| **R4** | Condition 与 Rule 是装配关系 | 判定条件硬编码在 Operation 的 if-else 里——违反涌现自治 |
| **R5** | 执行顺序由 EffectType 产销关系自动决定 | 手动调整 Rules 数组来影响执行顺序（Build 后一律按 SortedRules） |
| **R6** | 禁止循环依赖 | Rule A 消费 B 的产出，Rule B 消费 A 的产出——Build 失败 |

**深度解释与为什么这 6 条互相支撑**：[`架构设计.md §5.2`](./架构设计.md#5-2-r1-r6-本质约束深度解释)

## §4 蓝图实战四例

四种典型形态覆盖 95% 的使用场景。Condition 按 `_Effect` / `_Context` 二选一继承（详见 [`架构设计.md §3.4`](./架构设计.md)）。

### 实例 A：纯计算型（只产出，不依赖）

**典型场景**：攻击上下文的结构化封装。

```
Effect:  FSekiroAttackContext  // 包含 DamageAmount / bIsLightning / bIsInAir 等原始字段
Rule:    DR_AttackContext
         Operation: BP_Op_AttackContext
         Condition: 无
```

**Operation 的 Execute 流程**：从外部参数（或游戏 Actor 上的状态）构造 `FSekiroAttackContext`，`Set Instanced Struct Value` 到 OutEffect。

**特点**：下游所有 Rule 都能消费这个 Effect——攻击事件的单一源头。

### 实例 B：条件分支型（Effect-based Condition 门控）

**典型场景**：只有在"攻击类型是 Guard"时才执行格挡结算。

```
Effect:  FGuardEffect { PostureDamage: float, bJustGuard: bool }
Rule:    DR_Guard
         Operation: BP_Op_Guard
         Condition:
           UDamagePredicate(Op=Single)
             Condition: UDamageCondition_IsGuard : public UDamageCondition_Effect
               EffectType = FSekiroAttackContext (由子类 CDO 固定，实例不可改)
```

**Condition 的子类实现**（继承 `UDamageCondition_Effect`）：

```
UDamageCondition_IsGuard::Evaluate
  输入：Context, InEffect (类型 = FSekiroAttackContext，框架预取)
  流程：
    GetPtr<FSekiroAttackContext>(InEffect)
      → 非空：返回 ctx->bIsGuard
      → 空：返回 false (R3 缺失容错)
```

**Operation 的 Execute 流程**（仅在 Condition 通过时执行）：通过 Operation 基类的 `ReadEffect<T>(Context)` 读上游 Effect → 算 PostureDamage → 填 OutEffect。

**特点**：Condition 和 Operation 解耦。想切换"Guard"判定为"JustGuard + Guard 同时满足"？只需改 Condition 树，Operation 代码不动。

### 实例 C：Context-based Condition（看全局状态，不依赖 Effect）

**典型场景**：只在"当前 Tutorial 阶段"才触发某条 Rule。这种判定和任何 Effect 都无关，只看 Game 全局状态。

```
Game 侧 DamageContext 子类（Game 扩展字段）:
  UCLASS(Blueprintable)
  class USekiroDamageContext : public UDamageContext
  {
      UPROPERTY(BlueprintReadWrite) bool bIsInTutorial = false;
  };

Condition 子类（继承 UDamageCondition_Context）：
  UDamageCondition_InTutorial::Evaluate
    输入：Context（没有 InEffect！签名里根本没这个参数）
    流程：
      Cast<USekiroDamageContext>(Context)->bIsInTutorial
        → 访问 Game 扩展字段 ✅
      // ❌ 下面这行**编译失败**：GetEffect 是 protected
      // Context->GetEffect<FGuardEffect>();

Rule 里用法：
  DR_TutorialOnly
    Condition: UDamagePredicate(Op=Single, Condition=UDamageCondition_InTutorial)
```

**特点**：
- 不贡献 R5 产销依赖——Context-based Condition 不参与拓扑排序
- Graph 节点上无输入 Pin、无色块
- **编译期禁止**读任何 Effect——类型契约强制（详见 [`架构设计.md §6.4`](./架构设计.md)）

**何时选 Context-based**：判定只看 Game 全局状态（tutorial / boss 阶段 / 难度 / 调试开关）。如果需要同时看 Effect + Game 字段——用 Effect-based（实例 B），在 Evaluate 里同时读 InEffect 和 `Cast<GameContextSubclass>(Context)` 的扩展字段。

### 实例 D：信号式（无字段 Effect）

**典型场景**：告诉下游"这次伤害是空中接雷"——不需要传数据，存在即 true。

```
Effect:  FLightningInAirEffect {}  // 无字段
Rule:    DR_LightningInAir
         Operation: BP_Op_LightningInAir (只填一个空的 FLightningInAirEffect 到 OutEffect)
         Condition:
           UDamagePredicate(Op=AND)
             Children:
               UDamagePredicate(Op=Single, Condition=UDamageCondition_IsLightning)
               UDamagePredicate(Op=Single, Condition=UDamageCondition_IsInAir)
```

**下游 Rule 消费**：

```
DR_Death
  Condition:
    UDamagePredicate(Op=AND)
      Children:
        UDamagePredicate(Op=Single, Condition=UDamageCondition_HPZero)
        UDamagePredicate(Op=Single, Condition=UDamageCondition_HasLightningInAir, bReverse=true)
        // 如果空中接雷 → 跳过死亡判定（涌现：雷反不受伤）
```

**特点**：信号 Effect **存在即 true，不存在即 false**，不需要读字段。代码极简。

## §5 Condition 树配置语法速查

Condition 树是递归结构：容器（Predicate）嵌套容器，叶子是原子 Condition。

### 基础四种

```
// 1. 单条件（Single）
UDamagePredicate(Op=Single)
  Condition: UDamageCondition_IsGuard

// 2. AND 组合
UDamagePredicate(Op=AND)
  Children:
    UDamagePredicate(Op=Single, Condition=A)
    UDamagePredicate(Op=Single, Condition=B)
    UDamagePredicate(Op=Single, Condition=C)

// 3. OR 组合
UDamagePredicate(Op=OR)
  Children:
    UDamagePredicate(Op=Single, Condition=A)
    UDamagePredicate(Op=Single, Condition=B)

// 4. NOT 取反（bReverse=true）
UDamagePredicate(Op=Single, bReverse=true)
  Condition: UDamageCondition_HasLightningInAir
```

### 嵌套（任意层次）

```
// (A AND B) OR (C AND NOT D)
UDamagePredicate(Op=OR)
  Children:
    UDamagePredicate(Op=AND)
      Children:
        UDamagePredicate(Op=Single, Condition=A)
        UDamagePredicate(Op=Single, Condition=B)
    UDamagePredicate(Op=AND)
      Children:
        UDamagePredicate(Op=Single, Condition=C)
        UDamagePredicate(Op=Single, Condition=D, bReverse=true)
```

### 关键规则

- **bReverse 只在 Predicate 上**，不在 Condition 上——统一在容器层
- **空 Condition 字段** = 无条件触发（整棵树缺失 → true）
- **Single + 空 Condition** = true（Single 但没填具体 Condition 子类 → true）
- **AND 子列表为空** = true（数学约定：空 AND 为真）
- **OR 子列表为空** = false（数学约定：空 OR 为假）

## §6 "缺失即 false" 容错用法与反例

### 用法：动态管线的自洽

**场景**：Hades 式肉鸽。没选"空中感电 Boon" → 管线里没有 `DR_LightningInAir` → 从不会有 `FLightningInAirEffect` 写入 DC。

`DR_Death` 的 Condition 中有一项 `NOT HasLightningInAir`：

- **没选 Boon** → DC 里查 `FLightningInAirEffect` → 不存在 → 视为 false → NOT false = true → Death 正常触发
- **选了 Boon 且空中中雷** → `FLightningInAirEffect` 存在 → NOT true = false → Death 跳过（涌现：雷反不受伤）

**同一个 Condition 在两种管线配置下都正确工作。** 这是"缺失即 false"带来的自洽性。

### 反例一：数值默认当真值

**反例**：`UDamageCondition_LargeDamage` 检查 `FSekiroAttackContext.DamageAmount > 50`。

- 如果 `FSekiroAttackContext` **缺失**（没预填）→ GetPtr 返回 nullptr → Condition 作者若写成 "缺失返回 true" → 被错认为"大伤害"
- **正确做法**：Condition 作者显式处理 `ctx == nullptr ? false : ctx->DamageAmount > 50`

### 反例二：信号 Effect 的语义翻转

**反例**：`DR_Guard` 的 Condition 写成 `NOT HasGuard`（想表达"没有格挡时执行"）。

- 问题：`FGuardEffect` 是 `DR_Guard` 自己产出的——在自己的 Condition 里判断自己的 Produces 必然 false（因为 Build 时自己还没执行，DC 里没有）。
- **正确做法**：用消费**上游**（如 `FSekiroAttackContext`）的 Condition 判定。

## §7 常见错误 FAQ

### Q1：Build 报错 "Cycle detected"

**症状**：Toolbar Build 后 Graph 空白，Output Log 里 `[DamagePipeline] Cycle detected`。

**原因**：存在环依赖。A 的 Condition 消费 B 的 Produces，B 的 Condition 消费 A 的 Produces。

**排查**：
1. Output Log 里会列出参与环的 Rule 名
2. 检查每条 Rule 的 Condition 里用了哪些 Condition 子类
3. 每个 Condition 子类有固定的 EffectType——画出 "Rule.Produces → Condition.EffectType → 下游 Rule" 的依赖图
4. 找环并消除（通常是某条 Rule 的 Condition 消费了不该消费的 Effect）

### Q2：Build 报错 "EffectType null"

**症状**：某条 Rule Build 失败。

**原因**：`OperationClass` 留空；或 Operation 子类没重写 `GetProducesEffectType()`。

**修复**：
- C++：确保 Operation 子类 override `GetProducesEffectType`
- 蓝图：在 Operation 蓝图的 Class Defaults 面板填 `Produces Effect Type`

### Q3：Execute 没调用 Rule 的 Execute 事件

**症状**：运行时 Rule 没执行，ExecutionLog 里该条 `bConditionPassed = false`。

**原因**：Condition 没通过。

**排查**：
1. ExecutionLog 里已指出是 Condition 失败还是根本没到这条 Rule
2. 如果是 Condition 失败——检查 Condition 子类 Evaluate 的返回值
3. Condition 消费的 EffectType 可能在 DC 里不存在（R3 缺失视为 false）
4. 用 Graph Editor 查看节点的 Condition ASCII 树，对照 DC 实际状态

### Q4：蓝图里看不到 Get Effect / Set Effect 节点

**症状**：蓝图 Action Menu 里搜 "Get Effect" 找不到通用的 DamageContext 读写节点。

**原因**：v4.7 删除了旧的 `UDamageContextLibrary`（曾经的通用蓝图读写桥），目的是防止 Condition/Operation 内部绕过 R5 产销声明。

**修复**：
- **Game 侧**（Execute 前填攻击上下文、Execute 后读 Rule 产出）：搜 **"Read Pipeline Result"** / **"Write Pipeline Input"**——`UDamagePipelineResults` 提供的新蓝图节点，带通配结构体引脚 + Valid/NotValid exec
- **Condition 内部**：不需要"读 Effect"节点——`_Effect` 子类的 Evaluate 事件已经把 InEffect 作为参数传入；`_Context` 子类设计上就不能读 Effect
- **Operation 内部**：Operation 基类的 `ReadEffect<T>` 模板（C++ 侧 `ReadEffect<T>(Context)`）；蓝图 Operation 如果需要读上游 Effect，通过 Operation 基类暴露的节点

详见 [`架构设计.md §6.4`](./架构设计.md) DC 访问分层。

### Q4b：我的老代码直接调 `Context->GetEffect<T>()` 现在报错 "GetEffect is protected"

**这是正常的——v4.7 的设计目的**。修复方式按调用场景：

| 场景 | 替代 API |
|---|---|
| Game 侧 C++（Execute 前填 / 后读） | `UDamagePipelineResults::ReadEffect<T>(Context)` / `WriteEffect<T>(Context, Val)` |
| Operation 子类的 Execute 实现 | `ReadEffect<T>(Context)`（Operation 基类提供的 protected 模板） |
| Condition_Effect 子类 | 不需要——InEffect 已作为 Evaluate 参数传入 |
| Condition_Context 子类 | **不应该读 Effect**（设计上禁止）——要么改成 Effect-based 子类，要么通过 Cast 访问 Game 扩展字段 |

### Q5：Execute 后 Ctx 里没有我 Rule 产出的 Effect

**症状**：ExecutionLog 显示 Rule 执行了（bConditionPassed = true），但 `GetEffect<T>` 返回 nullptr。

**原因**：Operation 的 Execute 没有调用 `Set Instanced Struct Value` 填 OutEffect——OutEffect 仍是空的 Effect（`UScriptStruct*` 已设好，但字段都是默认值），框架仍会 SetEffectByType。如果 `GetEffect<T>` 用 `GetPtr<T>()` 判空通过但字段全是默认值，有可能误判"Rule 没执行"。

**修复**：
- Operation 必须调 `Set Instanced Struct Value(OutEffect, 我要写的具体 struct)`
- 消费方不应假设"能 Get 到就是执行过"——看 ExecutionLog 或用信号式 Effect 明确表达

### Q6：Graph Editor 里 Rule 的 Condition 没显示

**症状**：节点里只显示 "(always)" 或 Condition 行空白。

**原因**：Rule 的 Condition 字段为空（正常），或 Condition 子类没有实例化（异常——检查 Detail 面板填没填）。

**修复**：如果是 "(always)"——这是预期行为，表示 Condition 字段留空 = 无条件触发。
