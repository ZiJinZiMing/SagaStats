# 模块化伤害管线 — 快速入门

> 5 分钟理解动作游戏伤害管线的本质区别与核心概念

---

## 一句话定义

**模块化伤害管线**：动作游戏中"攻击命中后发生什么"的完整处理流程。它把每个战斗机制（格挡、弹反、雷击、毒蓄积……）封装为独立的积木块，通过共享上下文自动协作，产生涌现效果。

---

## 为什么需要它？

### 传统做法的痛点

```
// 传统做法：一坨巨型函数
void ApplyDamage(Attacker, Defender, DamageInfo) {
    if (Defender.IsGuarding) {
        if (DamageInfo.IsLightning && Defender.IsInAir) {
            // 空中接雷特殊处理...
        } else if (Defender.IsJustGuard) {
            // 完美格挡...
            if (Attacker.PostureDamage > Threshold) {
                // 破防...
            }
        }
        // 200行 if-else 嵌套...
    }
}
```

**问题**：
- 加一个新机制 → 改 10 个地方
- 新人不敢碰，老人不敢改
- 策划改不了，程序加班改
- 无法运行时动态组装（肉鸽游戏死路一条）

### 模块化做法

```
// 模块化做法：每个机制是独立积木块
DPU_Mixup     → 博弈判定（攻/防/闪）
DPU_Guard     → 格挡处理 + 架势结算
DPU_Lightning → 雷击处理
DPU_Death     → 死亡判定

// 积木块之间不直接调用，通过共享上下文（DC）通信
// 执行顺序自动拓扑排序，不需要手动排列
```

**效果**：
- 加新机制 = 加新积木块 + 配 Condition，已有积木不动
- 策划可配规则，程序维护模块稳定
- 运行时可动态组装不同的积木块子集

## 核心概念速览

### 三个核心概念

```
┌──────────────────────────────────────────────┐
│                    DC                         │
│              （共享上下文）                     │
│                                              │
│  上下文信息：攻击方属性、受击方属性、碰撞结果   │
│  DPU产出：  IsGuard、PostureResult、IsDead    │
│                                              │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐     │
│  │DPU_Guard│  │DPU_Light│  │DPU_Death│     │
│  │ 读DC    │  │ 读DC    │  │ 读DC    │     │
│  │ 执行逻辑│  │ 执行逻辑│  │ 执行逻辑│     │
│  │ 写回DC  │  │ 写回DC  │  │ 写回DC  │     │
│  └────┬────┘  └────┬────┘  └────┬────┘     │
│       │            │            │            │
│  ┌────┴────┐  ┌────┴────┐  ┌────┴────┐     │
│  │Condition│  │Condition│  │Condition│     │
│  │Guard=T  │  │Light&&  │  │!IsLight │     │
│  │         │  │InAir    │  │InAir    │     │
│  └─────────┘  └─────────┘  └─────────┘     │
└──────────────────────────────────────────────┘
```

| 概念 | 类比 | 核心规则 |
|------|------|---------|
| **DPU**（机制处理体） | 乐高积木块 | 自洽独立。只通过 DC 通信，不知道其他 DPU 的存在 |
| **DC**（共享上下文） | 公共白板 | 生命周期瞬时。事件开始创建，结束销毁 |
| **Condition**（条件门控） | 积木块的开关 | 只读 DC，输出 true/false。缺失字段视为 false |

### 两阶段处理

一次伤害事件分两步：

```
Attack 命中
  → 【机制阶段】各 DPU 判定和结算（谁生效？结果是什么？）
  → 【表现选取阶段】多个结果抢有限的表现资源（播哪个动画？哪个音效？）
  → 表现执行
```

### 自动拓扑排序

DPU 声明自己"生产什么"（produces），Condition 隐含"消费什么"（consumes）。系统自动生成执行顺序：

```
DPU_Mixup   produces: [MixupResult]
DPU_Guard   Condition: MixupResult == Guard  ← 消费 MixupResult
                                              ← 所以 Mixup 在 Guard 之前执行

设计者不需要手动排列顺序。
```

## 动作游戏 vs 数值游戏：一张图看懂

```
数值游戏（MMO/ARPG）：
  伤害 = f(攻击, 防御, 倍率, buff...)
  玩家做的事：把系数调到最大
  流程：────────────────→ （一条线，调刻度）

动作游戏（只狼/鬼泣/Hades）：
  伤害 = 你选哪个函数
  玩家做的事：选走哪条链路
  流程：──┬── 格挡流 → 架势结算 → 破防
         ├── 弹刀流 → 反击窗口
         ├── 雷反流 → 空中接雷不受伤
         └── 直接命中 → HP 结算 → 死亡判定
```

**数值游戏是刻度不同，动作游戏是尺子本身就不一样。**

## 关键设计原则

### 四条设计思想（记住这四句话就够了）

| # | 思想 | 一句话 |
|---|------|--------|
| ① | **机制优先** | 先有"格挡""弹刀""雷反"这些体验机制，再有数值填充 |
| ② | **数值兜底** | 数值不驱动体验，但保证数学基础正确 |
| ③ | **反馈清晰可感** | 玩家感知不到的机制等于不存在 |
| ④ | **涌现自治** | 多机制组合效果不穷举，由各机制在共享上下文上自然产生 |

### 六条本质约束（删掉任何一条都会出问题）

| 规则 | 说明 |
|------|------|
| R1 | DC 是唯一通信介质——DPU 之间不直接调用 |
| R2 | DPU 自洽——不知道自己被什么 Condition 门控 |
| R3 | Condition 只读 DC，缺失字段视为 false |
| R4 | Condition 与 DPU 是装配关系——新增涌现 = 调整 Condition，不改 DPU 内部 |
| R5 | 执行顺序由依赖关系决定，支持动态管线 |
| R6 | 禁止循环依赖——互相依赖 = 不是独立机制 = 合并 |

---

## 固定管线 vs 动态管线

| 模式 | DPU 子集何时确定 | 典型游戏 |
|------|---------------|---------|
| **固定管线** | 设计时确定，保存为 Asset | 只狼、鬼泣 |
| **动态管线** | 运行时根据 Boon/祝福确定 | Hades、Slay the Spire |

两种模式的运行时执行逻辑**完全相同**——区别只在于 Pipeline Asset 的来源。

## 蓝图实战：搭建一套伤害管线

框架的全部核心 API 已暴露给蓝图。以下以 `ADPUPipelineTestActor`（只狼 MVP）为参考，演示如何**纯蓝图**搭建一条伤害管线。

### 步骤总览

```
1. 创建 DPU 逻辑蓝图（继承 UDPULogicBase，重写 Execute 事件）
2. 创建 DPU 定义资产（DataAsset，填写 Produces / Condition / LogicClass）
3. 创建管线 Actor 蓝图（组装 DPU、构建 DC、调用 PipelineManager）
```

### 步骤一：创建 DPU 逻辑蓝图

> 每个 DPU 逻辑 = 一个蓝图类，负责读 DC → 执行机制 → 写回 DC

1. Content Browser → 右键 → Blueprint Class → 父类选 **DPULogicBase**
2. 命名为 `BP_DPULogic_Guard`（格挡判定示例）
3. 打开蓝图，在 **Functions** 面板重写 **Execute** 事件
4. 在 Execute 事件图中：

```
Execute 事件 (输入: DC)
  │
  ├─ DC → Get Float ("guard_level") → 存为局部变量 GuardLevel
  ├─ DC → Get Float ("DmgLevel")    → 存为局部变量 DmgLevel
  │
  ├─ DC → Set Bool ("GuardSuccess", GuardLevel >= DmgLevel)
  └─ DC → Set Bool ("IsJustGuard",  GuardLevel > DmgLevel)
```

**关键规则**：
- DPU 内部**只和 DC 交互**，不引用任何其他 DPU 或外部系统（R1 + R2）
- 写入的字段必须和 DPU 定义中的 Produces 声明一致

同理创建其他 DPU 逻辑蓝图：

| 蓝图名 | 读取 DC | 写入 DC |
|--------|---------|---------|
| `BP_DPULogic_Mixup` | guard_level | Guard (bool) |
| `BP_DPULogic_Guard` | guard_level, DmgLevel | GuardSuccess, IsJustGuard |
| `BP_DPULogic_Death` | CurrentHP, DmgLevel | IsDeath |
| `BP_DPULogic_LightningInAir` | (无需读取) | IsLightningInAir = true |

### 步骤二：创建 DPU 定义资产（DataAsset）

> 每个 DPU 定义 = 一个 DataAsset，声明"我生产什么"和"什么条件下执行"

1. Content Browser → 右键 → Miscellaneous → **Data Asset** → 选 **DPUDefinition**
2. 命名为 `DA_DPU_Guard`
3. 填写属性：

```
DA_DPU_Guard:
  DPU Name:     "DPU_Guard"
  Description:  "格挡判定 + 架势结算"
  Produces:     ["GuardSuccess", "IsJustGuard"]
  Condition:    StringConditionExpression
                  Expression = "Guard && !IsLightningInAir"
  Logic Class:  BP_DPULogic_Guard
  Base Priority: 60
```

**Condition 表达式支持的语法**：

| 表达式 | 含义 |
|--------|------|
| `Guard` | DC 中 Guard 字段为 true |
| `!IsLightningInAir` | DC 中 IsLightningInAir 不存在或为 false |
| `Guard && !IsLightningInAir` | 两者同时满足 |
| `guard_level >= 3` | DC 中 guard_level 的数值 ≥ 3 |
| `DmgLevel > CurrentHP` | 两个 DC 字段的数值比较 |
| `AttackType == HeavySlash` | Name 类型相等比较 |

**完整运算符**：`&&`, `||`, `!`, `==`, `!=`, `>`, `<`, `>=`, `<=`, `()`

同理创建其他 DPU 定义：

| DataAsset | Produces | Condition | Logic Class |
|-----------|----------|-----------|-------------|
| DA_DPU_Mixup | [Guard] | (空 = 永远执行) | BP_DPULogic_Mixup |
| DA_DPU_Guard | [GuardSuccess, IsJustGuard] | `Guard && !IsLightningInAir` | BP_DPULogic_Guard |
| DA_DPU_Death | [IsDeath] | `!(Guard && GuardSuccess) && !IsLightningInAir` | BP_DPULogic_Death |
| DA_DPU_LightningInAir | [IsLightningInAir] | `Lightning && IsInAir` | BP_DPULogic_LightningInAir |

### 步骤三：创建管线 Actor 蓝图

> 管线 Actor = 组装 DPU、构建 DC、驱动 PipelineManager 执行

1. 创建 Actor 蓝图 `BP_DamagePipelineRunner`
2. 添加变量：

```
PipelineManager : UPipelineManager (在 BeginPlay 中 Construct Object)
ActiveDPUs      : Array<UDPUDefinition> (在编辑器中拖入 DA_DPU_xxx 资产引用)
```

3. **BeginPlay** 中初始化 PipelineManager：

```
BeginPlay
  │
  └─ Construct Object from Class (UPipelineManager) → 存为 PipelineManager
```

4. 创建 **RunDamageEvent** 自定义函数（可绑定到碰撞检测/AnimNotify 等）：

```
RunDamageEvent (输入: Attacker, Defender, AttackData)
  │
  ├─ 【构建 DC】
  │   Construct Object (UDamageContext) → 存为 DC
  │   DC → Set Float ("DmgLevel", AttackData.DmgLevel)
  │   DC → Set Float ("guard_level", Defender.GuardLevel)
  │   DC → Set Float ("CurrentHP", Defender.CurrentHP)
  │   DC → Set Bool  ("Lightning", AttackData.bIsLightning)
  │   DC → Set Bool  ("IsInAir", Defender.bIsInAir)
  │
  ├─ 【执行管线】
  │   PipelineManager → Execute Pipeline (ActiveDPUs, DC)
  │   返回 ExecutionLog (Array<FDPUExecutionEntry>)
  │
  └─ 【读取结果】
      DC → Get Bool ("IsDeath")   → 如果 true → 执行死亡逻辑
      DC → Get Bool ("IsJustGuard") → 如果 true → 播放弹刀反馈
      DC → Get Bool ("IsLightningInAir") → 如果 true → 播放雷反特效
```

### 完整蓝图数据流

```
 Attack 命中（碰撞/AnimNotify 触发）
   │
   ▼
 构建 DC（填入攻击方/受击方/事件上下文）
   │
   ▼
 PipelineManager → ExecutePipeline(ActiveDPUs, DC)
   │  内部自动：
   │  1. 稳定拓扑排序（Produces/Consumes 依赖关系）
   │  2. 遍历每个 DPU：评估 Condition → 执行 / 跳过
   │  3. 自动导出 Mermaid DAG（调试用）
   │
   ▼
 从 DC 读取结果，驱动表现和游戏逻辑
   ├─ IsDeath = true  → 播放死亡
   ├─ IsJustGuard     → 播放弹刀 + 弹开攻击方
   ├─ IsLightningInAir → 播放雷反
   └─ ...
```

### C++ 参考对照

上述蓝图流程和 `ADPUPipelineTestActor`（C++ 实现）完全对应：

| 蓝图步骤 | C++ 对应 | 文件位置 |
|---------|----------|---------|
| 创建 DPU 逻辑蓝图 | `DPUTestLogics.h` 中 13 个 UDPULogic_TestXxx | `Public/DPUFramework/DPUTestLogics.h` |
| 创建 DPU 定义 | `MakeRTDPU()` 辅助函数 | `Private/DPUFramework/DPUPipelineTestActor.cpp:13` |
| 组装管线 | `BuildSekiroPipeline()` | `Private/DPUFramework/DPUPipelineTestActor.cpp:48` |
| 构建 DC + 执行 | `RunScenario_NormalHit()` 等 | `Private/DPUFramework/DPUPipelineTestActor.cpp:128` |

**蓝图和 C++ 可以混用**——C++ 写的 DPU 逻辑和蓝图写的 DPU 逻辑可以放在同一条管线中执行，PipelineManager 不区分来源。

### 调试技巧

1. **Mermaid DAG 自动导出**：PipelineManager 默认在每次 ExecutePipeline 后导出 `.mmd` 文件到 `Saved/Graphs/`，可以在 [mermaid.live](https://mermaid.live/) 中可视化查看执行结果
2. **DC Dump**：调用 `DC → Dump To String` 可以在运行时打印所有 DC 字段的当前值
3. **执行日志**：ExecutePipeline 返回的 `TArray<FDPUExecutionEntry>` 记录了每个 DPU 的 EXEC/SKIP 状态
