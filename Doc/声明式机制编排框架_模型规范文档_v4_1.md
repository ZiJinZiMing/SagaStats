## 一、问题域的本质复杂度

以下三个概念经过删除测试、来源追溯、换表达验证——确认为问题域的本质复杂度，无法消除、无法简化、无法合并。

### 1.1 三个核心概念

| 概念 | 一句话定义 | 为什么是本质的 |
|------|-----------|--------------|
| **DPU** | 机制处理体——自洽的积木块，声明produces字段，读取DC、执行机制逻辑、将结果写回DC | 删掉它：机制逻辑无处承载。换GAS表达：Gameplay Ability重新出现。从只狼ExecDamage责任链中长出来。 |
| **DC** | 共享上下文——本次事件的完整数据环境，承载上下文信息+DPU产出 | 删掉它：机制间无法通信。换GAS表达：FGameplayEffectContext重新出现。从只狼DamageContext逆向中长出来。 |
| **Condition** | 执行条件门控——读取DC全部信息，输出true/false，决定DPU是否执行，隐含consumes声明 | 删掉它：涌现只能藏在DPU内部，无法外显。换GAS表达：ActivationRequirements重新出现。从只狼Guard/JustGuard博弈结果中长出来。 |

三者的关系：

```
DPU 是积木块——形状和功能固定，自洽完整
Condition 是卡扣——决定这块积木在什么条件下被激活
DC 是桌面——所有积木共享的工作台

换一个卡扣 = 改变激活条件，积木本身不变
加一个新涌现 = 新积木 + 新卡扣，已有积木不改
```

### 1.2 涌现

涌现是多个机制组合产生的效果。它不是一个独立的模型概念——它是DPU+Condition在DC上运行的自然结果。

涌现的本质认知：**DPU内部的每一条if-else分支，都可能是另一个独立机制的产出与当前机制的涌现。** 识别"什么是独立机制、什么是机制间的涌现"是设计者的核心工作，不可由工具或AI代劳。

```
示例：DPU_Lightning 内部的 if(IsInAir)

原始理解：这是Lightning的内部规则
深层理解："角色在空中"是一个独立机制（动作系统的产出），
          "空中 + 雷击"是两个机制的涌现
          → IsInAir应该是DC中的独立事实
          → "空中雷击的特殊处理"应该是一个带Condition的独立DPU
```

### 1.3 依赖关系与产销声明

机制之间存在信息依赖——A写入DC的结果，B需要读取。这决定了执行顺序：A必须在B之前。

#### 产销声明

依赖关系通过两种声明确定：

| 声明 | 来源 | 含义 |
|------|------|------|
| **Produces** | DPU显式声明自己写入DC的字段列表 | "我是这些DC字段的生产者" |
| **Consumes** | 从Condition中自动提取引用的DC字段 | "我依赖这些DC字段的存在"——不需要额外声明 |

```
示例：

DPU_Death {
  produces: [IsDead, DeathType]                    ← 显式声明
  Condition: !(Guard && GuardSuccess)              ← 隐含consumes: [Guard, GuardSuccess]
  逻辑：HP结算
}

DPU_LightningDeath {
  produces: [LightningDeathAnim]                   ← 显式声明
  Condition: Lightning && IsDead                   ← 隐含consumes: [Lightning, IsDead]
  逻辑：雷电死亡表现
}

自动推导的依赖：
  IsDead 的生产者是 DPU_Death
  IsDead 的消费者是 DPU_LightningDeath
  → DPU_Death 必须在 DPU_LightningDeath 之前执行
```

#### 自动拓扑排序

系统根据产销关系自动生成执行顺序：

1. 收集所有DPU的produces声明和Condition的consumes引用
2. 匹配：字段X的生产者 → 字段X的所有消费者 → 建立有向边
3. 拓扑排序：生产者在消费者之前执行
4. 无依赖关系的DPU之间顺序不确定（可并行执行）

**设计者不需要手动排列DPU顺序。** 设计者只需要定义每个DPU的produces和Condition——顺序由系统自动推导。

#### 防环规则

如果两个DPU互相依赖对方的产出（A的Condition引用B的produces字段，B的Condition引用A的produces字段），系统无法确定谁先执行——这意味着它们不是两个独立机制，应该合并为一个DPU。

```
检测到环：
  DPU_A produces [X], Condition consumes [Y]
  DPU_B produces [Y], Condition consumes [X]
  → A依赖B，B依赖A → 环 → A和B应该合并
```

#### DC初始字段的特殊性

DC的上下文信息（攻击方属性、受击方属性、事件上下文）在管线开始前就已填入——它们没有DPU生产者。所有引用这些字段的Condition不产生依赖边——这些字段对所有DPU可见，不影响排序。

```
DC初始字段（无生产者DPU）：
  damageElement, IsInAir, IsPlayer, attackPower, ...
  → 任何DPU的Condition引用这些字段都不产生依赖
  → 这些DPU可以自由排序
```

### 1.4 静态规则与动态组装

框架将**规则定义**和**组合选择**完全分离为两个独立阶段。

#### 两阶段分离原则

| 阶段 | 谁做 | 做什么 | 何时做 | 频率 |
|------|------|--------|--------|------|
| **开发阶段：配规则** | 设计者 | 创建DPU、声明produces、配置Condition | 开发过程中，每次加入新机制时 | 一次性 |
| **运行阶段：配组合** | 系统/玩家 | 从全量DPU池中选择激活哪些DPU子集 | 每轮游戏运行时 | 每轮不同 |

**规则不随组合变化。** Condition是静态的游戏规则，DPU的激活子集是动态的玩家选择。两者完全解耦。

#### 开发阶段：一次性配好规则

当一个新机制首次加入时，设计者做三件事：

1. **创建新DPU**——声明produces，编写内部逻辑
2. **配置新DPU的Condition**——引用DC字段作为执行条件
3. **调整受影响的已有DPU的Condition**——在已有Condition中增加对新produces字段的响应

```
示例：首次加入"空中感电"机制

1. 创建DPU_LightningInAir，produces: [IsLightningInAir]
2. 配置Condition: Lightning && IsInAir
3. 给Death/Collapse/Hurt等DPU的Condition加上 && !IsLightningInAir

做完之后，这些规则永远不需要再改。
```

#### 运行阶段：自动处理任意组合

运行时，系统根据当前激活的DPU子集自动执行：

1. 确定本轮激活的DPU子集（如：玩家获得了哪些Boon）
2. 从激活DPU子集中提取produces和consumes
3. 对激活DPU做拓扑排序
4. 依次执行——Condition中引用的DC字段如果不存在（生产者DPU未激活），视为false

```
示例：

第1轮——玩家没有获得空中感电Boon：
  激活DPU：[Mixup, Guard, Death, Collapse, Hurt]
  DPU_LightningInAir不在管线中 → IsLightningInAir永远不写入DC
  Death的Condition: !(Guard && GuardSuccess) && !IsLightningInAir
  → IsLightningInAir缺失 → 视为false → !false = true → 不影响Death
  → 一切正常，和没有空中感电这个机制的时候一模一样

第2轮——玩家获得了空中感电Boon：
  激活DPU：[Mixup, Guard, Death, Collapse, LightningInAir, Hurt]
  空中被雷击时 → LightningInAir执行 → 写入IsLightningInAir = true
  Death的Condition: !IsLightningInAir = !true = false → Death跳过
  → 空中接雷不受伤，涌现自动生效

两轮中Death的Condition完全相同，没有任何修改。
```

#### 为什么这是本质的

三步检验：

- **删除测试**：删掉"缺失字段视为false"规则 → 每次DPU动态增减时都必须同步修改其他DPU的Condition → 规则与组合耦合 → 无法支撑动态组装场景 → 丢失
- **来源追溯**：从Hades的Boon系统需求中长出来——每轮运行的Boon组合不同，但游戏规则（Boon之间的交互逻辑）是预定义的
- **换表达验证**：换成GAS → GameplayTag的HasTag在Tag不存在时返回false → 同样的机制重新出现

## 二、本质约束规则

以下规则经过Essential/Accidental检验，确认为问题域的本质约束——删掉任何一条，模型的表达能力会受损。

| 编号 | 规则 | 为什么是本质的 |
|------|------|--------------|
| **R1** | **DC是唯一通信介质**：DPU之间不直接调用，所有数据交换通过DC。 | 删掉：DPU之间产生直接耦合，无法自由组合。 |
| **R2** | **DPU自洽**：DPU内部逻辑完整且独立。DPU不知道自己被什么Condition门控，也不知道其他DPU的存在。 | 删掉：DPU之间产生知识依赖，无法独立替换和扩展。 |
| **R3** | **Condition可读DC全部内容，缺失字段视为false**：Condition可读取DC中的一切。输出true/false，决定DPU是否执行。不写入任何数据。**当Condition引用的DC字段不存在时（因为其生产者DPU未加入当前管线），该字段视为false/null，条件按"不满足"处理。** | 删掉：Condition无法表达跨机制的涌现条件。"缺失视为false"删掉：DPU动态增减时必须同步修改Condition，规则与组合耦合。 |
| **R4** | **Condition与DPU是装配关系**：Condition不是DPU的一部分。新增涌现=调整Condition装配逻辑+添加新DPU，原则上不调整已有DPU内部实现。 | 删掉：涌现变化必须修改DPU内部，积木块不再稳定。 |
| **R5** | **执行顺序由依赖关系决定，支持动态管线**：DPU声明produces字段，Condition隐含consumes字段。系统自动拓扑排序——生产者在消费者之前执行。无依赖关系的DPU之间顺序不确定。**管线可在运行时动态组装——每次运行激活不同的DPU子集，拓扑排序只对当前激活的DPU进行。未激活DPU的produces字段不产生依赖边。** | 删掉：回到手动排序，规模膨胀时不可维护。"动态管线"删掉：无法支撑Roguelike等每轮运行不同的场景。 |
| **R6** | **禁止循环依赖**：如果两个DPU互相依赖对方的produces产出，它们应该合并为一个DPU。互相依赖=不是独立机制=不应拆分。 | 删掉：可能出现循环依赖导致管线无法执行。从DAG的基本数学性质长出来。 |

## 三、当前阶段的设计选择

以下内容经过Essential/Accidental检验，确认为**当前阶段合理的设计选择**，而非问题域的本质约束。

### 3.1 图即全部真相

**当前选择**：所有DPU、Condition、produces声明、依赖关系都在图表上可见，没有隐藏的内部规则。

**为什么不是本质**：涌现粒度是设计者的视图选择。原型期可能接受粗粒度DPU（涌现藏在内部），成熟期/肉鸽期需要细粒度DPU（涌现全部外显）。

**本质是什么**：涌现的可见性程度取决于DPU粒度选择。粒度越细，图上可见的涌现越多。粒度选择是项目决策，不是模型约束。

### 3.2 DPU粒度

**当前选择**：一个DPU对应一个策划能独立描述的体验意图。

**为什么不是本质**：DPU粒度没有客观正确答案。同一个机制可以是一个粗DPU，也可以拆成多个细DPU+各自Condition。程序实现上都是子类继承——已验证的机制逻辑通过继承保持不变。

**本质是什么**：DPU内部的每一条if-else，都可能是另一个独立机制的涌现。识别"什么该拆出来成为独立DPU+Condition"是设计者的核心判断。

粒度演进路径：

| 阶段 | 粒度 | 示例 |
|------|------|------|
| **原型期** | 粗——一个DPU覆盖一大块体验 | DPU_Lightning包含空中/地面/崩躯干全部变体 |
| **验证期** | 中——按涌现拆分 | DPU_LightningInAir + DPU_LightningGround + DPU_CollapseLightning |
| **肉鸽期** | 细——每个可被Boon独立修改的行为都是独立DPU | 每个DPU可被外部Boon的Condition+DPU独立影响 |

## 四、设计方法论

### 4.1 Essential/Accidental Complexity

**Essential Complexity（本质复杂度）**：问题域本身固有的、无法消除的复杂度。即使用最完美的工具和方法，它依然存在。

**Accidental Complexity（偶然复杂度）**：因为表达方式、工具、技术选择带来的额外复杂度。它是人为的、可消除的。

**核心指引**：模型规范中只应包含本质复杂度。偶然复杂度属于实现层、工具层、教学层——不进入模型规范。

**信号识别**：当你感到设计"拧巴"时，大概率是在和偶然复杂度搏斗。正确的响应不是在当前框架内优化，而是检查框架本身是否引入了偶然复杂度。

### 4.2 三步检验程序

每次引入新概念、新结构、新规则时，走以下三步：

**第一步：删除测试**

> "如果我把这个概念从模型中彻底删掉，问题域中有什么事情表达不了？"

丢了 = 本质。没丢 = 偶然。

```
删掉DC → DPU之间无法通信 → 丢了 → DC是本质的
删掉Chooser Table → DPU+Condition还能表达所有涌现 → 没丢 → Chooser Table是偶然的
```

**第二步：来源追溯**

> "这个概念是从问题域中'长出来'的，还是从其他领域'借来'的？"

长出来的大概率是本质的。借来的需要用第一步额外验证。

```
DPU → 从只狼ExecDamage责任链逆向中长出来 → 本质
Chooser Table → 从UE5 Chooser概念借来 → 做删除测试 → 偶然
```

**第三步：换表达验证**

> "如果我用完全不同的方式表达同一个问题域，这个概念还会重新出现吗？"

还在 = 本质。不在 = 偶然。

```
换成GAS表达：DPU → Gameplay Ability重新出现 → 本质
换成GAS表达：线性序列 → GAS用事件驱动，不用线性序列 → 偶然（当前选择）
```

### 4.3 涌现识别清单

每次在DPU内部写if-else时，问自己：

> "这个if的条件部分，是不是另一个独立机制的产出？"

```
if (IsInAir) → "角色在空中"是不是一个独立机制？
  → 是（动作系统的产出）→ 应该是DC中的独立事实
  → 提取为独立DPU+Condition

if (damageElement == Lightning) → "雷电伤害"是不是一个独立机制？
  → 是 → 同上

if (currentHP < threshold) → "HP低于阈值"是不是一个独立机制？
  → 不是，这是当前DPU内部的数值计算中间结果
  → 留在DPU内部
```

**判断标准：这个条件未来是否可能被其他机制独立引用。** 如果只在这一个DPU中有意义→留在内部。如果多个机制可能引用→提取为DC中的独立事实。

这个判断只有设计者能做——它取决于对游戏设计方向的理解，AI没有这个上下文。

### 4.4 偶然复杂度的产生规律

回顾本框架v1.0→v3.1的迭代过程，偶然复杂度的产生有一个清晰的模式：

| 版本 | 引入的偶然复杂度 | 根因 |
|------|----------------|------|
| v1.7 | Fact Key自动DAG构建 | 抽象层级过高，超出当时认知带宽 |
| v2.0 | Condition独立图节点、多输出口、连线分流、拦截链 | 用拓扑结构穷举涌现 |
| v2.5 | 词缀选取表、缺省逻辑、词缀归属原则、装饰器 | 用中间抽象层包装本质概念 |
| v3.0–v3.1 | 手动线性序列排序 | 依赖关系是本质的，但"手动排列顺序"是偶然的（v4.0已解决） |

**规律：偶然复杂度最容易以"看起来合理的新抽象"的形式混入模型。** 每次感到需要引入新概念时，先跑三步检验。

### 4.5 AI协作中的注意事项

在与AI（包括本框架的设计过程中使用的AI）协作时：

**设计者的模糊表述会把AI带向偶然复杂度。** AI会用具体方案填充设计者的模糊——但AI的方案往往是从其他领域"借来"的（如Chooser Table借自UE5），不是从当前问题域"长出来"的。

**应对方式**：
- AI给出方案后，设计者用三步检验程序审视
- 要求AI在给出方案前先做删除测试
- 如果设计者自己对问题域的理解还不清晰，先通过drawio等可视化工具加深理解，再让AI产出具体方案

**分工原则**：
- 本质复杂度的识别和判断：设计者的工作，不可委托给AI
- 偶然复杂度的实现（文档生成、代码骨架、格式检查、场景验证）：AI的工作，设计者做价值判断即可

## 五、DPU 定义

### 5.1 DPU是什么

DPU（Damage Processing Unit）是管线中的机制处理体。每个DPU是一个自洽的积木块——内部逻辑完整且独立，不知道自己被什么Condition门控，也不知道其他DPU的存在。

DPU的核心行为：从DC中读取数据 → 执行机制逻辑 → 将结果写回DC。

### 5.2 Produces声明

每个DPU必须显式声明自己写入DC的字段列表（produces）。这是依赖关系自动推导的基础。

```
DPU_Guard {
  produces: [IsGuard, IsJustGuard, GuardSuccess, GuardLevel]
  Condition: Guard && !IsLightningInAir
  逻辑：格挡判定+表现
}

DPU_LightningInAir {
  produces: [IsLightningInAir]
  Condition: Lightning && IsInAir
  逻辑：空中蓄电
}
```

produces声明的约束：
- **完整性**：DPU写入DC的所有字段都必须在produces中声明
- **独占性**：同一个DC字段只能有一个生产者DPU（如果多个DPU都可能写入同一字段，说明它们的职责划分有问题）
- **与DPU内部逻辑一致**：produces声明的是DPU"会写什么"，不是"一定写什么"——Condition为false时DPU不执行，自然不写入

### 5.3 DPU的产出写入DC

DPU执行后将结果写入DC。产出内容完全由机制本身决定——可能是判定结果、可能是数值计算、可能包含动画Tag。模型不预设产出的内容分类。

### 5.4 DPU产出的双向作用目标

某些DPU的产出需要同时描述对多个作用目标的影响（如弹刀同时影响受击方和攻击方）。作用目标信息是DPU产出的组成部分。

### 5.5 DPU的约束规则

| 规则 | 说明 |
|------|------|
| **自洽性** | DPU的内部逻辑完整且独立。不依赖外部Condition的存在，不知道其他DPU的存在。 |
| **只通过DC通信** | DPU不直接调用其他DPU。上下游影响通过DC间接实现。 |
| **不感知管线结构** | DPU不知道自己在管线中的位置、不知道自己被什么Condition门控。 |
| **声明produces** | DPU必须显式声明自己写入DC的字段列表。这是依赖关系自动推导的基础。 |

## 六、DC 定义

### 6.1 DC中的两类内容

| 类别 | 是什么 | 来源 |
|------|--------|------|
| **上下文信息** | 描述"这次事件是什么"的原始数据 | 从参数表、角色属性、引擎碰撞结果中读取 |
| **DPU产出** | 描述"什么事被处理了，结果是什么"的加工产物 | DPU执行后写入 |

### 6.2 特征

| 特征 | 说明 |
|------|------|
| 生命周期 | 瞬时——事件开始时创建，结束时销毁 |
| 可见性 | 所有DPU和Condition均可读取DC中的任何内容 |
| 写入权 | 只有DPU可以写入DC。Condition不写入。 |

### 6.3 上下文信息的分类

| 来源 | 说明 | drawio颜色 |
|------|------|-----------|
| **攻击方** | 攻击方角色和武器的属性数据 | 浅红 #f8cecc / 红框 #b85450 |
| **受击方** | 受击方角色和武器的属性数据 | 浅绿 #d5e8d4 / 绿框 #82b366 |
| **事件上下文** | 碰撞信息等非角色属性的事件数据 | 浅灰 #f5f5f5 / 灰框 #666666 |

## 七、Condition 定义

### 7.1 Condition是什么

Condition是DPU的装配门控。读取DC中的全部信息，输出true/false，决定DPU是否执行。

### 7.2 Condition的特征

| 特征 | 说明 |
|------|------|
| **可读范围** | DC中的一切——上下文信息、双方属性、已有DPU产出 |
| **输出** | true/false |
| **写入权** | 无。Condition不写入DC |
| **与DPU的关系** | 装配关系。可随时调整、替换、移除，不影响DPU内部实现 |

### 7.3 无Condition的DPU

DPU可以没有Condition——此时DPU永远执行。

### 7.4 新增涌现时的操作

1. 添加新DPU（如果涌现产生了新行为）
2. 配置Condition——可能需要调整已有DPU的Condition以实现互斥

已有DPU的内部实现不修改。改变的只是Condition装配逻辑。

## 八、问题域界定

### 8.1 本模型解决什么

本模型解决**机制阶段**的问题——独立机制的处理、执行顺序、机制间涌现。

### 8.2 本模型不解决什么

**表现选取阶段**是独立的问题域。机制阶段执行完毕后，DC中已填满全部机制处理结果，表现选取阶段从DC中读取进行决策。

### 8.3 DPU产出中的表现描述信息

DPU的产出中可能包含表现描述信息（如动画Tag、Motion数据）。这不违反问题域界定——表现描述是机制结果的一部分，DPU不关心这些信息最终会不会被播放。

## 九、设计指南（非模型约束）

以下内容是游戏设计层面的指导原则，不是框架模型的本质约束。它们对设计出好的游戏体验有重要价值，但框架在技术上不依赖它们。

### 9.1 四条设计思想

| 编号 | 思想 | 关切层面 | 核心表述 |
|------|------|---------|---------|
| **①** | **机制优先** | 单机制·设计 | 体验由机制驱动，不是由数值驱动。 |
| **②** | **数值兜底** | 单机制·实现 | 数值不驱动体验，但保证体验的数学基础正确。 |
| **③** | **反馈清晰可感** | 单机制·表现 | 每个机制的结果必须通过多模态表现被玩家感知到。 |
| **④** | **涌现自治** | 多机制·协作 | 机制之间的组合效果不由拓扑穷举，由DPU+Condition在DC上自然产生。 |

**注意**：这四条是"用这个框架做好游戏"的指导，不是"框架能否工作"的约束。框架在不遵循这四条的情况下仍然能运行——只是设计出来的游戏体验可能不好。

### 9.2 排序相关说明

执行顺序由产销依赖关系自动决定（R5）。设计者不需要手动排序——只需正确声明每个DPU的produces字段和Condition表达式。

无依赖关系的DPU之间顺序不确定。如果设计者希望对无依赖DPU之间施加特定顺序（如"暗线积累优先于感知过滤"），可以通过引入人为依赖来实现——但这属于设计选择，不是模型要求。

## 十一、术语表

| 术语 | 定义 |
|------|------|
| **DC (DamageContext)** | 本次事件的共享上下文。承载上下文信息+DPU产出。涌现的数据载体。字段缺失时视为false。 |
| **DPU** | 管线中的机制处理体。自洽的积木块——声明produces字段，读取DC、执行机制逻辑、写入DC。唯一的读写执行体。 |
| **Condition** | DPU的装配门控。读取DC全部信息，输出true/false，决定DPU是否执行。不写入DC。与DPU是装配关系。隐含consumes声明。引用的DC字段缺失时视为false（天然容错）。 |
| **Produces** | DPU显式声明自己写入DC的字段列表。是依赖关系自动推导的生产者端。 |
| **Consumes** | Condition中引用的DC字段列表，自动提取。是依赖关系自动推导的消费者端。不需要额外声明。 |
| **依赖关系** | 机制之间的信息依赖。由produces和consumes自动匹配建立——生产者必须在消费者之前执行。 |
| **拓扑排序** | 根据依赖关系自动生成DPU执行顺序的算法。无依赖关系的DPU之间顺序不确定。支持动态管线——只对当前激活的DPU子集排序。 |
| **涌现** | 多个机制组合产生的效果。DPU+Condition在DC上运行的自然结果，不是独立的模型概念。 |
| **装配关系** | Condition与DPU之间的关系。Condition可随时调整而不影响DPU内部实现。 |
| **静态规则** | 开发阶段一次性配好的DPU+Condition+produces定义。是游戏的完整规则集。不随运行时组合变化。 |
| **动态组装** | 运行阶段从全量DPU池中选择激活哪些DPU子集。每轮运行可不同。Condition天然容错——未激活DPU的produces字段缺失时视为false。 |
| **全量DPU池** | 游戏中所有已定义的DPU的集合。开发阶段逐步扩充。运行阶段从中选择子集激活。 |
| **本质复杂度** | 问题域固有的、无法消除的复杂度。模型规范只应包含本质复杂度。 |
| **偶然复杂度** | 因表达方式或技术选择引入的额外复杂度。可消除、可替换。不应进入模型规范。 |
| **三步检验** | 判断概念是否为本质的方法：①删除测试 ②来源追溯 ③换表达验证。 |

---

*模型的核心叙事：三个本质概念——DPU是自洽的积木块，声明自己写入DC的字段（produces）；Condition是装配在DPU上的执行门控，引用DC字段作为条件（隐含consumes），缺失字段视为false；DC是所有积木块共享的工作台。系统根据produces和consumes自动推导依赖关系并拓扑排序。开发时配规则（创建DPU+配置Condition），运行时配组合（选择激活哪些DPU）——规则不随组合变化。涌现是DPU+Condition在DC上运行的自然结果。设计者的核心工作是识别"什么是独立机制、什么是机制间的涌现"——这是本质复杂度，不可委托。*
