# 第一部分：WHY——动作游戏伤害管线的本质原则

## 一、四条设计思想

动作游戏的伤害管线有其固有的设计特征——不是工具的选择，不是实现的偏好，而是这类游戏的体验本质所要求的。以下四条思想是伤害管线问题域的WHY。

| 编号 | 思想 | 关切层面 | 核心表述 |
|------|------|---------|---------|
| **①** | **机制优先** | 单机制·设计 | 体验由机制驱动，不是由数值驱动。先有"格挡""弹刀""雷反"这些体验机制，再有数值填充。 |
| **②** | **数值兜底** | 单机制·实现 | 数值不驱动体验，但保证体验的数学基础正确。HP公式、架势公式、毒蓄积公式——它们是机制的数学地基。 |
| **③** | **反馈清晰可感** | 单机制·表现 | 每个机制的结果必须通过多模态表现（动画、音效、特效、位移、镜头）被玩家感知到。玩家感知不到的机制等于不存在。 |
| **④** | **涌现自治** | 多机制·协作 | 多个机制同时生效时，组合效果不由设计者穷举，而由各机制在共享上下文上自然产生。 |

①②③保证每个积木块的质量——从设计到实现到表现的纵向全链路。④保证积木块组合后的系统质量——多机制共存和组合的横向协作标准。

### 1.1 四条思想之间的关系

```
① 机制优先（设计层）
    ↓ 机制确定后
② 数值兜底（实现层）
    ↓ 实现正确后
③ 反馈清晰可感（表现层）
    ↓ 单机制完整后
④ 涌现自治（多机制组合层）
    → 多个完整的机制在一起时，组合效果自治产生
```

①②③是纵向的（单机制从设计到落地），④是横向的（多机制组合）。先有好的积木块（①②③），才有好的组合（④）。

# 第二部分：WHAT——问题域模型

## 二、问题域界定

### 2.1 两阶段架构

一次伤害事件的处理分为两个独立阶段，分别对应设计思想①②④和③。

```
Attack命中
→ 【机制阶段】各机制的判定和结算（对应①②④）
→ 【表现选取阶段】多机制结果的表现择优（对应③）
→ 表现执行
```

### 2.2 机制阶段

解决"这次攻击命中后，哪些机制生效、产生什么结果"。

核心关切：机制优先（①）——先确定哪些机制参与。数值兜底（②）——各机制内部的数值结算正确。涌现自治（④）——多机制同时生效时，组合效果自然产生。

### 2.3 表现选取阶段

解决"多个机制同时生效时，有限的表现手段如何分配给玩家感知"。

核心关切：反馈清晰可感（③）——玩家必须感知到最重要的机制结果。

### 2.4 两阶段的关系

两阶段通过共享上下文（DC）连接——机制阶段写入DC，表现选取阶段读取DC。表现选取不写DC、不影响机制结果。

### 2.5 "机制涌现"与"表现组合"的本质区别

两者结构相似——都是"条件A+条件B→产出C"。**但设计目标完全不同：**

| 维度 | 机制涌现（④的产物） | 表现组合（③的产物） |
|------|-------------------|-------------------|
| **目标** | 产生新的游戏规则/结算逻辑 | 让玩家更清晰地感知已有结果 |
| **如果不做** | 游戏逻辑错误 | 游戏逻辑正确，但反馈不够清晰 |
| **新增的是什么** | 新的机制行为 | 新的表现资源 |

```
机制涌现：Lightning + InAir → 空中接雷不受伤（改变了游戏规则）
表现组合：Death + Lightning → 雷电死亡动画（反馈更清晰，死亡判定不变）
```

结构相同不等于语义相同。混淆两者会导致修改动画方案时影响机制逻辑。

## 三、涌现

涌现是多个机制组合产生的效果（设计思想④的具体体现）。

### 3.1 涌现的本质认知

**机制处理体内部的每一条if-else分支，都可能是另一个独立机制的产出与当前机制的涌现。** 识别"什么是独立机制、什么是机制间的涌现"是设计者的核心工作，不可由工具或AI代劳。

```
示例：DPU_Lightning 内部的 if(IsInAir)

原始理解：这是Lightning的内部规则
深层理解："角色在空中"是一个独立机制（动作系统的产出），
          "空中 + 雷击"是两个机制的涌现
          → IsInAir应该是共享上下文中的独立事实
          → "空中雷击的特殊处理"应该是一个独立的机制处理体
```

### 3.2 涌现的多维度来源

涌现不只来自伤害管线内部——它可能跨越多个系统维度：

| 维度 | 示例 |
|------|------|
| **伤害管线机制** | Death、Collapse、Guard、Lightning、Poison |
| **角色状态** | IsInAir、IsOnGround、IsSuperArmored |
| **角色类型** | IsPlayer、IsNPC |
| **玩家输入** | AttackInput、DashInput |

## 四、表现选取

### 4.1 问题陈述

一次伤害事件中，多个机制同时生效，各自产出表现信息。但表现手段有限——不能同时播两个全身受击动画，玩家的认知带宽也有限。

### 4.2 三个本质概念

| 概念 | 定义 |
|------|------|
| **Channel** | 表现通道。同一Channel内互斥（只能选一个），不同Channel间可叠加（同时呈现）。 |
| **Priority** | Channel内的选取优先级。数值越高越优先——"玩家越需要感知到这个机制"。支持basePriority继承+逐Channel覆盖。 |
| **CombinedPresentation** | 表现组合。一个表现资源同时刻画多个机制的结果。不是机制涌现，是反馈优化。 |

### 4.3 Channel

Channel是表现通道——一个可独立控制的表现维度。

```
Channel示例：
  FullBodyAnim  —— 全身受击动画（互斥）
  AdditiveAnim  —— 加法层动画（可叠加）
  SFX           —— 音效（可叠加）
  VFX           —— 特效（可叠加）
  Motion        —— 位移/击退（可叠加）
  CameraShake   —— 镜头震动（可叠加）
```

Channel的定义是项目层面的设计决策——不同游戏可能有不同的Channel划分。

### 4.4 Priority

每个机制处理体可声明basePriority，其所有Channel默认继承。允许逐Channel覆盖。

```
DPU_Poison {
  basePriority: 50
  表现: {
    FullBodyAnim: { priority: 继承50, asset: Poison_Writhe }
    SFX:          { priority: 覆盖85, asset: poison_gurgle }  ← 毒音效特别重要
  }
}
```

### 4.5 CombinedPresentation

一个表现资源同时刻画多个机制的结果。当其源机制全部生效时，替代各自的单独表现。

```
CombinedPresentation:
  sources: [Death, Lightning]
  FullBodyAnim: Death_Electrocute（雷电死亡动画）
```

### 4.6 选取算法

```
对于每个Channel：
  1. 收集所有机制处理体在该Channel上的表现信息
  2. 检查CombinedPresentation且其源机制都已生效 → 有则选它
  3. 否则按Priority选最高的
  
最终输出 = 所有Channel各自选取结果的叠加
```

# 第三部分：HOW——L3声明式编排工具

## 五、工具层概述

本部分定义用于实现模块化伤害管线的L3声明式编排工具。工具的设计目标是忠实表达第一部分（WHY）的设计思想和第二部分（WHAT）的问题域模型。

工具层的核心概念：

| 概念 | 对应的问题域需求 |
|------|----------------|
| **DPU（机制处理体）** | 承载独立机制的判定和结算逻辑（①②） |
| **Fact（结构化产出）** | DPU的产出——有类型定义的数据结构，自带领域查询方法 |
| **DC（共享上下文）** | 机制间的数据通信，涌现的载体（④）。以DPU身份标识为索引存储Fact实例。 |
| **Condition（执行条件门控）** | 机制间涌现的条件表达（④）。条件树结构，引用DPU名+领域方法。 |
| **Produces/Consumes** | 依赖关系声明，自动确定执行顺序。Condition引用DPU名，自动建立依赖。 |
| **Channel/Priority** | 表现选取的分组和择优（③） |

## 六、核心概念

### 6.1 DPU（Damage Processing Unit）

机制处理体——自洽的积木块。读取DC、执行机制逻辑、将结果作为Fact写回DC。

#### Fact——DPU的结构化产出

DPU的产出不是散装的键值对，而是**Fact**——有类型定义的结构体，自带领域查询方法。每个DPU产出一个Fact，以**DPU自身的身份标识**存入DC。

| 概念 | 是什么 | 存在时机 |
|------|--------|---------|
| **Fact类型** | UStruct结构体定义（数据字段+领域方法） | 编译时 |
| **Fact实例** | DPU执行后创建的结构体实例，以DPU身份标识存入DC | 运行时 |

**Fact不存在独立的Key概念**——DC中Fact的索引键就是产出它的DPU的身份标识。使用者的思考路径是"我依赖DPU_Guard的结果"，而不是"我依赖一个叫GuardResult的Key"。

**Fact的两个组成部分**：

| 组成 | 是什么 | 谁使用 |
|------|--------|--------|
| **数据字段** | 机制处理结果（Bool/Int/Float/Enum） | DPU内部逻辑、后续DPU |
| **领域方法** | 有业务含义的查询函数，返回Bool | Condition条件树 |

```
FMixupResult {
  // 数据字段
  ResultType: EMixupType         // Guard, JustGuard, Hurt, Mikiri, Jump, Lightning
  DamageElement: EDamageElement   // None, Lightning, Poison, Fire
  DamageLevel: int
  AttackPower: float
  
  // 领域方法（通过FactMethodRegistry注册）
  IsGuard()       → ResultType == Guard || ResultType == JustGuard
  IsJustGuard()   → ResultType == JustGuard
  IsLightning()   → DamageElement == Lightning
  IsHeavyAttack() → DamageLevel > 2
}
```

**两种Fact形态**：

| 形态 | 结构 | 示例 | Condition中的引用 |
|------|------|------|------------------|
| **复杂Fact** | 多字段+领域方法 | FMixupResult, FGuardResult | `Mixup.IsGuard` `Guard.IsParry` |
| **信号Fact** | 纯存在性标记 | FLightningInAirResult | `LightningInAir`（存在即true） |

#### Produces声明

每个DPU声明自己产出的**Fact类型**：

```
DPU_Guard {
  produces: FGuardResult
  Condition: Mixup.IsGuard
  逻辑：格挡判定
}

DPU_LightningInAir {
  produces: FLightningInAirResult
  Condition: Mixup.IsLightning && IsInAir
  逻辑：空中蓄电
}
```

Fact在DC中以DPU名为索引存储——不需要单独声明Fact Key。DPU身份标识→Fact类型的映射自然存在于produces声明中。

#### 多个互斥DPU产出同类概念

多个互斥DPU各自产出自己的Fact，消费者用OR组合：

```
DPU_Collapse         → 产出自己的Fact
DPU_CollapseGuard    → 产出自己的Fact

DPU_LightningCollapse Condition:
  OR
  ├─ Collapse.IsCollapsed
  └─ CollapseGuard.IsCollapsed
```

规模膨胀（10+个互斥DPU）时，应引入**汇总DPU**统一产出，消费者只依赖汇总节点。

#### 领域方法注册（FactMethodRegistry）

Fact的领域方法通过全局注册表注册，映射(Fact类型, 方法名) → 查询函数。支持C++模板+lambda和蓝图RegisterBP+UFunction两条注册路径。

领域方法的约束：纯函数（不修改任何状态）、返回Bool、命名有领域含义、C++/蓝图均可定义。

#### DPU产出中的表现信息

DPU产出的Fact中可能包含表现信息，需标注Channel和Priority。DPU不关心这些信息最终会不会被播放。

#### DPU产出的双向作用目标

某些DPU的产出需要同时描述对多个作用目标的影响（如弹刀同时影响受击方和攻击方）。

#### DPU的约束规则

| 规则 | 说明 |
|------|------|
| **自洽性** | 内部逻辑完整独立。不知道自己被什么Condition门控，不知道其他DPU的存在。 |
| **只通过DC通信** | 不直接调用其他DPU。上下游影响通过DC间接实现。 |
| **不感知管线结构** | 不知道自己在管线中的位置、被什么Condition门控。 |
| **声明produces** | 必须显式声明产出的Fact类型。 |

### 6.2 DC（DamageContext）

共享上下文——本次事件的完整数据环境。以DPU身份标识为索引存储Fact实例。

#### DC中的两类内容

| 类别 | 是什么 | 来源 |
|------|--------|------|
| **上下文信息** | "这次事件是什么"的原始数据 | 参数表、角色属性、碰撞结果 |
| **DPU产出（Fact）** | "什么事被处理了，结果是什么" | DPU执行后以DPU身份标识写入 |

#### 存储结构

DC内部以`TMap<FName, FInstancedStruct>`存储——DPU身份标识（FName）作为索引，FInstancedStruct持有该DPU产出的Fact实例。

#### 特征

| 特征 | 说明 |
|------|------|
| 生命周期 | 瞬时——事件开始时创建，结束时销毁 |
| 可见性 | 所有DPU和Condition均可读取DC中的任何内容 |
| 写入权 | 只有DPU可以写入。Condition不写入。 |
| Fact缺失 | DPU身份标识在DC中不存在时视为false（DPU未执行或未加入管线） |

#### 上下文信息的分类

| 来源 | drawio颜色 |
|------|-----------|
| **攻击方** | 浅红 #f8cecc / 红框 #b85450 |
| **受击方** | 浅绿 #d5e8d4 / 绿框 #82b366 |
| **事件上下文** | 浅灰 #f5f5f5 / 灰框 #666666 |

### 6.3 Condition

执行条件门控——读取DC中的Fact，输出true/false，决定DPU是否执行。

#### 特征

| 特征 | 说明 |
|------|------|
| **可读范围** | DC中的一切（上下文信息+所有Fact） |
| **输出** | true/false |
| **写入权** | 无 |
| **与DPU的关系** | 装配关系。可随时调整而不影响DPU内部实现。 |
| **Fact缺失** | 引用的DPU在DC中没有Fact时视为false（DPU未执行或未加入管线——天然容错） |

#### 条件树结构

Condition以**条件树**形式配置（属性面板内联编辑，不需要写字符串或蓝图代码）：

| 节点类型 | 逻辑 |
|---------|------|
| **AND** | 所有子节点为true才true |
| **OR** | 任一子节点为true即true |
| **FactQuery** | 引用DPU名 + 领域方法名 |
| **Compare** | 引用DPU名 + 字段名 + 运算符 + 值 |

所有节点都有**bReverse取反开关**。

```
属性面板中的呈现示例——DPU_Death的Condition：

  ▼ [AND]
    ├─ [FactQuery] DPU: [Mixup ▾] Method: [IsGuardSuccess ▾]  bReverse: ✓
    └─ [FactQuery] DPU: [LightningInAir ▾] Method: (无)       bReverse: ✓

等价于：!Mixup.IsGuardSuccess && !LightningInAir
策划读法："博弈结果不是格挡成功，且不是空中接雷"
```

#### FactQuery节点的联动下拉

用户选DPU名后，MethodName下拉自动联动——只显示该DPU的Fact类型已注册的领域方法：

```
用户选 DPU = "Guard"
→ 从DPU池中找到DPU_Guard → 其produces类型为FGuardResult
→ 从FactMethodRegistry中查到 FGuardResult 有 ["IsParry", "IsGuardBreak"]
→ Method下拉显示这两个选项
```

#### 表达力

Condition表达式通过条件树节点组合，支持：

| 能力 | 实现方式 | 示例 |
|------|---------|------|
| 布尔逻辑 | AND/OR节点 + bReverse | `!Mixup.IsGuard && !LightningInAir` |
| 领域查询 | FactQuery节点 | `Guard.IsParry` |
| 数值比较 | Compare节点（>, <, >=, <=, ==, !=） | `Mixup.DamageLevel > 2` |
| 多源OR | OR节点组合多个互斥DPU | `Collapse \|\| CollapseGuard \|\| CollapseJustGuard` |

#### 产销关系自动提取

每个条件树节点实现GetConsumedDPUs()——递归收集所有引用的DPU名：

```
AND.GetConsumedDPUs()       → 递归所有Children → 合并去重
OR.GetConsumedDPUs()        → 同上
FactQuery.GetConsumedDPUs() → return { DPUName }
Compare.GetConsumedDPUs()   → return { DPUName }
```

**完全自动，无需手动声明consumes。**

#### 无Condition的DPU

DPU可以没有Condition——此时永远执行。

#### 新增涌现时的操作

1. 添加新DPU（如果涌现产生了新行为）
2. 配置Condition——可能需要调整已有DPU的Condition以实现互斥

已有DPU的内部实现不修改。改变的只是Condition装配逻辑。

## 七、依赖关系与自动排序

### 7.1 产销声明

| 声明 | 来源 | 含义 |
|------|------|------|
| **Produces** | DPU显式声明的Fact类型 | "我产出这个Fact" |
| **Consumes** | 从Condition条件树递归提取（GetConsumedDPUs） | "我依赖这些DPU的Fact" |

### 7.2 自动拓扑排序

系统根据产销关系自动生成执行顺序：

1. 每个DPU的produces标识了自己（DPU身份标识 = DC中Fact的索引）
2. 从每个DPU的Condition条件树中递归提取依赖的DPU名列表
3. 匹配：被依赖的DPU → 当前DPU → 建立有向边
4. 拓扑排序：被依赖方在依赖方之前执行
5. 无依赖关系的DPU之间顺序不确定

```
DPU_Guard        身份标识: "Guard"
DPU_CollapseJG   Condition中GetConsumedDPUs(): ["Guard"]
                 → "Guard"匹配DPU_Guard
                 → DPU_Guard必须在DPU_CollapseJG之前执行
```

产销关系直接用**DPU身份标识**匹配——不需要中间的Key层。

### 7.3 DC初始字段

DC的上下文信息在管线开始前就已填入——没有DPU生产者。Condition引用这些字段不产生依赖边。

### 7.4 防环规则

两个DPU互相依赖对方的produces → 它们不是独立机制 → 应该合并。

## 八、静态规则与动态组装

### 8.1 两阶段分离原则

| 阶段 | 谁做 | 做什么 | 频率 |
|------|------|--------|------|
| **开发阶段：配规则** | 设计者 | 创建DPU、声明produces、配置Condition | 一次性 |
| **运行阶段：配组合** | 系统/玩家 | 从全量DPU池中选择激活哪些DPU子集 | 每轮不同 |

**规则不随组合变化。** Condition是静态的游戏规则，DPU的激活子集是动态的玩家选择。两者完全解耦。

### 8.2 开发阶段

新机制首次加入时：

1. 创建新DPU，声明produces
2. 配置新DPU的Condition
3. 调整受影响的已有DPU的Condition

做完后规则永不再改。

### 8.3 运行阶段

1. 确定本轮激活的DPU子集
2. 对激活DPU做拓扑排序
3. 依次执行——Fact缺失视为false

```
没有空中感电Boon → DPU_LightningInAir不在管线中
→ "LightningInAir"的Fact永远不写入DC
→ Death的Condition中 !LightningInAir → 缺失视为false → !false = true → Death正常执行

有空中感电Boon → DPU_LightningInAir加入管线
→ 空中被雷击时写入LightningInAir的Fact → Death的Condition中 !LightningInAir = false → Death跳过

两轮中Death的Condition完全相同，没有任何修改。
```

## 九、本质约束规则

以下规则是工具层的本质约束——删掉任何一条，工具的表达能力会受损。

| 编号 | 规则 |
|------|------|
| **R1** | **DC是唯一通信介质**：DPU之间不直接调用，所有数据交换通过DC。 |
| **R2** | **DPU自洽**：DPU内部逻辑完整且独立。不知道自己被什么Condition门控，不知道其他DPU的存在。 |
| **R3** | **Condition可读DC全部内容，Fact缺失视为false**：输出true/false。不写入任何数据。Condition引用的DPU在DC中没有Fact时按"不满足"处理（天然容错——DPU动态增减时无需修改已有Condition）。 |
| **R4** | **Condition与DPU是装配关系**：新增涌现=调整Condition+添加新DPU，不调整已有DPU内部实现。 |
| **R5** | **执行顺序由依赖关系决定，支持动态管线**：每个DPU产出以自身为标识的Fact，Condition条件树递归提取依赖的DPU列表（GetConsumedDPUs），自动拓扑排序。管线可运行时动态组装不同DPU子集。 |
| **R6** | **禁止循环依赖**：互相依赖=不是独立机制=合并为一个DPU。 |

# 第四部分：方法论

## 十、设计方法论

### 10.1 Essential/Accidental Complexity

**Essential Complexity（本质复杂度）**：问题域固有的、无法消除的复杂度。

**Accidental Complexity（偶然复杂度）**：因表达方式或技术选择引入的额外复杂度。可消除、可替换。

**核心指引**：模型规范中只应包含本质复杂度。偶然复杂度属于实现层、工具层、教学层。

**信号识别**：当你感到设计"拧巴"时，大概率是在和偶然复杂度搏斗。

### 10.2 三步检验程序

每次引入新概念时：

**第一步：删除测试** — "删掉它，丢什么？"丢了=本质，没丢=偶然。

**第二步：来源追溯** — "长出来的还是借来的？"借来的要额外验证。

**第三步：换表达验证** — "换种方式表达，它还在吗？"还在=本质，不在=偶然。

### 10.3 涌现识别清单

每次在DPU内部写if-else时：

> "这个if的条件部分，是不是另一个独立机制的产出？"

是→应该提取为DC中的独立事实+独立DPU+Condition。
不是（只是本DPU内部的数值计算中间结果）→留在内部。

**判断标准：这个条件未来是否可能被其他机制独立引用。** 这个判断只有设计者能做。

### 10.4 偶然复杂度的产生规律

| 版本 | 引入的偶然复杂度 | 根因 |
|------|----------------|------|
| v1.7 | Fact Key自动DAG | 抽象过高，超出认知带宽 |
| v2.0 | 多输出口、连线分流、拦截链 | 用拓扑穷举涌现 |
| v2.5 | 词缀选取表、缺省逻辑、装饰器 | 用中间层包装本质概念 |
| v2.5→v4.2 | 文档标题"声明式机制编排框架" | 工具层身份替代了内容域身份 |
| v3.0–v3.1 | 手动线性序列 | 依赖关系是本质的，手动排序是偶然的 |
| v3.0–v4.3 | DC扁平键值对+字符串Condition | DPU产出的结构被展平，Condition拼字符串——可用但可读性和扩展性不足 |
| v4.4 Fact v1 | 独立Fact Key概念 | 从v1.7借来，不是当前使用者需求长出来的。使用者想说"我依赖DPU_Guard"，不需要中间的Key层 |

### 10.5 AI协作中的注意事项

**设计者的模糊表述会把AI带向偶然复杂度。** AI给出方案后用三步检验审视。

**分工原则**：
- 本质复杂度的识别和判断：设计者的工作，不可委托
- 偶然复杂度的实现（文档、代码、格式检查）：AI的工作，设计者做价值判断

### 10.6 内容域与工具层的区分

本文档涉及两个维度——内容域（模块化伤害管线）和工具层（L3声明式编排）。这两个维度描述同一件事的不同侧面，但不能混淆或替代。

| 维度 | 是什么 | 具体内容 |
|------|--------|---------|
| **内容域（WHAT）** | 模块化伤害管线 | 四条设计思想、机制间涌现、表现选取——要解决的游戏设计问题 |
| **工具层（HOW）** | L3声明式编排 | DPU+Condition+DC+produces/consumes+拓扑排序——解决问题的工具方法 |

**工具服务于内容域，不是反过来。** 文档标题应该是内容域（你造的是什么），不是工具层（你用什么造的）。

## 十三、术语表

| 术语 | 定义 |
|------|------|
| **模块化伤害管线** | 本项目的内容域。动作游戏中"攻击命中后发生什么"的完整处理流程。 |
| **L3声明式编排** | 本项目的工具层。基于DPU+Condition+DC的声明式积木块编排方法。 |
| **DPU** | 机制处理体。自洽的积木块——声明produces的Fact类型，读DC、执行逻辑、将Fact以DPU身份标识写入DC。 |
| **DPU身份标识** | DPU的FName。同时用于：DC中Fact的索引键、Condition中的引用、产销关系匹配。 |
| **Fact类型** | UStruct结构体定义。包含数据字段和领域方法。由DPU开发者定义。 |
| **Fact实例** | DPU执行后创建的结构体实例。以DPU身份标识写入DC。 |
| **领域方法** | Fact上的查询函数。返回Bool，名字有业务含义。通过FactMethodRegistry注册。 |
| **信号Fact** | 无内部字段的Fact。存在即true，缺失即false。 |
| **FactMethodRegistry** | 全局方法注册表。映射(Fact类型, 方法名)→查询函数。支持C++/蓝图双注册路径。 |
| **DC** | 共享上下文。以DPU身份标识为索引存储Fact实例（FInstancedStruct）。Fact缺失视为false。 |
| **Condition** | DPU的装配门控。条件树结构（AND/OR/FactQuery/Compare），属性面板内联编辑。引用DPU名+领域方法。Fact缺失视为false。 |
| **条件树** | Condition的配置结构。节点嵌套组成，递归GetConsumedDPUs()自动收集依赖的DPU列表。 |
| **FactQuery** | 条件树原子节点。持有DPU名+MethodName，通过Registry调用领域方法。 |
| **Compare** | 条件树原子节点。持有DPU名+FieldName+比较运算符+数值。 |
| **Produces** | DPU声明产出的Fact类型。生产者端。 |
| **Consumes** | Condition条件树中引用的DPU列表。消费者端。自动递归提取。 |
| **涌现** | 多个机制组合产生的效果。DPU+Condition在DC上运行的自然结果。 |
| **汇总DPU** | 多个互斥DPU产出同类概念时，可引入的统一结算节点。设计选择，非必须。 |
| **Channel** | 表现通道。通道内互斥，通道间叠加。 |
| **Priority** | Channel内选取优先级。支持basePriority继承+逐Channel覆盖。 |
| **CombinedPresentation** | 表现组合。同时刻画多个机制结果。不是机制涌现，是反馈优化。 |
| **静态规则** | 开发阶段配好的DPU+Condition+produces。不随运行时组合变化。 |
| **动态组装** | 运行阶段选择激活哪些DPU子集。Condition天然容错。 |
| **本质复杂度** | 问题域固有的复杂度。不可消除。 |
| **偶然复杂度** | 表达方式引入的额外复杂度。可消除。 |

*模型的核心叙事：动作游戏的伤害管线由四条设计思想驱动——机制优先、数值兜底、反馈清晰可感、涌现自治。管线分为机制阶段和表现选取阶段。机制阶段用L3声明式编排工具实现——DPU是自洽的积木块（声明produces的Fact类型），Fact是DPU的结构化产出（数据字段+领域方法），Condition是装配门控（条件树结构，直接引用DPU名+领域方法，Fact缺失视为false），DC以DPU身份标识为索引存储Fact实例。系统从条件树递归提取依赖的DPU列表，自动拓扑排序，支持动态管线。表现选取阶段按Channel分组、按Priority择优。开发时配规则，运行时配组合，两者解耦。*
