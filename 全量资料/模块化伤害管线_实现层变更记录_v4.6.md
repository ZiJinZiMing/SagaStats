# 模块化伤害管线：实现层变更记录 v4.6

> **文档性质**：实现层变更记录。记录v4.6实现层相对于v4.7模型规范文档的架构差异。
> **目标读者**：负责维护模型规范文档的网页Claude。
> **目的**：让模型层文档在后续版本中能准确反映实现层的决策。

---

## 一、术语对齐（已完成）

实现层已完成与v4.7模型文档的术语对齐：

| 模型文档概念 | 实现层类名 | 说明 |
|---|---|---|
| DamageRule | `UDamageRuleDefinition` | DataAsset形式的规则定义 |
| RuleName | `UDamageRuleDefinition::RuleName` | DamageRule身份标识（FName） |
| DamageOperation | `UDamageOperationBase` | 逻辑基类，子类重写Execute |
| DamageEffect | USTRUCT（如`FMixupEffect`） | 规则产出的结构化数据 |
| DamageCondition条件树 | `UDamagePredicate` + `UDamageCondition` | 见差异1 |
| DamagePipeline | `UDamagePipeline` | 持有DamageRules + Build() + Execute() |
| DC | `UDamageContext` | 共享上下文（不变） |

## 二、实现层与模型文档的架构差异

以下7个差异是实现层在迭代过程中做出的架构决策，与当前v4.7模型文档的描述不同。每个差异都有明确的WHY，建议模型文档在后续版本中评估是否需要更新。

### 差异1：双层条件树（Predicate + Condition）

**模型文档描述**：单层——`UDamageConditionNode`作为统一基类，AND/OR复合节点和叶子节点（按DamageRule分的子类）都直接继承它。

**实现层实际**：双层——
- `UDamagePredicate`（容器层）：AND/OR/Single三种组合方式。持有bReverse取反开关。
- `UDamageCondition`（原子层）：每个子类IS一个域检查。不持有bReverse。

```
UDamagePredicate (容器基类, 有 bReverse)
├── UDamagePredicate_Single  → 包装一个 UDamageCondition
├── UDamagePredicate_And     → 组合多个 UDamagePredicate
└── UDamagePredicate_Or      → 组合多个 UDamagePredicate

UDamageCondition (原子基类, 无 bReverse)
├── UDamageCondition_IsGuard          → 消费FMixupEffect，判断bIsGuard
├── UDamageCondition_GuardSuccess     → 消费FGuardEffect，判断Success
├── UDamageCondition_ContextCheck     → 检查DC事件上下文字段
└── ...（每个域方法一个子类）
```

**WHY**：参照AIToken插件的`UAITokenConditionPredicate` + `UAITokenCondition`双层模式。容器负责组合逻辑（AND/OR/取反），原子只负责单一的域检查。职责分离更清晰。

**影响**：模型文档中"条件树结构"一节需要区分两层概念。

### 差异2：子类即域方法（无MethodName）

**模型文档描述**：每个DamageRule对应一个DamageConditionNode子类，子类上定义多个领域方法（UFUNCTION），通过MethodName下拉选择。

**实现层实际**：去掉了MethodName。每个域方法是一个独立的DamageCondition子类。选子类类型 = 选域方法。

```
模型文档：
  UDamageConditionNode_Guard
    ├── UFUNCTION: IsParry()       ← MethodName下拉选择
    └── UFUNCTION: IsGuardBreak()

实现层：
  UDamageCondition_GuardSuccess     ← 选这个类 = 判断格挡成功
  UDamageCondition_GuardIsJustGuard ← 选这个类 = 判断弹刀
```

**WHY**：
1. 选类型=选域方法，属性面板中一步完成，比"选类型+选方法名"更直接
2. 避免了反射调用域方法的复杂度（CallDomainMethod、IsValidDomainMethod等偶然复杂度）
3. 每个DamageCondition子类的Evaluate签名接收`const FInstancedStruct& ConsumedFact`——框架自动从DC取出对应DamageEffect传入，子类直接读取，不需要自己查DC

**影响**：模型文档中"DamageConditionNode按DamageRule分子类"和"领域方法UFUNCTION+方法名下拉"的描述需要调整。子类数量会比模型文档描述的多（每个域方法一个类而非每个DamageRule一个类）。

### 差异3：UScriptStruct*（EffectType）作为DC键

**模型文档描述**：DC以`TMap<FName, FInstancedStruct>`存储DamageEffect，FName（RuleName）作为索引键。

**实现层实际**：DC使用双存储架构：
- `TMap<FName, FInstancedStruct> ContextFacts` — 事件上下文（标量API：SetBool/GetFloat等）
- `TMap<UScriptStruct*, FInstancedStruct> DamageEffects` — DamageRule产出，以DamageEffect类型（UScriptStruct*）为键

```cpp
// C++模板API（编译时类型安全）
DC->SetEffect<FGuardEffect>(Result);
const FGuardEffect* Guard = DC->GetEffect<FGuardEffect>();

// 蓝图API（运行时类型安全）
DC->SetEffectByType(FGuardEffect::StaticStruct(), Value);
DC->GetEffectByType(FGuardEffect::StaticStruct());
```

**WHY**：EffectType是万能连接器，串联四个系统：
1. **DamageOperation** 声明 `GetProducesEffectType() → FGuardEffect::StaticStruct()`
2. **DC** 以 `FGuardEffect::StaticStruct()` 为键存储
3. **DamageCondition** 声明 `GetConsumedEffectType() → FGuardEffect::StaticStruct()`
4. **DamagePipeline::Build()** 匹配 ProducesEffectType == ConsumedEffectType 自动建立依赖

不需要通过FName间接关联，类型本身就是连接。同时，DamageRule:DamageEffect是1:1关系，类型唯一确定生产者。

**影响**：模型文档中"DC存储结构"和"产销关系收集"的描述需要更新。特别是`GetConsumedRules()`的实现机制——不再是"通过DamageEffect类型反查哪个DamageRule produces它"，而是"EffectType直接匹配"。

### 差异4：OutEffect直传的Execute签名

**模型文档描述**：DamageOperation执行后返回DamageEffect实例 → DamagePipeline以RuleName写入DC。

**实现层实际**：
```cpp
// DamageOperationBase
UFUNCTION(BlueprintNativeEvent)
void Execute(UDamageContext* DC, UPARAM(ref) FInstancedStruct& OutEffect);
```

Framework创建空的OutEffect（按ProducesEffectType默认初始化）传给Operation，Operation填充字段，Framework负责类型校验后写入DC。

```cpp
// DamagePipeline::Execute 中
FInstancedStruct OutEffect(EffectType);     // 框架创建
Logic->Execute(DC, OutEffect);              // Operation填充
if (OutEffect.IsValid() && OutEffect.GetScriptStruct() == EffectType)
    DC->SetEffectByType(EffectType, OutEffect);  // 框架写入DC
```

**WHY**：
1. **R2恢复**：DamageOperation不需要知道自己的身份标识（RuleName），因为不需要自己写DC。类型IS the key。
2. **框架负责校验**：OutEffect的类型必须与声明的ProducesEffectType一致，不一致则报错。防止Operation写错类型。
3. **蓝图友好**：蓝图子类在Execute中只需修改OutEffect的字段，不需要创建FInstancedStruct。

**影响**：模型文档中"DamageOperation执行后返回DamageEffect"的描述需要更新为"框架传入OutEffect，Operation填充"。

### 差异5：bReverse仅在容器层

**模型文档描述**：所有节点（复合+叶子）都有bReverse取反开关。

**实现层实际**：bReverse只在UDamagePredicate（容器层）上。UDamageCondition（原子层）没有bReverse。

```
属性面板呈现：
  [AND]                          bReverse: □
  ├── [Single]                   bReverse: ✓    ← 取反在这里
  │   └── IsGuard (Condition)                    ← 原子无bReverse
  └── [Single]                   bReverse: ✓
      └── LightningInAir (Condition)
```

**WHY**：
1. 双层bReverse容易混淆——Predicate取反AND整体，Condition取反单个条件，两层同时有bReverse时语义不明确
2. 单层bReverse在Predicate_Single上实现取反功能完全等价，且更简洁
3. 原子只回答"是/否"，取反是组合层的逻辑，不是原子的职责

**影响**：模型文档中"所有节点都有bReverse"的描述需要调整。

### 差异6：ConsumedEffectType隐藏实例面板

**实现层特有**：DamageCondition的`ConsumedEffectType`属性使用`meta=(HideInDetailPanel)`，在实例编辑面板上不可见、不可修改。

```cpp
UPROPERTY(EditDefaultsOnly, Category = "DamageCondition", 
    meta = (DisplayName = "Consumed Effect Type", HideInDetailPanel))
UScriptStruct* ConsumedEffectType = nullptr;
```

**WHY**：ConsumedEffectType是子类的类级属性（定义"我消费哪种DamageEffect"），C++子类通过`GetConsumedEffectType()`返回硬编码值，蓝图子类在类默认值中设置。实例层面不应该能修改——否则同一个Condition子类的不同实例可能消费不同类型，违反类型一致性。

### 差异7：Build时EffectType校验

**实现层特有**：`DamagePipeline::Build()`在拓扑排序前增加了EffectType校验：

1. 每个DamageRule的Operation必须配置ProducesEffectType（不为null）
2. Condition树中每个非ContextCheck的叶子节点必须配置ConsumedEffectType（不为null）
3. 运行时Execute中校验OutEffect类型与ProducesEffectType一致

**WHY**：早发现配置错误，比运行时出现难以追踪的null指针或类型不匹配更好。Build是低频操作，增加校验不影响执行性能。
