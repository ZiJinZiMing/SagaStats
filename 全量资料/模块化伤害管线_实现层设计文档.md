# 模块化伤害管线：实现层设计文档

> **文档性质**：实现层设计。定义运行时系统架构、编辑器UX方案、分步实施计划。基于模型规范v4.3。
> **内容域**：模块化伤害管线——动作游戏中"攻击命中后发生什么"的完整处理流程。
> **工具层**：基于L3声明式编排实现——DPU+Condition+DC+产销关系自动拓扑排序。
> **与模型规范的关系**：模型规范定义"是什么"（WHY+WHAT），本文档定义"怎么做"（HOW的实现细节）。本文档的所有内容都是偶然复杂度——可以被替换为其他实现方案而不影响模型的本质结构。

---

## 一、核心设计决策

### 1.1 运行时系统是本质，编辑器是偶然

| 组件 | Essential/Accidental | 依据 |
|------|---------------------|------|
| 运行时拓扑排序引擎 | **本质** | 删掉→回到手动排序→丢失表达能力 |
| Pipeline Asset（管线资产） | **本质** | 删掉→每次运行都要从头配DPU组合→无法复用 |
| 编辑器DAG视图 | 偶然（高优先级） | 删掉→设计者看不到管线全貌→体验差但功能不丢 |
| 运行时快照导出 | 偶然（高价值） | 删掉→无法在编辑器中调试特定Boon组合→体验差但功能不丢 |

### 1.2 图表是视图，不是数据

**数据层**：DPU定义存储在UDataAsset中（produces声明、Condition表达式、逻辑类引用、表现信息）。这些数据在设计时和运行时完全相同。

**视图层**：DAG图表从数据层自动生成。设计者编辑的是DataAsset，图表自动刷新——不需要手动连线。连线由produces/consumes产销关系自动推导。

**区别于EQS/行为树**：EQS和行为树的Graph是"编辑器即数据"——Graph本身就是数据存储。本框架中Graph只是数据的可视化投影。原因：

- 手动连线与自动拓扑排序矛盾（R5要求执行顺序由依赖关系自动决定）
- 运行时动态组装时没有预设的Graph可加载（肉鸽模式每轮DPU子集不同）

### 1.3 一套引擎，两种模式，Pipeline Asset双向流通

编辑器模式和运行时模式共用同一套引擎。Pipeline Asset作为两种模式之间的数据交换格式，支持**双向流通**——既可以从编辑器流向运行时（设计者Build保存→游戏加载执行），也可以从运行时流回编辑器（运行时快照导出→编辑器离线调试）。

```
编辑器模式：                          运行时模式：
  手动选DPU类                           从池中选激活子集（Boon决定）
  ↓                                    ↓
  同一套运行时引擎                       同一套运行时引擎
  自动拓扑排序                          自动拓扑排序
  ↓                                    ↓
  渲染DAG视图                           按顺序执行
  可预览/调试                           调试面板实时显示
  保存为Pipeline Asset                  可导出为Pipeline Asset
       │                                    │
       │  ──── 方向1：编辑器→运行时 ────→     │
       │        游戏加载Asset执行             │
       │                                    │
       │  ←─── 方向2：运行时→编辑器 ────      │
       │        快照导出回编辑器调试           │
```

编辑器中Build生成的Pipeline Asset和运行时动态生成的Pipeline Asset是**同一种资产类型**。编辑器可以打开运行时导出的快照进行离线调试。

## 二、三层架构

### 2.1 数据层：Pipeline Asset

Pipeline Asset是管线的静态定义——一组DPU类引用及其配置。

```
Pipeline Asset 包含：
  DPU列表：
    DPU_Mixup:
      LogicClass: UDPULogic_Mixup
      Produces: [MixupResult, damageElement]
      Condition: (无)
      BasePriority: 0
      
    DPU_Guard:
      LogicClass: UDPULogic_Guard
      Produces: [IsGuard, IsJustGuard, GuardSuccess, GuardLevel]
      Condition: Guard && !IsLightningInAir
      BasePriority: 60
      ChannelOverrides: {}
      
    DPU_LightningInAir:
      LogicClass: UDPULogic_LightningInAir
      Produces: [IsLightningInAir]
      Condition: Lightning && IsInAir
      BasePriority: 90
      
    ...（其他DPU）
```

Pipeline Asset有两种来源：

| 来源 | 方式 | 适用场景 |
|------|------|---------|
| **编辑器手动配置** | 设计者选择DPU类 → Build → 保存 | 只狼等固定管线游戏 |
| **运行时动态生成** | 系统根据激活Boon自动组装 → 可导出保存 | Hades等肉鸽游戏 |

两种来源生成的Asset格式完全相同，运行时引擎不区分。

### 2.2 引擎层：Pipeline Manager

Pipeline Manager是运行时的核心——接收Pipeline Asset（或DPU子集），自动拓扑排序，依次执行。

```
Pipeline Manager 的职责：

1. 接收输入：Pipeline Asset 或 激活DPU子集
2. 提取产销关系：从每个DPU的Produces和Condition中提取
3. 稳定拓扑排序：生产者在消费者之前，无依赖DPU保留数组原始顺序
4. 防环检测：发现循环依赖时报错
5. 依次执行：
   a. 评估Condition（DC字段缺失视为false）
   b. Condition为true → 执行DPU逻辑 → 写入DC
   c. Condition为false → 跳过
6. 执行完毕后：表现选取（按Channel/Priority择优）
7. 输出：最终表现指令
```

#### 稳定拓扑排序（Stable Topological Sort）

拓扑排序的基本特征：对于同一个DAG，满足依赖关系的合法拓扑序可能有多个。无依赖关系的DPU之间谁先谁后，标准拓扑排序不保证。

这会带来两个问题：

- **Build不确定性**：使用者什么都没改，每次Build可能产出不同顺序，造成困惑
- **漏声明依赖的放大**：使用者忘了声明某个产销关系，算法可能把顺序排反，产出不可预见的结果

**设计决策：采用稳定拓扑排序——在满足产销依赖的前提下，尽量保留DPU在数组中的原始顺序。**

```
实现方式（基于Kahn算法的BFS变体）：

标准Kahn算法：
  维护"入度为0"的候选队列 → 每次取任意一个输出

稳定变体：
  候选队列改为按DPU在原始数组中的下标排序的优先队列
  每次取下标最小的那个输出

效果：
  有依赖关系的DPU → 严格按依赖排（生产者在前）
  无依赖关系的DPU → 按数组原始顺序排（谁在前就谁先）
  同一个DPU池 + 同一个数组顺序 → 每次Build结果完全一致
```

稳定拓扑排序解决的三个问题：

| 问题 | 标准拓扑排序 | 稳定拓扑排序 |
|------|------------|------------|
| 未声明的隐含依赖 | 可能排出错误顺序 | 保留数组顺序作为兜底意图，降低影响 |
| Build确定性 | 每次可能不同 | 同输入同输出，完全确定 |
| 使用门槛 | 必须完美声明所有依赖 | 逐步补全即可，数组顺序兜底 |

**三步检验**：删掉"稳定"约束→表达能力不丢失，但使用体验显著下降。**结论：稳定拓扑排序是偶然复杂度（实现选择），但是极高价值的实现选择。**

**同一套代码**在编辑器Preview和运行时执行中使用——保证编辑器中看到的结果和实际游戏中一致。

### 2.3 视图层：编辑器模块

编辑器模块提供可视化的管线编辑和调试能力。

#### 编辑器呈现形式

参考UE5 EQS编辑器的Generator+Test列表形式——**DPU列表纵向排列，每个DPU节点展示其Condition、Produces、描述信息**。

```
编辑器视图结构（类似EQS的Generator+Test列表）：

  ┌──────────────────────────────────────┐
  │ Pipeline: 只狼伤害管线 V3.5           │  ← Pipeline Asset名称
  │ DPU Count: 15  |  [Build] [Preview]  │  ← 操作按钮
  ├──────────────────────────────────────┤
  │                                      │
  │  ┌─ Condition: (无) ──────────────┐  │
  │  │ DPU_Mixup                      │  │  ← 蓝色DPU节点
  │  │ produces: [MixupResult,        │  │
  │  │           damageElement]       │  │
  │  │ 博弈判定                        │  │  ← Description
  │  └────────────────────────────────┘  │
  │          │                           │
  │          ▼ (依赖连线，自动生成)        │
  │  ┌─ Condition: Lightning&&IsInAir ─┐  │  ← 黄色Condition
  │  │ DPU_LightningInAir             │  │
  │  │ produces: [IsLightningInAir]   │  │
  │  │ 空中蓄电                        │  │
  │  └────────────────────────────────┘  │
  │          │                           │
  │          ▼                           │
  │  ┌─ Condition: Guard&&!IsLi... ───┐  │
  │  │ DPU_Guard                      │  │
  │  │ produces: [IsGuard, ...]       │  │
  │  │ 格挡表现+架势                   │  │
  │  └────────────────────────────────┘  │
  │          │                           │
  │         ...                          │
  │                                      │
  └──────────────────────────────────────┘
```

#### 编辑器功能

| 功能 | 说明 |
|------|------|
| **DPU类选择** | 从项目中所有UDPULogic子类中选择，添加到管线 |
| **Condition编辑** | 直接编辑条件表达式，引用DC字段名 |
| **Produces编辑** | 编辑DPU声明写入的DC字段列表 |
| **Build** | 调用运行时引擎的拓扑排序，刷新DAG布局，检测环 |
| **Preview** | 用测试数据运行一遍管线，高亮执行/跳过的DPU，显示DC内容 |
| **保存** | 将当前配置保存为Pipeline Asset |
| **环检测** | Build时自动检测循环依赖，标红冲突的DPU对 |

#### 运行时调试面板

| 功能 | 说明 |
|------|------|
| **实时DAG** | 显示当前轮激活的DPU子集和拓扑排序结果 |
| **执行追踪** | 每次Attack命中时，高亮正在执行的DPU，灰显被跳过的 |
| **DC监视** | 实时显示DC中所有字段的当前值 |
| **快照导出** | 将当前轮的DPU子集导出为Pipeline Asset，可在编辑器中离线调试 |

## 三、分步实施计划

### Phase 1：运行时系统

**目标**：能跑通只狼V3.5的15个DPU管线。

**交付物**：

| 组件 | 类名 | 职责 |
|------|------|------|
| DC | `UDamageContext` | 共享上下文，TMap<FName, FDCValue>存储字段，缺失字段返回false |
| DPU定义 | `UDPUDefinition` (DataAsset) | 存储Produces、Condition表达式、LogicClass引用、BasePriority、表现信息 |
| DPU逻辑基类 | `UDPULogicBase` | 虚函数Execute(DC)，子类实现具体机制 |
| Condition | `UConditionExpression` | 解析条件表达式，读DC字段，输出bool，缺失字段视为false |
| 管线管理器 | `UPipelineManager` | 接收DPU子集→拓扑排序→依次执行→表现选取 |
| Pipeline Asset | `UPipelineAsset` (DataAsset) | 存储一组DPUDefinition的引用，可序列化 |

**验证方式**：

- 测试关卡中手动触发Attack
- 日志打印拓扑排序结果和DC内容
- 对照只狼V3.5的验证场景逐个确认：
  1. 普通斩击命中（无特殊机制）
  2. 弹刀（JustGuard）
  3. 格挡+架势满→破防
  4. 雷电+死亡→雷电死亡表现
  5. 空中接雷→不受伤

**扩展性预留**（为Phase 2准备，但Phase 1不实现）：

- UPipelineManager的拓扑排序逻辑独立为可复用函数（编辑器Build时调用同一函数）
- UDPUDefinition的Produces和Condition数据结构支持序列化/反序列化（编辑器需要读写）
- 日志输出格式规范化（编辑器调试面板可解析同一格式）

### Phase 2：编辑器模块

**前提**：Phase 1完全稳定。

**目标**：可视化DAG编辑器 + Build预览 + Asset管理。

**交付物**：

| 组件 | 说明 |
|------|------|
| Pipeline Asset编辑器 | 自定义Asset Editor，类似EQS的列表形式 |
| DAG自动布局 | 从Produces/Consumes自动推导依赖关系并渲染 |
| Build功能 | 调用Phase 1的拓扑排序函数，刷新视图，检测环 |
| Preview功能 | 用测试数据运行管线，高亮执行状态 |
| 运行时调试面板 | 实时DAG + DC监视 + 快照导出 |

## 四、关键设计约束

### 4.1 编辑器和运行时共用同一套引擎

编辑器的Build和Preview功能必须调用运行时系统的代码——不能有两套排序/执行逻辑。否则编辑器中看到的结果可能和实际运行不一致。

### 4.2 Pipeline Asset是唯一的数据交换格式

编辑器保存的Asset和运行时加载的Asset是同一种类型。运行时动态生成的管线也可以导出为同一种Asset。这保证了"编辑器中的管线"和"游戏中的管线"是同一个东西。

### 4.3 编辑器不存在于运行时

编辑器模块（Editor Module）在打包后的游戏中不存在。运行时系统（Runtime Module）独立于编辑器。编辑器依赖运行时，反过来不成立。

### 4.4 两种管线模式

| 模式 | DPU子集何时确定 | 拓扑排序时机 | 典型游戏 |
|------|---------------|------------|---------|
| **固定管线** | 设计时（编辑器Build保存为Asset） | 设计时一次 | 只狼、鬼泣 |
| **动态管线** | 运行时（每轮根据Boon确定） | 每轮开始时 | Hades、Slay the Spire |

两种模式的运行时执行逻辑完全相同——区别只在于Pipeline Asset的来源。固定管线从磁盘加载预制Asset，动态管线从全量DPU池中按激活子集实时组装。

## 五、与模型规范的映射

| 模型规范概念 | 实现层对应 |
|------------|----------|
| DPU | UDPUDefinition (DataAsset) + UDPULogicBase (逻辑子类) |
| DC | UDamageContext (TMap<FName, FDCValue>) |
| Condition | UConditionExpression (解析条件表达式) |
| Produces | UDPUDefinition::Produces (TArray<FName>) |
| Consumes | 从UConditionExpression中自动提取引用的字段名 |
| 拓扑排序 | UPipelineManager::TopologicalSort() |
| 防环检测 | UPipelineManager::DetectCycles() |
| 静态规则 | UDPUDefinition中的配置（设计时定义，运行时不变） |
| 动态组装 | UPipelineManager接受DPU子集作为输入 |
| 全量DPU池 | 项目中所有UDPUDefinition DataAsset的集合 |
| Channel | 表现信息中的FName标识 |
| Priority | UDPUDefinition::BasePriority + ChannelOverrides |
| CombinedPresentation | DPU产出中标注sources的表现信息 |
| Pipeline Asset | UPipelineAsset (DataAsset) |
