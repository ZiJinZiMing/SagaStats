# 模块化伤害管线：实现层设计文档 v4.4

> **文档性质**：实现层设计。定义运行时系统架构、编辑器UX方案、分步实施计划。基于模型规范v4.4 + Fact模型设计v1。
> **内容域**：模块化伤害管线——动作游戏中"攻击命中后发生什么"的完整处理流程。
> **工具层**：基于L3声明式编排实现——DPU+Fact+Condition条件树+DC+产销关系自动拓扑排序。
> **与模型规范的关系**：模型规范定义"是什么"（WHY+WHAT），本文档定义"怎么做"（HOW的实现细节）。本文档的所有内容都是偶然复杂度——可以被替换为其他实现方案而不影响模型的本质结构。

---

## 一、核心设计决策

### 1.1 运行时系统是本质，编辑器是偶然

| 组件 | Essential/Accidental | 依据 |
|------|---------------------|------|
| 运行时拓扑排序引擎 | **本质** | 删掉→回到手动排序→丢失表达能力 |
| Pipeline Asset（管线资产） | **本质** | 删掉→每次运行都要从头配DPU组合→无法复用 |
| 编辑器DAG视图 | 偶然（高优先级） | 删掉→设计者看不到管线全貌→体验差但功能不丢 |
| 运行时快照导出 | 偶然（高价值） | 删掉→无法在编辑器中调试特定Boon组合→体验差但功能不丢 |

### 1.2 图表是视图，不是数据

**数据层**：DPU定义存储在UDataAsset中（Produces声明、Condition条件树、逻辑类引用、表现信息）。这些数据在设计时和运行时完全相同。

**视图层**：DAG图表从数据层自动生成。设计者编辑的是DataAsset，图表自动刷新——不需要手动连线。连线由produces/consumes产销关系自动推导。

**区别于EQS/行为树**：EQS和行为树的Graph是"编辑器即数据"——Graph本身就是数据存储。本框架中Graph只是数据的可视化投影。原因：

- 手动连线与自动拓扑排序矛盾（R5要求执行顺序由依赖关系自动决定）
- 运行时动态组装时没有预设的Graph可加载（肉鸽模式每轮DPU子集不同）

### 1.3 一套引擎，两种模式，Pipeline Asset双向流通

编辑器模式和运行时模式共用同一套引擎。Pipeline Asset作为两种模式之间的数据交换格式，支持**双向流通**——既可以从编辑器流向运行时（设计者Build保存→游戏加载执行），也可以从运行时流回编辑器（运行时快照导出→编辑器离线调试）。

```
编辑器模式：                          运行时模式：
  手动选DPU类                           从池中选激活子集（Boon决定）
  ↓                                    ↓
  同一套运行时引擎                       同一套运行时引擎
  自动拓扑排序                          自动拓扑排序
  ↓                                    ↓
  渲染DAG视图                           按顺序执行
  可预览/调试                           调试面板实时显示
  保存为Pipeline Asset                  可导出为Pipeline Asset
       │                                    │
       │  ──── 方向1：编辑器→运行时 ────→     │
       │        游戏加载Asset执行             │
       │                                    │
       │  ←─── 方向2：运行时→编辑器 ────      │
       │        快照导出回编辑器调试           │
```

编辑器中Build生成的Pipeline Asset和运行时动态生成的Pipeline Asset是**同一种资产类型**。编辑器可以打开运行时导出的快照进行离线调试。

## 二、三层架构

### 2.1 数据层：Pipeline Asset

Pipeline Asset是管线的静态定义——一组DPU定义的引用。

每个DPU定义（UDataAsset）包含：

| 字段 | 内容 | 说明 |
|------|------|------|
| DPUName | FName | 身份标识 |
| Produces | Fact Key列表（TArray\<FName\>） | 声明写入DC的Fact Key，不绑定Fact类型（模型规范：Key→类型映射在哪声明是实现选择） |
| Condition | 条件树根节点（UConditionNode*, Instanced） | 属性面板内联编辑，不需要写代码 |
| LogicClass | DPU逻辑类引用 | 子类重写Execute(DC) |
| BasePriority | float | 表现选取的基础优先级 |

Pipeline Asset有两种来源：

| 来源 | 方式 | 适用场景 |
|------|------|---------|
| **编辑器手动配置** | 设计者选择DPU类 → Build → 保存 | 只狼等固定管线游戏 |
| **运行时动态生成** | 系统根据激活Boon自动组装 → 可导出保存 | Hades等肉鸽游戏 |

两种来源生成的Asset格式完全相同，运行时引擎不区分。

### 2.2 引擎层：Pipeline Manager

Pipeline Manager是运行时的核心——接收Pipeline Asset（或DPU子集），自动拓扑排序，依次执行。

```
Pipeline Manager 的职责：

1. 接收输入：Pipeline Asset 或 激活DPU子集
2. 提取产销关系：
   - 从每个DPU的Produces获取Fact Key列表（生产者）
   - 从每个DPU的Condition条件树递归GetConsumedFacts()获取Fact Key列表（消费者）
3. 稳定拓扑排序：生产者在消费者之前，无依赖DPU保留数组原始顺序
4. 防环检测：发现循环依赖时报错
5. 依次执行：
   a. 评估Condition条件树（Fact缺失视为false）
   b. Condition为true → 执行DPU逻辑 → 将Fact写入DC
   c. Condition为false → 跳过
6. 执行完毕后：表现选取（按Channel/Priority择优）
7. 输出：最终表现指令
```

#### 稳定拓扑排序（Stable Topological Sort）

拓扑排序的基本特征：对于同一个DAG，满足依赖关系的合法拓扑序可能有多个。无依赖关系的DPU之间谁先谁后，标准拓扑排序不保证。

这会带来两个问题：

- **Build不确定性**：使用者什么都没改，每次Build可能产出不同顺序，造成困惑
- **漏声明依赖的放大**：使用者忘了声明某个产销关系，算法可能把顺序排反，产出不可预见的结果

**设计决策：采用稳定拓扑排序——在满足产销依赖的前提下，尽量保留DPU在数组中的原始顺序。**

```
实现方式（基于Kahn算法的BFS变体）：

标准Kahn算法：
  维护"入度为0"的候选队列 → 每次取任意一个输出

稳定变体：
  候选队列改为按DPU在原始数组中的下标排序的优先队列
  每次取下标最小的那个输出

效果：
  有依赖关系的DPU → 严格按依赖排（生产者在前）
  无依赖关系的DPU → 按数组原始顺序排（谁在前就谁先）
  同一个DPU池 + 同一个数组顺序 → 每次Build结果完全一致
```

稳定拓扑排序解决的三个问题：

| 问题 | 标准拓扑排序 | 稳定拓扑排序 |
|------|------------|------------|
| 未声明的隐含依赖 | 可能排出错误顺序 | 保留数组顺序作为兜底意图，降低影响 |
| Build确定性 | 每次可能不同 | 同输入同输出，完全确定 |
| 使用门槛 | 必须完美声明所有依赖 | 逐步补全即可，数组顺序兜底 |

**同一套代码**在编辑器Preview和运行时执行中使用——保证编辑器中看到的结果和实际游戏中一致。

### 2.3 视图层：编辑器模块

编辑器模块提供可视化的管线编辑和调试能力。

#### 编辑器呈现形式

参考UE5 EQS编辑器的Generator+Test列表形式——**DPU列表纵向排列，每个DPU节点展示其Condition条件树、Produces Fact Key列表、描述信息**。

#### 编辑器功能

| 功能 | 说明 |
|------|------|
| **DPU类选择** | 从项目中所有DPULogic子类中选择，添加到管线 |
| **Condition条件树编辑** | 属性面板内联编辑——选节点类型（AND/OR/FactQuery/Compare）→ 选Fact Key → 选领域方法（联动下拉） |
| **Produces编辑** | 编辑DPU声明的Fact Key列表 |
| **Build** | 调用运行时引擎的拓扑排序，刷新DAG布局，检测环 |
| **Preview** | 用测试数据运行一遍管线，高亮执行/跳过的DPU，显示DC中的Fact内容 |
| **保存** | 将当前配置保存为Pipeline Asset |
| **环检测** | Build时自动检测循环依赖，标红冲突的DPU对 |

#### 运行时调试面板

| 功能 | 说明 |
|------|------|
| **实时DAG** | 显示当前轮激活的DPU子集和拓扑排序结果 |
| **执行追踪** | 每次Attack命中时，高亮正在执行的DPU，灰显被跳过的 |
| **DC监视** | 实时显示DC中所有Fact的内容 |
| **快照导出** | 将当前轮的DPU子集导出为Pipeline Asset，可在编辑器中离线调试 |

## 三、Fact模型的实现要点

### 3.1 DC存储

DC使用`TMap<FName, FInstancedStruct>`存储Fact——Fact Key作为索引，FInstancedStruct持有任意UStruct类型的Fact实例。

FInstancedStruct同时支持C++定义的USTRUCT和蓝图编辑器中创建的UUserDefinedStruct。DPU开发者可以在蓝图层定义新的Fact类型。

DC提供两条读写路径：C++模板路径（编译时类型安全）和通用FInstancedStruct路径（蓝图/反射）。两条路径对Condition评估层完全透明。

### 3.2 Condition条件树

Condition使用**条件树**实现（复用AIToken Condition系统的Predicate模式）：

| 节点类型 | 职责 | UClass标记 |
|---------|------|-----------|
| 基类 | EvaluateCondition（处理bReverse取反）+ GetConsumedFacts（递归收集） | Abstract, DefaultToInstanced, EditInlineNew, Blueprintable |
| AND | 所有子节点为true才true | 子类 |
| OR | 任一子节点为true即true | 子类 |
| FactQuery | 引用Fact Key + 领域方法名，通过FactMethodRegistry调用 | 子类 |
| Compare | 引用Fact Key + 字段名 + 比较运算符 + 数值，走UE5属性反射 | 子类 |

所有节点都有**bReverse取反开关**——基类统一处理，子类不需要关心。

**产销关系自动收集**：每个节点实现GetConsumedFacts()，AND/OR递归子节点合并去重，FactQuery/Compare返回自己的FactName。系统调用根节点即可获得完整consumes列表。

### 3.3 FactMethodRegistry

全局方法注册表（UEngineSubsystem），映射(UScriptStruct*, FName MethodName) → 查询函数。

**双注册路径**：

| 路径 | 适用场景 | 特点 |
|------|---------|------|
| **C++路径** | C++定义的USTRUCT | 模板+lambda，编译时类型安全 |
| **蓝图路径** | 蓝图定义的UUserDefinedStruct | UFunction引用，运行时反射 |

两条路径注册的方法在Registry中统一存储。CallMethod对调用者完全透明——不区分注册方式。

Registry还提供`GetMethodsForStruct(UScriptStruct*)`——返回该类型已注册的所有方法名列表，供属性面板FactQuery节点的联动下拉使用。

### 3.4 FactQuery节点的联动下拉

用户选Fact Key后，MethodName下拉需要联动更新。实现链路：

```
用户选 FactName
→ 从Fact Key→类型映射中查到对应的UScriptStruct*
→ 从FactMethodRegistry中查到该Struct的方法名列表
→ MethodName下拉显示这些方法
```

**Fact Key→类型映射的来源**：模型规范指出"在哪声明是实现选择"。实现时需要选择一种路径（DPU produces中附带、FactMethodRegistry注册时顺带建立、或独立全局注册表），保证属性面板和运行时都能查到。

### 3.5 Compare节点的反射访问

Compare节点需要按字段名访问Fact内部数据——统一走UE5属性反射（FProperty*），对C++和蓝图结构体都适用。通过`UScriptStruct::FindPropertyByName()`找字段，通过`FProperty::GetValue_InContainer()`取值。

## 四、分步实施计划

### Phase 1：运行时系统

**目标**：能跑通只狼V3.5的15个DPU管线。

**交付物**：

| 组件 | 类名 | 职责 |
|------|------|------|
| DC | `UDamageContext` | 共享上下文，TMap<FName, FInstancedStruct>存储Fact，Fact缺失返回false |
| Fact结构体 | 各USTRUCT | 各DPU的结构化产出（FMixupResult, FGuardResult, FDeathResult等） |
| Fact方法注册表 | `UFactMethodRegistry` | 全局方法注册表，支持C++和蓝图双注册路径 |
| DPU定义 | `UDPUDefinition` (DataAsset) | 存储Produces（Fact Key列表）、Condition条件树、LogicClass引用 |
| DPU逻辑基类 | `UDPULogicBase` | 虚函数Execute(DC)，子类实现具体机制，构造Fact写入DC |
| Condition条件树 | `UConditionNode` + 子类 | AND/OR/FactQuery/Compare，属性面板内联编辑，递归GetConsumedFacts() |
| 管线管理器 | `UPipelineManager` | 接收DPU子集→稳定拓扑排序→依次执行 |
| Pipeline Asset | `UPipelineAsset` (DataAsset) | 存储一组DPUDefinition的引用 |

**验证方式**：

- 测试关卡中手动触发Attack
- 日志打印拓扑排序结果和DC中的Fact内容
- 对照只狼V3.5的验证场景逐个确认：
  1. 普通斩击命中（无特殊机制）
  2. 弹刀（JustGuard）
  3. 格挡+架势满→破防
  4. 雷电+死亡→雷电死亡表现
  5. 空中接雷→不受伤（Fact缺失容错验证）

**扩展性预留**（为Phase 2准备，但Phase 1不实现）：

- PipelineManager的拓扑排序逻辑独立为可复用函数（编辑器Build时调用同一函数）
- DPUDefinition的Produces和Condition数据结构支持序列化/反序列化（编辑器需要读写）
- 日志输出格式规范化（编辑器调试面板可解析同一格式）

### Phase 2：编辑器模块

**前提**：Phase 1完全稳定。

**目标**：可视化DAG编辑器 + Build预览 + Asset管理。

**交付物**：

| 组件 | 说明 |
|------|------|
| Pipeline Asset编辑器 | 自定义Asset Editor，类似EQS的列表形式 |
| DAG自动布局 | 从Produces/Consumes自动推导依赖关系并渲染 |
| Build功能 | 调用Phase 1的拓扑排序函数，刷新视图，检测环 |
| Preview功能 | 用测试数据运行管线，高亮执行状态 |
| FactQuery联动下拉 | 选Fact Key后从Registry查方法列表 |
| 运行时调试面板 | 实时DAG + DC Fact监视 + 快照导出 |

## 五、关键设计约束

### 5.1 编辑器和运行时共用同一套引擎

编辑器的Build和Preview功能必须调用运行时系统的代码——不能有两套排序/执行逻辑。否则编辑器中看到的结果可能和实际运行不一致。

### 5.2 Pipeline Asset是唯一的数据交换格式

编辑器保存的Asset和运行时加载的Asset是同一种类型。运行时动态生成的管线也可以导出为同一种Asset。这保证了"编辑器中的管线"和"游戏中的管线"是同一个东西。

### 5.3 编辑器不存在于运行时

编辑器模块（Editor Module）在打包后的游戏中不存在。运行时系统（Runtime Module）独立于编辑器。编辑器依赖运行时，反过来不成立。

### 5.4 两种管线模式

| 模式 | DPU子集何时确定 | 拓扑排序时机 | 典型游戏 |
|------|---------------|------------|---------|
| **固定管线** | 设计时（编辑器Build保存为Asset） | 设计时一次 | 只狼、鬼泣 |
| **动态管线** | 运行时（每轮根据Boon确定） | 每轮开始时 | Hades、Slay the Spire |

两种模式的运行时执行逻辑完全相同——区别只在于Pipeline Asset的来源。

### 5.5 FactMethodRegistry双注册路径

C++路径（模板+lambda）和蓝图路径（UFunction引用）在Registry中统一存储。CallMethod对调用者透明。所有环节对C++和蓝图定义的Fact结构体都兼容。

### 5.6 Fact Key→类型映射

模型规范指出"映射在哪声明是实现选择"。实现时需要选择一种路径并保证：属性面板配置时和运行时都能查到此映射。具体路径在实现时根据实际情况决定。

## 六、与模型规范v4.4的映射

| 模型规范概念 | 实现层对应 |
|------------|----------|
| DPU | UDPUDefinition (DataAsset) + UDPULogicBase (逻辑子类) |
| Fact类型 | UStruct结构体定义（FMixupResult等，支持C++和蓝图） |
| Fact Key | FName，存储在UDPUDefinition::Produces中 |
| Fact实例 | FInstancedStruct，运行时存储在DC的TMap中 |
| 领域方法 | UFactMethodRegistry中注册的查询函数（C++lambda或蓝图UFunction） |
| DC | UDamageContext — TMap<FName, FInstancedStruct> |
| Condition条件树 | UConditionNode继承体系（And/Or/FactQuery/Compare），EditInlineNew+DefaultToInstanced |
| Produces | UDPUDefinition::Produces — TArray\<FName\> |
| Consumes | UConditionNode::GetConsumedFacts() 递归提取 |
| 拓扑排序 | UPipelineManager — 稳定Kahn算法（原始数组下标作为tiebreaker） |
| 防环检测 | UPipelineManager — 拓扑排序输出数量 < 输入数量则有环 |
| R3缺失容错 | UDamageContext::EvaluateFactMethod — Fact不存在返回false |
| bReverse | UConditionNode基类统一处理 |
| Pipeline Asset | UPipelineAsset — TArray\<UDPUDefinition*\> |
| 静态规则 | DPUDefinition中的配置（设计时定义，运行时不变） |
| 动态组装 | UPipelineManager接受DPU子集作为输入 |
