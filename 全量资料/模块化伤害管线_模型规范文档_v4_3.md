# 第一部分：WHY——动作游戏伤害管线的本质原则

## 一、四条设计思想

动作游戏的伤害管线有其固有的设计特征——不是工具的选择，不是实现的偏好，而是这类游戏的体验本质所要求的。以下四条思想是伤害管线问题域的WHY。

| 编号 | 思想 | 关切层面 | 核心表述 |
|------|------|---------|---------|
| **①** | **机制优先** | 单机制·设计 | 体验由机制驱动，不是由数值驱动。先有"格挡""弹刀""雷反"这些体验机制，再有数值填充。 |
| **②** | **数值兜底** | 单机制·实现 | 数值不驱动体验，但保证体验的数学基础正确。HP公式、架势公式、毒蓄积公式——它们是机制的数学地基。 |
| **③** | **反馈清晰可感** | 单机制·表现 | 每个机制的结果必须通过多模态表现（动画、音效、特效、位移、镜头）被玩家感知到。玩家感知不到的机制等于不存在。 |
| **④** | **涌现自治** | 多机制·协作 | 多个机制同时生效时，组合效果不由设计者穷举，而由各机制在共享上下文上自然产生。 |

①②③保证每个积木块的质量——从设计到实现到表现的纵向全链路。④保证积木块组合后的系统质量——多机制共存和组合的横向协作标准。

### 1.1 四条思想之间的关系

```
① 机制优先（设计层）
    ↓ 机制确定后
② 数值兜底（实现层）
    ↓ 实现正确后
③ 反馈清晰可感（表现层）
    ↓ 单机制完整后
④ 涌现自治（多机制组合层）
    → 多个完整的机制在一起时，组合效果自治产生
```

①②③是纵向的（单机制从设计到落地），④是横向的（多机制组合）。先有好的积木块（①②③），才有好的组合（④）。

# 第二部分：WHAT——问题域模型

## 二、问题域界定

### 2.1 两阶段架构

一次伤害事件的处理分为两个独立阶段，分别对应设计思想①②④和③。

```
Attack命中
→ 【机制阶段】各机制的判定和结算（对应①②④）
→ 【表现选取阶段】多机制结果的表现择优（对应③）
→ 表现执行
```

### 2.2 机制阶段

解决"这次攻击命中后，哪些机制生效、产生什么结果"。

核心关切：机制优先（①）——先确定哪些机制参与。数值兜底（②）——各机制内部的数值结算正确。涌现自治（④）——多机制同时生效时，组合效果自然产生。

### 2.3 表现选取阶段

解决"多个机制同时生效时，有限的表现手段如何分配给玩家感知"。

核心关切：反馈清晰可感（③）——玩家必须感知到最重要的机制结果。

### 2.4 两阶段的关系

两阶段通过共享上下文（DC）连接——机制阶段写入DC，表现选取阶段读取DC。表现选取不写DC、不影响机制结果。

### 2.5 "机制涌现"与"表现组合"的本质区别

两者结构相似——都是"条件A+条件B→产出C"。**但设计目标完全不同：**

| 维度 | 机制涌现（④的产物） | 表现组合（③的产物） |
|------|-------------------|-------------------|
| **目标** | 产生新的游戏规则/结算逻辑 | 让玩家更清晰地感知已有结果 |
| **如果不做** | 游戏逻辑错误 | 游戏逻辑正确，但反馈不够清晰 |
| **新增的是什么** | 新的机制行为 | 新的表现资源 |

```
机制涌现：Lightning + InAir → 空中接雷不受伤（改变了游戏规则）
表现组合：Death + Lightning → 雷电死亡动画（反馈更清晰，死亡判定不变）
```

结构相同不等于语义相同。混淆两者会导致修改动画方案时影响机制逻辑。

## 三、涌现

涌现是多个机制组合产生的效果（设计思想④的具体体现）。

### 3.1 涌现的本质认知

**机制处理体内部的每一条if-else分支，都可能是另一个独立机制的产出与当前机制的涌现。** 识别"什么是独立机制、什么是机制间的涌现"是设计者的核心工作，不可由工具或AI代劳。

```
示例：DPU_Lightning 内部的 if(IsInAir)

原始理解：这是Lightning的内部规则
深层理解："角色在空中"是一个独立机制（动作系统的产出），
          "空中 + 雷击"是两个机制的涌现
          → IsInAir应该是共享上下文中的独立事实
          → "空中雷击的特殊处理"应该是一个独立的机制处理体
```

### 3.2 涌现的多维度来源

涌现不只来自伤害管线内部——它可能跨越多个系统维度：

| 维度 | 示例 |
|------|------|
| **伤害管线机制** | Death、Collapse、Guard、Lightning、Poison |
| **角色状态** | IsInAir、IsOnGround、IsSuperArmored |
| **角色类型** | IsPlayer、IsNPC |
| **玩家输入** | AttackInput、DashInput |

## 四、表现选取

### 4.1 问题陈述

一次伤害事件中，多个机制同时生效，各自产出表现信息。但表现手段有限——不能同时播两个全身受击动画，玩家的认知带宽也有限。

### 4.2 三个本质概念

| 概念 | 定义 |
|------|------|
| **Channel** | 表现通道。同一Channel内互斥（只能选一个），不同Channel间可叠加（同时呈现）。 |
| **Priority** | Channel内的选取优先级。数值越高越优先——"玩家越需要感知到这个机制"。支持basePriority继承+逐Channel覆盖。 |
| **CombinedPresentation** | 表现组合。一个表现资源同时刻画多个机制的结果。不是机制涌现，是反馈优化。 |

### 4.3 Channel

Channel是表现通道——一个可独立控制的表现维度。

```
Channel示例：
  FullBodyAnim  —— 全身受击动画（互斥）
  AdditiveAnim  —— 加法层动画（可叠加）
  SFX           —— 音效（可叠加）
  VFX           —— 特效（可叠加）
  Motion        —— 位移/击退（可叠加）
  CameraShake   —— 镜头震动（可叠加）
```

Channel的定义是项目层面的设计决策——不同游戏可能有不同的Channel划分。

### 4.4 Priority

每个机制处理体可声明basePriority，其所有Channel默认继承。允许逐Channel覆盖。

```
DPU_Poison {
  basePriority: 50
  表现: {
    FullBodyAnim: { priority: 继承50, asset: Poison_Writhe }
    SFX:          { priority: 覆盖85, asset: poison_gurgle }  ← 毒音效特别重要
  }
}
```

### 4.5 CombinedPresentation

一个表现资源同时刻画多个机制的结果。当其源机制全部生效时，替代各自的单独表现。

```
CombinedPresentation:
  sources: [Death, Lightning]
  FullBodyAnim: Death_Electrocute（雷电死亡动画）
```

### 4.6 选取算法

```
对于每个Channel：
  1. 收集所有机制处理体在该Channel上的表现信息
  2. 检查CombinedPresentation且其源机制都已生效 → 有则选它
  3. 否则按Priority选最高的
  
最终输出 = 所有Channel各自选取结果的叠加
```

# 第三部分：HOW——L3声明式编排工具

## 五、工具层概述

本部分定义用于实现模块化伤害管线的L3声明式编排工具。工具的设计目标是忠实表达第一部分（WHY）的设计思想和第二部分（WHAT）的问题域模型。

工具层的核心概念：

| 概念 | 对应的问题域需求 |
|------|----------------|
| **DPU（机制处理体）** | 承载独立机制的判定和结算逻辑（①②） |
| **DC（共享上下文）** | 机制间的数据通信，涌现的载体（④） |
| **Condition（执行条件门控）** | 机制间涌现的条件表达（④） |
| **Produces/Consumes** | 依赖关系声明，自动确定执行顺序 |
| **Channel/Priority** | 表现选取的分组和择优（③） |

## 六、三个核心概念

### 6.1 DPU（Damage Processing Unit）

机制处理体——自洽的积木块。声明produces字段，读取DC、执行机制逻辑、将结果写回DC。

DPU的核心行为：从DC中读取数据 → 执行机制逻辑 → 将结果写回DC。

#### Produces声明

每个DPU必须显式声明自己写入DC的字段列表（produces）。

```
DPU_Guard {
  produces: [IsGuard, IsJustGuard, GuardSuccess, GuardLevel]
  Condition: Guard && !IsLightningInAir
  逻辑：格挡判定+表现
}
```

produces声明的约束：
- **完整性**：DPU写入DC的所有字段都必须在produces中声明
- **独占性**：同一个DC字段只能有一个生产者DPU
- **与内部逻辑一致**：produces声明的是"会写什么"，不是"一定写什么"

#### DPU产出中的表现信息

DPU产出可能包含表现信息，需标注Channel和Priority。DPU不关心这些信息最终会不会被播放。

```
DPU_Guard 写入DC：{
  IsGuard: true,                          ← 判定结果
  postureResult: { newPosture: 75 },      ← 数值计算
  表现: {                                  ← 表现信息
    FullBodyAnim: { priority: 60, asset: W_GuardSmall },
    SFX:          { priority: 60, asset: metal_clang_01 }
  }
}
```

#### DPU产出的双向作用目标

某些DPU的产出需要同时描述对多个作用目标的影响（如弹刀同时影响受击方和攻击方）。

#### DPU的约束规则

| 规则 | 说明 |
|------|------|
| **自洽性** | 内部逻辑完整独立。不知道自己被什么Condition门控，不知道其他DPU的存在。 |
| **只通过DC通信** | 不直接调用其他DPU。上下游影响通过DC间接实现。 |
| **不感知管线结构** | 不知道自己在管线中的位置、被什么Condition门控。 |
| **声明produces** | 必须显式声明写入DC的字段列表。 |

### 6.2 DC（DamageContext）

共享上下文——本次事件的完整数据环境。

#### DC中的两类内容

| 类别 | 是什么 | 来源 |
|------|--------|------|
| **上下文信息** | "这次事件是什么"的原始数据 | 参数表、角色属性、碰撞结果 |
| **DPU产出** | "什么事被处理了，结果是什么" | DPU执行后写入 |

#### 特征

| 特征 | 说明 |
|------|------|
| 生命周期 | 瞬时——事件开始时创建，结束时销毁 |
| 可见性 | 所有DPU和Condition均可读取DC中的任何内容 |
| 写入权 | 只有DPU可以写入。Condition不写入。 |
| 字段缺失 | 缺失字段视为false/null |

#### 上下文信息的分类

| 来源 | drawio颜色 |
|------|-----------|
| **攻击方** | 浅红 #f8cecc / 红框 #b85450 |
| **受击方** | 浅绿 #d5e8d4 / 绿框 #82b366 |
| **事件上下文** | 浅灰 #f5f5f5 / 灰框 #666666 |

### 6.3 Condition

执行条件门控——读取DC全部信息，输出true/false，决定DPU是否执行。隐含consumes声明。

#### 特征

| 特征 | 说明 |
|------|------|
| **可读范围** | DC中的一切 |
| **输出** | true/false |
| **写入权** | 无 |
| **与DPU的关系** | 装配关系。可随时调整而不影响DPU内部实现。 |
| **缺失字段** | 引用的DC字段不存在时视为false（天然容错） |

#### 表达力规范

Condition的条件表达式支持以下运算符：

| 类别 | 运算符 | 示例 |
|------|--------|------|
| **布尔逻辑** | `&&` `\|\|` `!` | `Guard && !IsLightningInAir` |
| **相等判断** | `==` `!=` | `MixupResult == Guard` |
| **比较运算** | `>` `<` `>=` `<=` | `PoisonBuildup >= PoisonThreshold` |

布尔逻辑和相等判断覆盖大多数涌现条件（"是不是格挡""有没有雷电"）。比较运算覆盖数值阈值条件（"毒蓄积是否达到触发阈值""架势值是否超过上限"）。

```
示例：

布尔门控：Guard && !IsLightningInAir
  → 格挡类攻击，且不是空中接雷

相等判断：MixupResult == JustGuard
  → 博弈结果为弹刀

比较运算：PoisonBuildup >= PoisonThreshold
  → 毒蓄积达到触发阈值

混合使用：!(Guard && GuardSuccess) && DamageLevel > 2
  → 非格挡成功，且伤害等级大于2
```

#### 无Condition的DPU

DPU可以没有Condition——此时永远执行。

#### 新增涌现时的操作

1. 添加新DPU（如果涌现产生了新行为）
2. 配置Condition——可能需要调整已有DPU的Condition以实现互斥

已有DPU的内部实现不修改。改变的只是Condition装配逻辑。

## 七、依赖关系与自动排序

### 7.1 产销声明

| 声明 | 来源 | 含义 |
|------|------|------|
| **Produces** | DPU显式声明 | "我写入这些DC字段" |
| **Consumes** | 从Condition自动提取 | "我依赖这些DC字段" |

### 7.2 自动拓扑排序

系统根据产销关系自动生成执行顺序：

1. 收集所有DPU的produces和Condition的consumes
2. 匹配：字段X的生产者 → 字段X的消费者 → 建立有向边
3. 拓扑排序：生产者在消费者之前执行
4. 无依赖关系的DPU之间顺序不确定

设计者不需要手动排列DPU顺序。

### 7.3 DC初始字段

DC的上下文信息在管线开始前就已填入——没有DPU生产者。引用这些字段的Condition不产生依赖边。

### 7.4 防环规则

两个DPU互相依赖对方的produces → 它们不是独立机制 → 应该合并。

## 八、静态规则与动态组装

### 8.1 两阶段分离原则

| 阶段 | 谁做 | 做什么 | 频率 |
|------|------|--------|------|
| **开发阶段：配规则** | 设计者 | 创建DPU、声明produces、配置Condition | 一次性 |
| **运行阶段：配组合** | 系统/玩家 | 从全量DPU池中选择激活哪些DPU子集 | 每轮不同 |

**规则不随组合变化。** Condition是静态的游戏规则，DPU的激活子集是动态的玩家选择。两者完全解耦。

### 8.2 开发阶段

新机制首次加入时：

1. 创建新DPU，声明produces
2. 配置新DPU的Condition
3. 调整受影响的已有DPU的Condition

做完后规则永不再改。

### 8.3 运行阶段

1. 确定本轮激活的DPU子集
2. 对激活DPU做拓扑排序
3. 依次执行——DC字段缺失视为false

```
没有空中感电Boon → DPU_LightningInAir不在管线中
→ IsLightningInAir永远不写入DC
→ Death的Condition中 !IsLightningInAir → 缺失视为false → !false = true → Death正常执行
→ 和没有这个机制的时候一模一样

有空中感电Boon → DPU_LightningInAir加入管线
→ 空中被雷击时写入IsLightningInAir = true → Death跳过
→ 涌现自动生效

两轮中Death的Condition完全相同，没有任何修改。
```

## 九、本质约束规则

以下规则是工具层的本质约束——删掉任何一条，工具的表达能力会受损。

| 编号 | 规则 |
|------|------|
| **R1** | **DC是唯一通信介质**：DPU之间不直接调用，所有数据交换通过DC。 |
| **R2** | **DPU自洽**：DPU内部逻辑完整且独立。不知道自己被什么Condition门控，不知道其他DPU的存在。 |
| **R3** | **Condition可读DC全部内容，缺失字段视为false**：输出true/false。不写入任何数据。DC字段不存在时按"不满足"处理。 |
| **R4** | **Condition与DPU是装配关系**：新增涌现=调整Condition+添加新DPU，不调整已有DPU内部实现。 |
| **R5** | **执行顺序由依赖关系决定，支持动态管线**：produces/consumes自动拓扑排序。管线可运行时动态组装不同DPU子集。 |
| **R6** | **禁止循环依赖**：互相依赖=不是独立机制=合并为一个DPU。 |

# 第四部分：方法论

## 十、设计方法论

### 10.1 Essential/Accidental Complexity

**Essential Complexity（本质复杂度）**：问题域固有的、无法消除的复杂度。

**Accidental Complexity（偶然复杂度）**：因表达方式或技术选择引入的额外复杂度。可消除、可替换。

**核心指引**：模型规范中只应包含本质复杂度。偶然复杂度属于实现层、工具层、教学层。

**信号识别**：当你感到设计"拧巴"时，大概率是在和偶然复杂度搏斗。

### 10.2 三步检验程序

每次引入新概念时：

**第一步：删除测试** — "删掉它，丢什么？"丢了=本质，没丢=偶然。

**第二步：来源追溯** — "长出来的还是借来的？"借来的要额外验证。

**第三步：换表达验证** — "换种方式表达，它还在吗？"还在=本质，不在=偶然。

### 10.3 涌现识别清单

每次在DPU内部写if-else时：

> "这个if的条件部分，是不是另一个独立机制的产出？"

是→应该提取为DC中的独立事实+独立DPU+Condition。
不是（只是本DPU内部的数值计算中间结果）→留在内部。

**判断标准：这个条件未来是否可能被其他机制独立引用。** 这个判断只有设计者能做。

### 10.4 偶然复杂度的产生规律

| 版本 | 引入的偶然复杂度 | 根因 |
|------|----------------|------|
| v1.7 | Fact Key自动DAG | 抽象过高，超出认知带宽 |
| v2.0 | 多输出口、连线分流、拦截链 | 用拓扑穷举涌现 |
| v2.5 | 词缀选取表、缺省逻辑、装饰器 | 用中间层包装本质概念 |
| v2.5→v4.2 | 文档标题"声明式机制编排框架" | 工具层身份替代了内容域身份 |
| v3.0–v3.1 | 手动线性序列 | 依赖关系是本质的，手动排序是偶然的 |

### 10.5 AI协作中的注意事项

**设计者的模糊表述会把AI带向偶然复杂度。** AI给出方案后用三步检验审视。

**分工原则**：
- 本质复杂度的识别和判断：设计者的工作，不可委托
- 偶然复杂度的实现（文档、代码、格式检查）：AI的工作，设计者做价值判断

### 10.6 内容域与工具层的区分

本文档涉及两个维度——内容域（模块化伤害管线）和工具层（L3声明式编排）。这两个维度描述同一件事的不同侧面，但不能混淆或替代。

| 维度 | 是什么 | 具体内容 |
|------|--------|---------|
| **内容域（WHAT）** | 模块化伤害管线 | 四条设计思想、机制间涌现、表现选取——要解决的游戏设计问题 |
| **工具层（HOW）** | L3声明式编排 | DPU+Condition+DC+produces/consumes+拓扑排序——解决问题的工具方法 |

**工具服务于内容域，不是反过来。** 文档标题应该是内容域（你造的是什么），不是工具层（你用什么造的）。

## 十三、术语表

| 术语 | 定义 |
|------|------|
| **模块化伤害管线** | 本项目的内容域。动作游戏中"攻击命中后发生什么"的完整处理流程。 |
| **L3声明式编排** | 本项目的工具层。基于DPU+Condition+DC的声明式积木块编排方法。 |
| **DPU** | 机制处理体。自洽的积木块——声明produces，读DC、执行逻辑、写DC。 |
| **DC** | 共享上下文。承载上下文信息+DPU产出。字段缺失时视为false。 |
| **Condition** | DPU的装配门控。读DC输出bool，不写DC。与DPU是装配关系。缺失字段视为false。 |
| **Produces** | DPU声明写入DC的字段列表。生产者端。 |
| **Consumes** | Condition引用的DC字段列表。消费者端。自动提取。 |
| **涌现** | 多个机制组合产生的效果。DPU+Condition在DC上运行的自然结果。 |
| **Channel** | 表现通道。通道内互斥，通道间叠加。 |
| **Priority** | Channel内选取优先级。支持basePriority继承+逐Channel覆盖。 |
| **CombinedPresentation** | 表现组合。同时刻画多个机制结果。不是机制涌现，是反馈优化。 |
| **静态规则** | 开发阶段配好的DPU+Condition+produces。不随运行时组合变化。 |
| **动态组装** | 运行阶段选择激活哪些DPU子集。Condition天然容错。 |
| **本质复杂度** | 问题域固有的复杂度。不可消除。 |
| **偶然复杂度** | 表达方式引入的额外复杂度。可消除。 |

*模型的核心叙事：动作游戏的伤害管线由四条设计思想驱动——机制优先、数值兜底、反馈清晰可感、涌现自治。管线分为机制阶段和表现选取阶段。机制阶段用L3声明式编排工具实现——DPU是自洽的积木块（声明produces），Condition是装配门控（隐含consumes，缺失字段视为false），DC是共享工作台。系统自动拓扑排序，支持动态管线。表现选取阶段按Channel分组、按Priority择优。开发时配规则，运行时配组合，两者解耦。*
