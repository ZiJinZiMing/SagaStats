# Level 3 声明式编排工具设计与落地指南（最终定稿）

## 一、业界通用分层模型：Level 1～4（内容生产能力成熟度分层）

### 核心定义

这套分层覆盖**从“静态配置”到“自动化生成”的全链路进化，是游戏/低代码领域判断编排工具工业化程度的核心标准，核心目标是让工具从实现域工具**升级为**创作者专属编排工具**。

### Level 1：数据驱动（Data-Driven）

* **核心特征**：结构固定、仅改参数，无任何逻辑组合能力
* **一句话定义**：只能填数值/配置，不能改变逻辑骨架
* **典型场景**：Excel/CSV配表、游戏DataTable、数值配置表
* **使用者**：策划/运营
* **本质**：静态参数驱动，无法自定义业务逻辑

### Level 2：策略/模板 + 实例重载（Strategy / Template Override）

* **核心特征**：预制行为模板库，支持“选模板+改参数”，无自定义流程能力
* **一句话定义**：在预设策略中选择组合，无法拼接新逻辑
* **典型场景**：UE蓝图Instance Override、技能系统策略库、行为树预制模板
* **使用者**：策划/战斗设计师
* **本质**：预制体复用，局限于固定逻辑框架内调整

### Level 3：声明式编排（Declarative Orchestration）

* **核心特征**：面向创作者（TD/任务/战斗/AI设计），**节点=意图原子，连线=体验流程**
* **一句话定义**：用节点和连线描述“要做什么”，而非“怎么实现”，支持自由组合自定义逻辑
* **典型场景**：规范用法UE蓝图、行为树、Flow Graph、任务流程图
* **使用者**：TD/策划/创作者
* **本质**：创作者主导的组合式编排，核心是“组合大于实现”

### Level 4：元工具 / 意图自动生成（Meta Tool / Intent Generation）

* **核心特征**：自然语言/需求描述→自动生成Level 3逻辑图，实现意图蒸馏
* **一句话定义**：人只提需求，工具自动完成节点编排与生成
* **典型场景**：AI辅助蓝图生成、Prompt-to-Logic、意图驱动自动编排
* **使用者**：策划/创作者（零代码/低代码）
* **本质**：自动化Level 3编排，是工业化编排的终极形态

## 二、Level 3 核心设计灵魂：黄金圈（Why→How→What）

Level 3 工具的本质是**创作者意图的可视化落地**，其设计必须严格遵循黄金圈逻辑，这是区别于“可视化代码”的核心分水岭。

### 1\. Why（最优先：创作者意图 \& 体验视角）

* **核心原则**：从任务、剧情、战斗、AI的**体验目标**出发，而非代码结构
* **节点定位**：一个节点对应**创作者一个完整、自洽的意图单元**，贴合人类自然逻辑
* **核心要求**：不考虑函数拆分、底层实现、技术细节，只聚焦“我要实现什么体验”
* **示例**：创作者意图“等待某件事完成，无论已发生或未来发生”→ 对应WaitForCondition节点

### 2\. How（编排视角：流程与连线规则）

* **核心原则**：连线仅表达**体验流程关系**，不暴露任何技术机制
* **允许的流程关系**：顺序、条件分支、并行执行、等待、中断、跳转
* **核心要求**：只描述“流程逻辑”，不涉及协程、回调、代理等技术细节
* **本质**：用可视化流程还原创作者的思考路径，实现“图表即设计”

### 3\. What（实现视角：代码 \& 底层封装）

* **核心原则**：所有技术细节**彻底封装、完全隐藏**，绝不暴露给创作者
* **封装内容**：状态管理、事件监听、代理机制、缓存处理、轮询、协程、对象池等
* **核心要求**：创作者无需感知实现细节，仅需关注意图与流程
* **本质**：实现层与编排层严格隔离，降低创作者使用门槛

## 三、Level 3 节点设计终极准则（可直接落地）

### 核心金句

**Level 3 节点，按「创作者意图心智」拆分，不按「代码单一职责」拆分。**

* 代码最小单元：**函数/功能点**（追求技术层面的单一职责）
* Level 3 最小单元：**意图/体验动作**（追求创作者心智层面的单一自洽）

### 1\. 节点命名与语义铁律（AI可直接检查）

#### 动作节点

* 命名格式：**动词 + 业务对象**
* 正确示例：等待事件、播放对话、进入阶段、触发眩晕、给予道具
* 错误示例：计算伤害并暴击、处理受击逻辑、播放特效和声音

#### 判断节点

* 命名格式：**是否 + 条件**
* 正确示例：是否任务完成、是否玩家进入区域、是否血量低于30%
* 错误示例：判断状态、检测条件、检查结果

#### 绝对禁用词汇

技术类：`Func`、`Handler`、`Manager`、`Logic`、`Handle`、`Calc`、`Compute`、`Callback`

过程类：`先…再…`、`然后…`、`如果…就…`（流程由连线表达，节点不承载过程）

### 2\. 节点职责铁律

* 一个节点 = **一个完整创作者意图**，仅做一件事
* 节点必须能用**一句人话清晰定义**，禁止复合语义
* 节点内部可封装复杂实现，但对外仅暴露**单一意图**，不泄露内部逻辑

### 3\. 流程与连线铁律

* 连线仅表达**体验流程**，不承载任何技术逻辑
* 重复≥2次的流程片段，必须抽为**子图/模板**，禁止平铺硬编码
* 禁止跨越多层的长连线、交叉连线，保证流程图可读性
* 流程结构必须贴合**问题域本身的逻辑**（如流程类天然DAG、状态类天然状态机）

### 4\. 分层隔离铁律

* 程序职责：仅提供**编排骨架**（顺序、并行、条件、等待、子图等基础能力），不设计业务节点
* 创作者职责：定义**业务语义与意图原子**，负责节点组合与流程编排
* 核心红线：Level 3 编排层绝对不泄露实现域概念，实现层不干预编排层逻辑

## 四、经典案例解析：Cyberpunk 2077 —— WaitForCondition 节点

### 程序员视角（What 层：实现域）

* 节点行为：判断条件是否已成立 + 等待未来条件成立 + 处理已发生/未发生的回溯逻辑
* 程序员认知：违反单一职责原则，应拆分为`IsConditionTrue`（判断）、`WaitEvent`（等待）、`HandleBacktrack`（回溯）多个节点
* 核心矛盾：用**技术功能**的标准评判节点设计

### 创作者视角（Why 层：意图域）

* 创作者意图：**我要等【一件事】达成，无论它已经发生，还是未来才发生**
* 节点价值：封装成**一个完整心智单元**，贴合创作者的核心诉求，无需关注底层实现
* 设计本质：以**创作者意图**为核心，将技术细节隐藏在节点内部，是Level 3 节点的完美范例

## 五、常见错误与本质成因（深度剖析）

### 错误 1：把 Level 3 做成「可视化代码」

* **表现**：节点按函数/模块/系统拆分，充斥技术词汇，流程模仿代码调用链
* **本质成因**：用**What（代码）思维**设计Why（意图）层面的工具，混淆了实现与编排的边界

### 错误 2：节点拆得太碎，违背创作者心智

* **表现**：机械照搬代码单职责规范，将一个完整意图拆分为多个细碎节点，增加创作复杂度
* **本质成因**：缺乏Level 3 拆分原则认知，不懂“意图原子”与“功能原子”的差异

### 错误 3：节点过大，沦为黑盒大函数

* **表现**：一个节点包揽多个独立业务逻辑，策划/创作者无法拆分、无法复用
* **本质成因**：未做**领域建模**，直接堆砌功能，未提炼出核心意图原子

### 错误 4：需求变更导致全图重写，无复用性

* **表现**：第一版节点完全为一次性场景设计，第二版需求变更后需重新实现所有节点
* **本质成因**：未从**可组合积木**角度设计节点，仅做了“一次性实现”，缺乏抽象与复用机制

### 错误 5：创作者使用门槛过高，最终回归程序编写

* **表现**：工具开放了高自由度，但节点设计不符合创作者心智，导致策划/TD无法使用，最终由程序完成所有编排
* **本质成因**：设计者为程序员视角，而非创作者视角，未遵循黄金圈设计逻辑

## 六、落地执行指南（团队可直接照搬）

### 1\. 角色分工（核心前提）

* **程序团队**：负责搭建编排骨架（基础流程能力、节点运行时、封装实现逻辑），不参与业务节点设计
* **创作者团队（TD/策划）**：负责定义业务语义、设计意图原子、制定节点组合规则
* **架构师**：负责把控分层边界、审核节点设计、推动领域建模落地

### 2\. 落地步骤（从0到1）

1. **领域建模**：由创作者主导，梳理业务域核心实体（如任务、技能、剧情）、行为（如等待、执行、判断）、关系（如先后、并行、依赖）
2. **定义意图原子**：基于领域模型，提炼单一创作者意图，形成节点语义库（命名+职责+示例）
3. **搭建编排骨架**：程序团队实现基础流程能力（顺序、并行、条件、等待、子图），隐藏所有技术细节
4. **制定约束规则**：将本指南中的命名、职责、分层规则落地为工具校验标准（含AI检查规则）
5. **创作者培训**：聚焦“意图拆分”“流程设计”“子图复用”，不讲解技术实现，培养创作者的问题域结构认知
6. **迭代优化**：收集创作者使用反馈，持续优化节点语义、流程骨架与约束规则

### 3\. AI 检查规则（直接复制使用）

#### 节点命名检查

* 动作节点必须符合「动词+业务对象」，判断节点必须符合「是否+条件」
* 禁止包含技术词汇、过程描述、复合语义
* 节点名称长度≤15字，且能一句人话讲清核心意图

#### 节点职责检查

* 一个节点仅对应一个创作者意图，无复合逻辑
* 节点内部无分支、循环、隐藏逻辑，仅封装单一意图的实现
* 节点不包含流程控制（如先后、条件、并行），流程由连线表达

#### 流程与连线检查

* 连线仅表达体验流程，无技术逻辑
* 相同逻辑片段重复≥2次，必须抽为子图/模板
* 流程图无交叉连线、无跨越多层的长连线，结构清晰可读

#### 分层隔离检查

* 节点名称、参数、逻辑中无任何实现域概念（如协程、代理、对象池）
* 程序未设计业务节点，业务节点均由创作者定义

## 七、核心总结（钉在团队墙的金句）

1. **Level 1 改参数，Level 2 选策略，Level 3 拼意图，Level 4 自动生成。**
2. **程序员拆功能，Level 3 设计者拆意图；代码实现“怎么做”，意图设计“为什么做”。**
3. **Level 3 的终极价值，是把创作者的思考结构可视化，让知识显性化，实现设计与知识的可传承、可维护。**
