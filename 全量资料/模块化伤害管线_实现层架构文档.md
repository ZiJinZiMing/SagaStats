# 模块化伤害管线：实现层架构文档 v4.6

> **文档性质**：实现层架构设计。定义运行时系统架构、编辑器UX方案、分步实施计划。基于模型规范v4.6 + Fact模型设计v2。
> **内容域**：模块化伤害管线——动作游戏中"攻击命中后发生什么"的完整处理流程。
> **工具层**：基于L3声明式编排实现——DPU+Fact+Condition条件树+DC+产销关系自动拓扑排序。
> **与模型规范的关系**：模型规范定义"是什么"（WHY+WHAT），本文档定义"怎么做"（HOW的实现细节）。本文档的所有内容都是偶然复杂度——可以被替换为其他实现方案而不影响模型的本质结构。

---

## 一、核心设计决策

### 1.1 运行时系统是本质，编辑器是偶然

| 组件 | Essential/Accidental | 依据 |
|------|---------------------|------|
| 运行时拓扑排序引擎 | **本质** | 删掉→回到手动排序→丢失表达能力 |
| Pipeline Asset（管线资产） | **本质** | 删掉→每次运行都要从头配DPU组合→无法复用 |
| 编辑器DAG视图 | 偶然（高优先级） | 删掉→设计者看不到管线全貌→体验差但功能不丢 |
| 运行时快照导出 | 偶然（高价值） | 删掉→无法在编辑器中调试特定Boon组合→体验差但功能不丢 |

### 1.2 图表是视图，不是数据

**数据层**：DPU定义存储在UDataAsset中（Produces声明、Condition条件树、逻辑类引用、表现信息）。这些数据在设计时和运行时完全相同。

**视图层**：DAG图表从数据层自动生成。设计者编辑的是DataAsset，图表自动刷新——不需要手动连线。连线由DPU间的依赖关系自动推导。

**区别于EQS/行为树**：EQS和行为树的Graph是"编辑器即数据"——Graph本身就是数据存储。本框架中Graph只是数据的可视化投影。原因：

- 手动连线与自动拓扑排序矛盾（R5要求执行顺序由依赖关系自动决定）
- 运行时动态组装时没有预设的Graph可加载（肉鸽模式每轮DPU子集不同）

### 1.3 一套引擎，两种模式，Pipeline Asset双向流通

编辑器模式和运行时模式共用同一套引擎。Pipeline Asset作为两种模式之间的数据交换格式，支持**双向流通**——既可以从编辑器流向运行时（设计者Build保存→游戏加载执行），也可以从运行时流回编辑器（运行时快照导出→编辑器离线调试）。

```
编辑器模式：                          运行时模式：
  手动选DPU类                           从池中选激活子集（Boon决定）
  ↓                                    ↓
  同一套运行时引擎                       同一套运行时引擎
  自动拓扑排序                          自动拓扑排序
  ↓                                    ↓
  渲染DAG视图                           按顺序执行
  可预览/调试                           调试面板实时显示
  保存为Pipeline Asset                  可导出为Pipeline Asset
       │                                    │
       │  ──── 方向1：编辑器→运行时 ────→     │
       │        游戏加载Asset执行             │
       │                                    │
       │  ←─── 方向2：运行时→编辑器 ────      │
       │        快照导出回编辑器调试           │
```

## 二、三层架构

### 2.1 数据+引擎层：Pipeline Asset（自洽执行）

Pipeline Asset既是管线的数据容器（持有DPU列表），也是管线的执行引擎（自带Build和Execute能力）。不需要独立的PipelineManager——Pipeline Asset自己就知道怎么排序和执行。

**Pipeline Asset持有的数据**：

每个DPU定义（UDataAsset）包含：

| 字段 | 内容 | 说明 |
|------|------|------|
| DPUName | FName | 身份标识——同时用于DC中Fact索引和ConditionNode子类的反查 |
| Produces | Fact类型（UScriptStruct*） | 声明该DPU产出的Fact结构体类型 |
| Condition | 条件树根节点（Instanced） | 属性面板内联编辑——复合节点（AND/OR）+叶子节点（ConditionNode子类） |
| LogicClass | DPU逻辑类引用 | 子类重写Execute——读DC、执行逻辑、返回Fact |
| BasePriority | float | 表现选取的基础优先级 |

**Pipeline Asset的两个核心能力**：

```
Build()：
  1. 从自身的DPUDefinitions提取依赖关系
     - 每个DPU的身份标识 = 被依赖的标识
     - 从每个DPU的Condition条件树递归GetConsumedDPUs()
       → ConditionNode子类通过绑定的Fact类型反查DPU
  2. 稳定拓扑排序（Kahn BFS变体）
  3. 防环检测
  4. 缓存排序结果 → 标记为已烘焙
  5. 编辑器中修改DPU配置或Condition时自动标记为未烘焙，提示需要重新Build

Execute(DC)：
  1. 未烘焙则自动Build
  2. 按缓存的排序顺序依次：
     a. 评估Condition条件树（Fact缺失视为false）
     b. Condition为true → 执行DPU逻辑 → DPU返回Fact → PipelineAsset以DPU身份标识写入DC
     c. Condition为false → 跳过
  3. 表现选取（按Channel/Priority择优）
  4. 返回执行日志
```

**Build结果序列化保存在Asset中**——固定管线（只狼）编辑器Build一次保存，运行时零开销直接执行。动态管线（Hades）运行时组装DPU后Build一次，整轮复用。

**Pipeline Asset有两种来源**：

| 来源 | 方式 | 适用场景 |
|------|------|---------|
| **编辑器手动配置** | 设计者选择DPU类 → Build → 保存（烘焙排序结果到Asset） | 只狼等固定管线游戏 |
| **运行时动态生成** | 系统根据激活Boon组装DPU → Build → 整轮复用 | Hades等肉鸽游戏 |

两种来源的Build/Execute逻辑完全相同。

#### 稳定拓扑排序（Stable Topological Sort）

**设计决策：采用稳定拓扑排序——在满足依赖关系的前提下，尽量保留DPU在数组中的原始顺序。**

```
实现方式（基于Kahn算法的BFS变体）：

标准Kahn算法：
  维护"入度为0"的候选队列 → 每次取任意一个输出

稳定变体：
  候选队列改为按DPU在原始数组中的下标排序的优先队列
  每次取下标最小的那个输出

效果：
  有依赖关系的DPU → 严格按依赖排（被依赖方在前）
  无依赖关系的DPU → 按数组原始顺序排（谁在前就谁先）
  同一个DPU池 + 同一个数组顺序 → 每次Build结果完全一致
```

| 问题 | 标准拓扑排序 | 稳定拓扑排序 |
|------|------------|------------|
| 未声明的隐含依赖 | 可能排出错误顺序 | 保留数组顺序作为兜底意图，降低影响 |
| Build确定性 | 每次可能不同 | 同输入同输出，完全确定 |
| 使用门槛 | 必须完美声明所有依赖 | 逐步补全即可，数组顺序兜底 |

**同一套Build/Execute代码**在编辑器Preview和运行时执行中使用。

### 2.2 视图层：编辑器模块

#### 编辑器功能

| 功能 | 说明 |
|------|------|
| **DPU类选择** | 从项目中所有DPULogic子类中选择，添加到管线 |
| **Condition条件树编辑** | 属性面板内联编辑——选ConditionNode子类（类名=DPU名）→ 选领域方法（UFUNCTION反射自动填充下拉） |
| **Produces编辑** | 选择DPU产出的Fact类型（UScriptStruct*，支持C++和蓝图结构体） |
| **Build** | 调用PipelineAsset::Build()，烘焙排序结果，保存到Asset |
| **Preview** | 用测试数据运行一遍管线，高亮执行/跳过的DPU，显示DC中的Fact内容 |
| **保存** | Build结果随Asset序列化保存 |

#### 运行时调试面板

| 功能 | 说明 |
|------|------|
| **实时DAG** | 显示当前轮激活的DPU子集和拓扑排序结果 |
| **执行追踪** | 每次Attack命中时，高亮正在执行的DPU，灰显被跳过的 |
| **DC监视** | 实时显示DC中所有DPU的Fact内容 |
| **快照导出** | 将当前轮的DPU子集导出为Pipeline Asset |

## 三、Fact模型与Condition的实现要点

### 3.1 DC存储

DC使用`TMap<FName, FInstancedStruct>`存储Fact——**DPU身份标识（FName）**作为索引，FInstancedStruct持有该DPU产出的Fact实例。

FInstancedStruct同时支持C++定义的USTRUCT和蓝图编辑器中创建的UUserDefinedStruct。

DC提供两条读写路径：C++模板路径（编译时类型安全）和通用FInstancedStruct路径（蓝图/反射）。两条路径对Condition评估层完全透明。

### 3.2 Condition条件树

条件树由两类节点组成：

| 类别 | 节点 | 职责 | UClass标记 |
|------|------|------|-----------|
| **基类** | UConditionNode | EvaluateCondition（处理bReverse取反）+ GetConsumedDPUs（递归收集依赖） | Abstract, DefaultToInstanced, EditInlineNew, Blueprintable |
| **复合节点** | AND / OR | 组合子节点的布尔逻辑 | 子类 |
| **叶子节点** | 按DPU分的ConditionNode子类 | 绑定一个Fact类型，承载该DPU的领域方法（UFUNCTION） | 子类 |

**领域方法直接定义在ConditionNode子类上**——标记为UFUNCTION，方法列表通过UE5反射自动发现。属性面板中选方法名下拉由反射自动填充，不需要全局注册表。

**蓝图扩展**：蓝图继承UConditionNode创建子类，蓝图函数天然是UFunction，反射路径和C++一致。

### 3.3 ConditionNode子类的联动下拉

用户选ConditionNode子类后，方法下拉自动联动：

```
用户选 ConditionNode类型 = ConditionNode_Guard
→ 节点绑定的Fact类型为FGuardResult
→ UE5反射发现ConditionNode_Guard上的UFUNCTION列表 ["IsParry", "IsGuardBreak"]
→ Method下拉显示这两个选项
```

不需要FactMethodRegistry——方法发现完全由UE5反射机制完成。

### 3.4 产销关系收集

每个ConditionNode子类通过绑定的Fact类型反查哪个DPU produces它：

```
ConditionNode_Guard.GetConsumedDPUs()
  → 绑定FGuardResult → 到DPU池查找谁produces FGuardResult → return ["Guard"]

AND/OR.GetConsumedDPUs()
  → 递归所有Children → 合并去重
```

### 3.5 新建一套DPU的工作流

```
1. 新建Fact结构体（纯数据字段）
2. 新建ConditionNode子类（绑定Fact类型，定义领域方法UFUNCTION）
3. 新建DPU定义（Produces=Fact类型，LogicClass，Condition条件树）

三者通过Fact类型建立联系。不需要全局注册表。
```

## 四、分步实施计划

### Phase 1：运行时系统

**目标**：能跑通只狼V3.5的15个DPU管线。

**交付物**：

| 组件 | 类名 | 职责 |
|------|------|------|
| DC | `UDamageContext` | 共享上下文，TMap<FName, FInstancedStruct>存储Fact，以DPU身份标识为索引，Fact缺失返回false |
| Fact结构体 | 各USTRUCT | 各DPU的纯数据产出（FMixupResult, FGuardResult, FDeathResult等） |
| DPU定义 | `UDPUDefinition` (DataAsset) | 存储DPU名、Produces（Fact类型）、Condition条件树、LogicClass引用 |
| DPU逻辑基类 | `UDPULogicBase` | 子类重写Execute——读DC、执行逻辑、返回Fact。不负责写入DC。 |
| Condition条件树 | `UConditionNode` + 子类 | 复合节点（AND/OR）+叶子节点（按DPU分的ConditionNode子类，承载领域方法UFUNCTION），递归GetConsumedDPUs() |
| Pipeline Asset | `UPipelineAsset` (DataAsset) | 持有DPU列表 + Build（稳定拓扑排序烘焙）+ Execute（按排序执行，DPU返回Fact后写入DC） |

**验证场景**：

| # | 场景 | 验证重点 |
|---|------|---------|
| 1 | 普通斩击命中 | 基础执行流程 |
| 2 | 弹刀（JustGuard） | ConditionNode子类引用领域方法 |
| 3 | 格挡+架势满→破防 | 多DPU依赖链（通过Fact类型反查自动建立） |
| 4 | 雷电+死亡→雷电死亡表现 | 多ConditionNode子类组合Condition |
| 5 | 空中接雷→不受伤 | Fact缺失容错（DPU未激活时） |

**扩展性预留**（Phase 1不实现）：

- PipelineAsset的Build/Execute逻辑编辑器和运行时共用
- DPUDefinition的Produces和Condition数据结构支持序列化/反序列化
- 日志输出格式规范化

### Phase 2：编辑器模块

**前提**：Phase 1完全稳定。

| 组件 | 说明 |
|------|------|
| Pipeline Asset编辑器 | 自定义Asset Editor，类似EQS的列表形式 |
| DAG自动布局 | 从DPU间依赖关系（ConditionNode子类→Fact类型→DPU反查）自动推导并渲染 |
| Build功能 | 编辑器中的Build按钮调用PipelineAsset::Build()，烘焙排序结果并序列化保存到Asset |
| Preview功能 | 用测试数据运行管线，高亮执行状态 |
| ConditionNode方法下拉 | 选ConditionNode子类后UFUNCTION反射自动填充方法列表 |
| 运行时调试面板 | 实时DAG + DC Fact监视 + 快照导出 |

## 五、关键设计约束

### 5.1 编辑器和运行时共用PipelineAsset的Build/Execute

编辑器的Build按钮直接调用PipelineAsset::Build()，Preview直接调用PipelineAsset::Execute()——不存在独立的Manager或两套逻辑。

### 5.2 Pipeline Asset是唯一的数据交换格式

编辑器保存的Asset和运行时加载的Asset是同一种类型。运行时动态生成的管线也可以导出为同一种Asset。

### 5.3 编辑器不存在于运行时

编辑器模块（Editor Module）在打包后的游戏中不存在。运行时系统（Runtime Module）独立于编辑器。编辑器依赖运行时，反过来不成立。

### 5.4 两种管线模式

| 模式 | DPU子集何时确定 | 拓扑排序时机 | 典型游戏 |
|------|---------------|------------|---------|
| **固定管线** | 设计时（编辑器Build保存为Asset） | 设计时一次 | 只狼、鬼泣 |
| **动态管线** | 运行时（每轮根据Boon确定） | 每轮开始时 | Hades、Slay the Spire |

### 5.5 领域方法由ConditionNode子类承载

领域方法直接定义在ConditionNode子类上（UFUNCTION），通过UE5反射自动发现。不需要全局注册表。C++和蓝图ConditionNode子类的反射路径一致。

## 六、与模型规范v4.6的映射

| 模型规范概念 | 实现层对应 |
|------------|----------|
| DPU | UDPUDefinition (DataAsset) + UDPULogicBase (逻辑子类) |
| DPU身份标识 | UDPUDefinition::DPUName (FName) |
| Fact类型 | UStruct（纯数据结构，支持C++和蓝图定义） |
| Fact实例 | FInstancedStruct，运行时以DPU名为索引存储在DC中 |
| ConditionNode子类 | 按DPU分的UConditionNode子类，绑定Fact类型，承载领域方法（UFUNCTION） |
| 领域方法 | ConditionNode子类上的UFUNCTION，反射自动发现 |
| DC | UDamageContext — TMap<FName, FInstancedStruct>，DPU名为索引 |
| Condition条件树 | 复合节点（AND/OR）+叶子节点（具体ConditionNode子类），EditInlineNew+DefaultToInstanced |
| Produces | UDPUDefinition::Produces — UScriptStruct*（Fact类型） |
| Consumes | ConditionNode子类通过绑定的Fact类型反查DPU → GetConsumedDPUs()递归收集 |
| 拓扑排序 | UPipelineAsset::Build() — 稳定Kahn算法，排序结果缓存在Asset中 |
| R3缺失容错 | DC中DPU名不存在 → ConditionNode评估返回false |
| bReverse | UConditionNode基类统一处理 |
| Pipeline Asset | UPipelineAsset — 持有DPU列表 + Build() + Execute()，自洽执行 |
| 静态规则 | DPUDefinition中的配置（设计时定义，运行时不变） |
| 动态组装 | UPipelineAsset运行时组装DPU子集后调用Build() |
