# 模块化伤害管线：实现层架构文档 v4.7

> **文档性质**：实现层架构设计。定义运行时系统架构、编辑器 UX 方案、分步实施计划。基于模型规范 v4.7 + DamageEffect 模型设计 v3。
> **内容域**：模块化伤害管线——动作游戏中"攻击命中后发生什么"的完整处理流程。
> **工具层**：基于 L3 领域 DSL 实现——DamageRule + DamageEffect + DamageCondition + DamagePredicate + DC + 产销关系自动拓扑排序。
> **与模型规范的关系**：模型规范定义"是什么"（WHY + WHAT），本文档定义"怎么做"（HOW 的实现细节）。本文档的所有内容都是偶然复杂度——可以被替换为其他实现方案而不影响模型的本质结构。
> **最后更新**：2026-04-17（v4.6 的 7 项架构决策已反哺到此版本）

## 一、核心设计决策

### 1.1 运行时系统是本质，编辑器是偶然

| 组件 | Essential/Accidental | 依据 |
|---|---|---|
| 运行时拓扑排序引擎 | **本质** | 删掉 → 回到手动排序 → 丢失表达能力 |
| DamagePipeline（管线资产） | **本质** | 删掉 → 每次运行都要从头配 DamageRule 组合 → 无法复用 |
| 编辑器 DAG 视图 | 偶然（高优先级） | 删掉 → 设计者看不到管线全貌 → 体验差但功能不丢 |
| 运行时快照导出（Mermaid） | 偶然（高价值） | 删掉 → 无法在编辑器中调试特定 Boon 组合 → 体验差但功能不丢 |

### 1.2 图表是视图，不是数据

**数据层**：DamageRule 定义存储在 UDataAsset 中（Condition 树 + OperationClass）。

**视图层**：DAG 图表从数据层自动生成，不需要手动连线。连线由 DamageRule 间的 EffectType 产销关系自动推导（Produces 声明 ↔ Condition 树的 ConsumedEffectType）。

### 1.3 一套引擎，两种使用模式

编辑器模式（TD 预 Build 保存）和运行时模式（动态组装执行）共用同一套引擎——`UDamagePipeline::Build()` 和 `UDamagePipeline::Execute()`。DamagePipeline 作为 DataAsset 是数据容器，也是执行引擎。

### 1.4 v4.6 的 7 项架构决策（已反哺到 v4.7）

这 7 项决策是 v4.6 实现层相对于模型规范早期描述的调整，v4.6 → v4.7 演进期已确认它们是正确选择并反哺回模型：

1. **双层条件树** — `UDamagePredicate`（容器 AND/OR/Single + bReverse）+ `UDamageCondition`（原子，子类即域方法）
2. **子类即域方法** — 每个域方法一个 `UDamageCondition` 子类，不使用 MethodName 下拉或全局注册表
3. **`UScriptStruct*` 作 DC 键** — DC 存储用类型即 key，不使用 FName(RuleName)
4. **OutEffect 直传** — `Execute(Context, OutEffect&)`，框架预创建 OutEffect，Operation 只填字段，框架负责写 DC
5. **bReverse 仅在 Predicate 上** — 取反逻辑统一在容器层，Condition 上不存在 bReverse
6. **EffectType 在子类 CDO 固定** — `UDamageCondition` 子类用 `EditConditionHides` 等机制使 EffectType 在实例上不可修改
7. **Build 时 EffectType 产销校验** — `UDamagePipeline::Build()` 前置校验 Operation 的 `GetProducesEffectType()` 非空 + 环检测

**历史归档**：13 条 v4.3 → v4.7 演进期决策详见 [`doc/damagePipeline/_archive/已解决决策_v4.3-v4.7.md`](../doc/damagePipeline/_archive/已解决决策_v4.3-v4.7.md)。

## 二、两层架构

### 2.1 数据 + 引擎层：DamagePipeline（自洽执行）

DamagePipeline 既是数据容器（持有 DamageRule 列表），也是执行引擎（自带 Build 和 Execute 能力）。

**DamagePipeline 持有的数据**：

每个 DamageRule 定义（`UDamageRule`，UDataAsset 子类）包含：

| 字段 | 类型 | 说明 |
|---|---|---|
| Condition | `UDamagePredicate*`（Instanced） | 条件树根节点；属性面板内联编辑；空字段视为"无条件触发" |
| OperationClass | `TSubclassOf<UDamageOperationBase>` | 计算逻辑的类引用；运行时框架 NewObject 执行 |
| BasePriority | float | 表现选取的基础优先级（表现选取阶段用，当前代码未实施） |

**Rule 的身份**：不再存冗余的 RuleName 字段——直接用 DataAsset 的 `GetFName()` / `GetName()`。`FRuleExecutionEntry::RuleName` 保留作为日志接口。

**Produces 的反查**：Rule 不存 ProducesType 字段，而是从 OperationClass 的 CDO 调用 `GetProducesEffectType()` 动态查询——保证 Operation 子类与 Rule 的产出一致性。

**DamagePipeline 的两个核心能力**：

```
Build()：
  1. 构建 ProducerMap：EffectType → DamageRule
       - 遍历 DamageRules
       - ProducesType = rule.OperationClass CDO.GetProducesEffectType()
       - ProducerMap[ProducesType] = rule
  2. 构建依赖图：
       - 遍历每条 Rule 的 Condition 树，递归收集 GetDependencyEffectTypes()
       - 在 ProducerMap 中查 producer → 建依赖边
  3. 稳定拓扑排序（Kahn BFS 变体，候选队列按原始索引排序）
  4. 环检测（Sorted.Num() < Rules.Num() → 失败）
  5. 缓存 SortedRules + bIsBaked = true（SortedRules 作为 UPROPERTY 持久化保存）

Execute(Context)：
  前置：bIsBaked == true
  按 SortedRules 顺序遍历：
    ① Predicate 树递归求值（EvaluatePredicate）
    ② 通过 → 框架预创建 FInstancedStruct OutEffect(ProducesType)
    ③ Operation 实例化 + Execute(Context, OutEffect) 填字段
    ④ 框架 Context->SetEffectByType(OutEffect)
    ⑤ 返回 FRuleExecutionEntry 日志
  不通过 → 日志 bConditionPassed=false，跳过
```

**烘焙模型**：Build 结果（SortedRules + bIsBaked）序列化保存在 DamagePipeline 资产中。

| 来源 | 方式 | 适用场景 |
|---|---|---|
| **编辑器预 Build** | TD 编辑 → Build → SortedRules 持久化到资产 | 静态声明的管线（只狼等） |
| **运行时 Build** | 代码动态组装 DamageRules → 调 Build → 整轮复用 | 运行时动态拼装的管线（Hades 肉鸽等） |

两种来源都产生相同的 SortedRules，运行时 Execute 行为完全一致。

#### 稳定拓扑排序（Kahn BFS 变体）

在满足 EffectType 依赖关系的前提下，保留 DamageRule 在数组中的原始顺序作兜底。

| 问题 | 标准拓扑排序 | 稳定拓扑排序 |
|---|---|---|
| 未声明的隐含依赖 | 可能排错 | 数组顺序兜底 |
| Build 确定性 | 每次可能不同 | 同输入同输出 |
| 使用门槛 | 必须完美声明 | 逐步补全即可 |

实现细节：候选队列用优先队列按"Rule 在原始 DamageRules 数组中的下标"排序，每次出队最小的下标。

### 2.2 视图层：编辑器模块

| 功能 | 说明 |
|---|---|
| **DamageRule 选择** | 从项目中所有 Rule DataAsset 中选择加入 DamagePipeline |
| **Condition 树编辑** | 属性面板内联 —— UDamagePredicate 容器（AND/OR/Single + bReverse）+ UDamageCondition 叶子（类选择器选子类） |
| **OperationClass 引用** | TSubclassOf 选择；Operation 子类的 CDO 决定 ProducesEffectType |
| **Build** | 调用 `UDamagePipeline::Build()`，烘焙 SortedRules 保存到资产 |
| **Graph 视图** | 只读 DAG 可视化，展示 Rule → Effect → Rule 的产销连线 + Condition ASCII 树 |
| **资产创建工厂** | 右键菜单的 "Damage Pipeline" 类别包含 5 种类型：Pipeline / Rule / Condition 蓝图 / Operation 蓝图 / Effect 结构体 |

编辑器实现细节见 [`doc/damagePipelineEditor/GraphEditor实现.md`](../doc/damagePipelineEditor/GraphEditor实现.md)（L4 工具层）。

## 三、DamageEffect 与 DamageCondition 的实现要点

### 3.1 DC 存储与访问分层（v4.7+）

`UDamageContext` 使用 `TMap<TObjectPtr<UScriptStruct>, FInstancedStruct>` 存储 DamageEffect—— **类型即 key**，不使用独立的 FactKey 概念。

- `UScriptStruct*` 作为 key（编译时类型安全）
- `FInstancedStruct` 承载任意类型的 struct 实例（支持 C++ 和蓝图结构体）

**访问分层**（把 R5 契约从文档约束升级为编译器强制）：

| 场景 | 访问路径 | 允许读哪些 Effect |
|---|---|---|
| Game 侧（Execute 前填输入 / Execute 后读产出） | `UDamagePipelineResults::WriteEffect<T>` / `ReadEffect<T>` / `HasEffect<T>` / `GetAllEffects` | 任意 |
| DSL 内部 — Condition_Effect 预取 | 框架调 `Context->GetEffectByType(EffectType)`（friend 授权） | 自己声明的那一个 |
| DSL 内部 — Condition_Context 求值 | **无 Effect 访问路径**——签名没 InEffect，GetEffect 是 protected | 零 |
| DSL 内部 — Operation Execute | `UDamageOperationBase::ReadEffect<T>(Context)` 基类模板（本轮不做产销校验） | 任意（R5 收紧留下轮） |
| DSL 内部 — Pipeline Build/Execute | friend 直接调 Context API | 任意（框架职责） |

`UDamageContext` 的 Effect 读写 API 全部 `protected`；friend 列表：`UDamagePipeline`、`UDamagePipelineResults`、`UDamageCondition_Effect`、`UDamageOperationBase`。

**Game 侧蓝图入口**：`UDamagePipelineResults::WriteEffect` / `ReadEffect`（CustomThunk 通配结构体，显示名 "Write Pipeline Input" / "Read Pipeline Result"）。

**删除**：旧有 `UDamageContextLibrary`（v4.6 的通用蓝图读写桥，是绕过 R5 的漏洞来源）已在 v4.7 彻底删除——先移除 `BlueprintCallable`，审计确认无代码引用后删除类本身。Game 侧统一走 `UDamagePipelineResults`。详见 [`doc/damagePipeline/架构设计.md §6.4`](../doc/damagePipeline/架构设计.md)。

### 3.2 Condition 三层 + Predicate 双层

**Condition 采用三层继承**（v4.7+）：

```
UDamageCondition            —— 纯抽象基类（所有 Condition 的公共契约）
├─ UDamageCondition_Effect  —— 基于 Effect 的判定（贡献 R5 产销依赖）
└─ UDamageCondition_Context —— 基于 Context 的判定（不贡献产销依赖）
```

| 类别 | 类 | 职责 | UCLASS 标记 |
|---|---|---|---|
| **容器（组合逻辑）** | `UDamagePredicate` | Single / AND / OR + bReverse；递归求值与依赖收集 | `Abstract, DefaultToInstanced, EditInlineNew, BlueprintType` |
| **原子基类** | `UDamageCondition` | 公共抽象；定义 `EvaluateCondition`（PURE_VIRTUAL）+ `GetEffectType`（默认 nullptr）+ `GetDisplayString` | `Abstract, DefaultToInstanced, EditInlineNew, Blueprintable` |
| **Effect 原子** | `UDamageCondition_Effect` + 子类 | 绑定 EffectType；`Evaluate(Context, InEffect)`；贡献产销依赖 | `Abstract, Blueprintable` |
| **Context 原子** | `UDamageCondition_Context` + 子类 | 无 EffectType；`Evaluate(Context)`；**编译期禁止读任何 Effect** | `Abstract, Blueprintable` |

**子类即域方法**：每个 `_Effect` / `_Context` 子类就是一个领域方法（如 `UDamageCondition_IsLightning` 继承 `_Effect`；`UDamageCondition_InTutorial` 继承 `_Context`）。`_Effect` 子类的 `EffectType` 在 CDO 里固定（`EditCondition + EditConditionHides` + UFunction 判断 CDO 上下文）。

**Predicate 的求值**（伪码）：

```
EvaluatePredicate(Context):
  result = match Op:
    Single:  Condition ? Condition.EvaluateCondition(Context) : true
    AND:     all(child.EvaluatePredicate(Context) for child in Children)
    OR:      any(child.EvaluatePredicate(Context) for child in Children)
  return bReverse ? !result : result
```

**Condition 的求值**（伪码，两种形态）：

```
// _Effect 子类
UDamageCondition_Effect::EvaluateCondition(Context):
  InEffect = Context.GetEffectByType(this.EffectType)  // 框架自动预取
  return this.Evaluate(Context, InEffect)              // BlueprintNativeEvent

// _Context 子类
UDamageCondition_Context::EvaluateCondition(Context):
  return this.Evaluate(Context)                         // 直接 dispatch，无预取
```

**蓝图扩展**：策划可在 Content Browser 右键 → **Damage Pipeline / Damage Condition Effect (Blueprint)** 或 **Damage Condition Context (Blueprint)** 创建子类，实现新的领域方法。与 C++ 子类完全对等。

### 3.3 产销关系收集

```
UDamagePredicate::GetDependencyEffectTypes():
  match Op:
    Single: Condition ? [Condition.EffectType] : []
    AND/OR: flatten(child.GetDependencyEffectTypes() for child in Children)  → 去重

UDamageRule::GetConsumedEffectTypes():
  Condition ? Condition.GetDependencyEffectTypes() : []

UDamageRule::GetProducesEffectType():
  OperationClass.CDO.GetProducesEffectType()  // 从 Operation 子类反查
```

**为什么从 Condition 树推导 Consumes 而不是手动声明**：单一事实源（single source of truth）——TD 只需配 Condition 树，依赖关系自动推出；避免 Consumes 和 Condition 树两处漂移的可能。

### 3.4 新建一套 Rule + Effect + Condition + Operation 的工作流

```
1. 新建 DamageEffect 结构体（USTRUCT，纯数据字段或空的信号类型）
     → 右键 Damage Pipeline / Damage Effect Struct（UDS 蓝图结构体）
     → 或 C++ 直接写 struct

2. 新建 DamageCondition 子类（绑定到 Effect 类型，实现 Evaluate）
     → 右键 Damage Pipeline / Damage Condition Blueprint
     → 或 C++ 继承 UDamageCondition，在构造函数里设置 EffectType

3. 新建 DamageOperation 子类（override GetProducesEffectType，实现 Execute）
     → 右键 Damage Pipeline / Damage Operation Blueprint
     → 或 C++ 继承 UDamageOperationBase

4. 新建 DamageRule 资产（选 OperationClass，配 Condition 树）
     → 右键 Damage Pipeline / Damage Rule

5. 加入 DamagePipeline 资产的 Rules 数组 → Build
```

四者通过 `UScriptStruct*`（EffectType）建立联系——Condition 和 Operation 都声明自己处理哪种 Effect，Pipeline Build 时以 EffectType 为中介建立 Rule 间依赖。

## 五、关键设计约束

### 5.1 编辑器和运行时共用 DamagePipeline 的 Build/Execute

不存在独立的 Manager 或两套逻辑。DamagePipeline 自洽。

### 5.2 DamagePipeline 是唯一数据交换格式

编辑器保存的和运行时加载/动态生成的都是 `UDamagePipeline` 资产。Build 的结果（SortedRules + bIsBaked）作为 UPROPERTY 持久化——无论是 TD 在编辑器最后一次 Build 的结果，还是运行时动态 Build 的结果，都是同一种存储。

### 5.3 编辑器不存在于运行时

`SagaStatsEditor` 模块作为 `Editor` type，仅在编辑器打包，运行时 DLL 不包含。Runtime 模块（`SagaStats`）不依赖 Editor 模块。

### 5.4 DC 的 TMap key 是 UScriptStruct\*，不是 FName

所有历史版本提到 "RuleName（FName）作为 DC 索引" 的描述均已废弃。v4.7 使用 `TMap<TObjectPtr<UScriptStruct>, FInstancedStruct>`——类型即 key，见 §3.1。

### 5.5 Operation 不写 DC

`Execute(Context, OutEffect&)` 中，Operation 只负责填 OutEffect 字段。框架负责 `Context->SetEffectByType(OutEffect)`。这保证 R2 自洽——Operation 不需要知道自己产出到哪里。

### 5.6 bReverse 仅在 Predicate 上

取反逻辑统一在容器层。`UDamageCondition` 子类不暴露 bReverse 字段——如果需要取反，用 `UDamagePredicate(Op=Single, bReverse=true, Condition=X)` 包装。

### 5.7 领域方法由 `UDamageCondition` 子类承载

每个子类 = 一个领域方法。子类可以有 UPROPERTY 实例参数（如 `MinLevel` / `RequiredSkill` 等）。没有全局注册表，没有 MethodName 字符串下拉。

## 六、与模型规范 v4.7 的映射

| 模型规范概念 | 实现层对应 |
|---|---|
| DamageRule（伤害规则） | `UDamageRule` (DataAsset) + `UDamageOperationBase` (逻辑子类) |
| DamageRule 身份标识 | DataAsset 的 `GetFName()` / `GetName()`（不存冗余字段） |
| DamageEffect（规则产出） | UStruct（纯数据结构，支持 C++ 和蓝图） |
| DamageEffect 实例 | `FInstancedStruct`，以 `UScriptStruct*` 为索引存储在 DC 中 |
| DamageCondition（原子判定） | `UDamageCondition`（抽象基类）+ `UDamageCondition_Effect` / `UDamageCondition_Context` 两个具体子类 |
| DamageCondition_Effect（Effect-based） | 绑定 EffectType，贡献 R5 产销依赖，`Evaluate(Context, InEffect)` |
| DamageCondition_Context（Context-based） | 无 EffectType，不贡献产销依赖，`Evaluate(Context)`，编译期禁止读 Effect |
| DamagePredicate（组合容器） | `UDamagePredicate`，Op = Single/AND/OR + bReverse |
| 领域方法 | `_Effect` / `_Context` 子类（子类即域方法），C++ 和蓝图路径一致 |
| DC | `UDamageContext` —— `TMap<TObjectPtr<UScriptStruct>, FInstancedStruct>`；Effect 读写 API 为 protected |
| Game 侧 DC API | `UDamagePipelineResults`（WriteEffect/ReadEffect/HasEffect/GetAllEffects，唯一外部入口） |
| Condition 树 | `UDamagePredicate` 嵌套（容器）+ `_Effect` / `_Context` 叶子 |
| Produces 声明 | `UDamageOperationBase::GetProducesEffectType()` → `UScriptStruct*` |
| Consumes 声明 | `UDamagePredicate::GetDependencyEffectTypes()` 递归收集 |
| 拓扑排序 | `UDamagePipeline::Build()` —— Kahn BFS 稳定变体 |
| R3 缺失容错 | DamageEffect 不存在 → Condition 子类的 Evaluate 接收 invalid `FInstancedStruct`，默认返回 false |
| DamagePipeline | `UDamagePipeline` —— 持有 Rules + Build + Execute，自洽执行 + SortedRules 持久化 |
| DamageOperation | `UDamageOperationBase` 子类 —— `Execute(Context, OutEffect&)`，框架预创建 OutEffect，Operation 只填字段，框架负责写 DC |
