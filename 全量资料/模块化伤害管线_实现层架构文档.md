# 模块化伤害管线：实现层架构文档 v4.7

> **文档性质**：实现层架构设计。定义运行时系统架构、编辑器UX方案、分步实施计划。基于模型规范v4.7 + DamageEffect模型设计v3。
> **内容域**：模块化伤害管线——动作游戏中"攻击命中后发生什么"的完整处理流程。
> **工具层**：基于L3声明式编排实现——DamageRule+DamageEffect+DamageCondition条件树+DC+产销关系自动拓扑排序。
> **与模型规范的关系**：模型规范定义"是什么"（WHY+WHAT），本文档定义"怎么做"（HOW的实现细节）。本文档的所有内容都是偶然复杂度——可以被替换为其他实现方案而不影响模型的本质结构。

## 一、核心设计决策

### 1.1 运行时系统是本质，编辑器是偶然

| 组件 | Essential/Accidental | 依据 |
|------|---------------------|------|
| 运行时拓扑排序引擎 | **本质** | 删掉→回到手动排序→丢失表达能力 |
| DamagePipeline（管线资产） | **本质** | 删掉→每次运行都要从头配DamageRule组合→无法复用 |
| 编辑器DAG视图 | 偶然（高优先级） | 删掉→设计者看不到管线全貌→体验差但功能不丢 |
| 运行时快照导出 | 偶然（高价值） | 删掉→无法在编辑器中调试特定Boon组合→体验差但功能不丢 |

### 1.2 图表是视图，不是数据

**数据层**：DamageRule定义存储在UDataAsset中（Produces声明、DamageCondition条件树、DamageOperation逻辑类引用、表现信息）。

**视图层**：DAG图表从数据层自动生成，不需要手动连线。连线由DamageRule间的依赖关系自动推导。

### 1.3 一套引擎，两种模式，DamagePipeline双向流通

编辑器模式和运行时模式共用同一套引擎。DamagePipeline作为数据交换格式，支持双向流通——编辑器→运行时（Build保存）和运行时→编辑器（快照导出）。

## 二、两层架构

### 2.1 数据+引擎层：DamagePipeline（自洽执行）

DamagePipeline既是数据容器（持有DamageRule列表），也是执行引擎（自带Build和Execute能力）。

**DamagePipeline持有的数据**：

每个DamageRule定义（UDataAsset）包含：

| 字段 | 内容 | 说明 |
|------|------|------|
| RuleName | FName | 身份标识——同时用于DC中DamageEffect索引和DamageConditionNode的反查 |
| Produces | DamageEffect类型（UScriptStruct*） | 声明该DamageRule产出的DamageEffect结构体类型 |
| DamageCondition | 条件树根节点（Instanced） | 属性面板内联编辑——复合节点（AND/OR）+叶子节点（DamageConditionNode子类） |
| OperationClass | DamageOperation逻辑类引用 | 子类重写Execute——读DC、执行逻辑、返回DamageEffect |
| BasePriority | float | 表现选取的基础优先级 |

**DamagePipeline的两个核心能力**：

```
Build()：
  1. 从自身的DamageRules提取依赖关系
     - 每个DamageRule的身份标识 = 被依赖的标识
     - 从每个DamageRule的DamageCondition条件树递归GetConsumedRules()
       → DamageConditionNode子类通过绑定的DamageEffect类型反查DamageRule
  2. 稳定拓扑排序（Kahn BFS变体）
  3. 防环检测
  4. 缓存排序结果 → 标记为已烘焙
  5. 编辑器中修改配置时自动标记为未烘焙

Execute(DC)：
  1. 未烘焙则自动Build
  2. 按缓存的排序顺序依次：
     a. 评估DamageCondition条件树（DamageEffect缺失视为false）
     b. true → 执行DamageOperation → 返回DamageEffect → DamagePipeline以RuleName写入DC
     c. false → 跳过
  3. 表现选取（按Channel/Priority择优）
  4. 返回执行日志
```

**Build结果序列化保存在Asset中。** 固定管线（只狼）编辑器Build一次保存，运行时零开销。动态管线（Hades）运行时Build一次，整轮复用。

| 来源 | 方式 | 适用场景 |
|------|------|---------|
| **编辑器配置** | 设计者选DamageRule → Build → 保存 | 只狼等固定管线 |
| **运行时动态生成** | 按激活Boon组装 → Build → 复用 | Hades等肉鸽 |

#### 稳定拓扑排序

在满足依赖关系的前提下，保留DamageRule在数组中的原始顺序（Kahn BFS变体，候选队列按原始下标排序）。

| 问题 | 标准拓扑排序 | 稳定拓扑排序 |
|------|------------|------------|
| 未声明的隐含依赖 | 可能排错 | 数组顺序兜底 |
| Build确定性 | 每次可能不同 | 同输入同输出 |
| 使用门槛 | 必须完美声明 | 逐步补全即可 |

### 2.2 视图层：编辑器模块

| 功能 | 说明 |
|------|------|
| **DamageRule选择** | 从项目中所有DamageOperation子类中选择 |
| **DamageCondition编辑** | 属性面板内联——选DamageConditionNode子类（类名=DamageRule名）→ 选领域方法（UFUNCTION反射自动填充） |
| **Produces编辑** | 选择DamageEffect类型（UScriptStruct*，支持C++和蓝图结构体） |
| **Build** | 调用DamagePipeline::Build()，烘焙排序结果保存到Asset |
| **Preview** | 用测试数据运行管线，高亮执行/跳过状态 |

## 三、DamageEffect与DamageCondition的实现要点

### 3.1 DC存储

DC使用`TMap<FName, FInstancedStruct>`存储DamageEffect——**DamageRule身份标识（FName）**作为索引。FInstancedStruct支持C++和蓝图结构体。

DC提供两条读写路径：C++模板路径（编译时类型安全）和通用FInstancedStruct路径（蓝图/反射）。

### 3.2 DamageCondition条件树

条件树由两类节点组成：

| 类别 | 节点 | 职责 | UClass标记 |
|------|------|------|-----------|
| **基类** | UDamageConditionNode | EvaluateCondition（处理bReverse取反）+ GetConsumedRules（递归收集依赖） | Abstract, DefaultToInstanced, EditInlineNew, Blueprintable |
| **复合节点** | AND / OR | 组合子节点的布尔逻辑 | 子类 |
| **叶子节点** | 按DamageRule分的DamageConditionNode子类 | 绑定DamageEffect类型，承载领域方法（UFUNCTION） | 子类 |

**领域方法直接定义在DamageConditionNode子类上**——UFUNCTION，方法列表通过UE5反射自动发现。不需要全局注册表。

**蓝图扩展**：蓝图继承UDamageConditionNode，蓝图函数天然是UFunction，反射路径和C++一致。

### 3.3 DamageConditionNode的方法下拉

用户选DamageConditionNode子类后，方法下拉自动联动：

```
用户选 ConditionNode类型 = DamageConditionNode_Guard
→ 绑定的DamageEffect类型为FGuardEffect
→ UE5反射发现该子类上的UFUNCTION列表 ["IsParry", "IsGuardBreak"]
→ Method下拉显示
```

### 3.4 产销关系收集

```
DamageConditionNode_Guard.GetConsumedRules()
  → 绑定FGuardEffect → 到DamageRule池查找谁produces FGuardEffect → return ["Guard"]

AND/OR.GetConsumedRules()
  → 递归所有Children → 合并去重
```

### 3.5 新建一套DamageRule的工作流

```
1. 新建DamageEffect结构体（纯数据字段）
2. 新建DamageConditionNode子类（绑定DamageEffect类型，定义领域方法UFUNCTION）
3. 新建DamageRule定义（Produces=DamageEffect类型，OperationClass，DamageCondition条件树）

三者通过DamageEffect类型建立联系。
```

## 四、分步实施计划

### Phase 1：运行时系统

**目标**：跑通只狼V3.5的15个DamageRule管线。

**交付物**：

| 组件 | 类名 | 职责 |
|------|------|------|
| DC | `UDamageContext` | TMap<FName, FInstancedStruct>存储DamageEffect，DamageRule身份标识为索引 |
| DamageEffect结构体 | 各USTRUCT | 各DamageRule的纯数据产出 |
| DamageRule定义 | `UDamageRuleDefinition` (DataAsset) | RuleName、Produces、DamageCondition、OperationClass |
| DamageOperation基类 | `UDamageOperationBase` | 子类重写Execute——读DC、返回DamageEffect。不负责写入DC。 |
| DamageCondition条件树 | `UDamageConditionNode` + 子类 | AND/OR + 按DamageRule分的叶子节点（UFUNCTION领域方法），递归GetConsumedRules() |
| DamagePipeline | `UDamagePipeline` (DataAsset) | 持有DamageRule列表 + Build + Execute，自洽执行 |

**验证场景**：

| # | 场景 | 验证重点 |
|---|------|---------|
| 1 | 普通斩击 | 基础执行流程 |
| 2 | 弹刀 | DamageConditionNode子类引用领域方法 |
| 3 | 格挡+破防 | 多DamageRule依赖链 |
| 4 | 雷电+死亡 | 多DamageConditionNode组合 |
| 5 | 空中接雷 | DamageEffect缺失容错 |

### Phase 2：编辑器模块

**前提**：Phase 1稳定。

| 组件 | 说明 |
|------|------|
| DamagePipeline编辑器 | 自定义Asset Editor |
| DAG自动布局 | 从DamageRule间依赖关系自动推导 |
| Build/Preview | 调用DamagePipeline::Build()/Execute() |
| ConditionNode方法下拉 | UFUNCTION反射自动填充 |
| 运行时调试面板 | 实时DAG + DC监视 + 快照导出 |

## 五、关键设计约束

### 5.1 编辑器和运行时共用DamagePipeline的Build/Execute

不存在独立的Manager或两套逻辑。

### 5.2 DamagePipeline是唯一数据交换格式

编辑器保存的和运行时加载的是同一种Asset。

### 5.3 编辑器不存在于运行时

Runtime Module不依赖Editor Module。

### 5.4 两种管线模式

| 模式 | DamageRule子集何时确定 | 典型游戏 |
|------|---------------------|---------|
| **固定管线** | 设计时Build | 只狼、鬼泣 |
| **动态管线** | 运行时每轮 | Hades |

### 5.5 领域方法由DamageConditionNode子类承载

UFUNCTION定义，反射自动发现。C++和蓝图路径一致。不需要全局注册表。

## 六、与模型规范v4.7的映射

| 模型规范概念 | 实现层对应 |
|------------|----------|
| DamageRule（伤害规则） | UDamageRuleDefinition (DataAsset) + UDamageOperationBase (逻辑子类) |
| DamageRule身份标识 | UDamageRuleDefinition::RuleName (FName) |
| DamageEffect（规则产出） | UStruct（纯数据结构，支持C++和蓝图） |
| DamageEffect实例 | FInstancedStruct，以RuleName为索引存储在DC中 |
| DamageConditionNode子类 | 按DamageRule分的UDamageConditionNode子类，承载领域方法（UFUNCTION） |
| 领域方法 | DamageConditionNode子类上的UFUNCTION，反射自动发现 |
| DC | UDamageContext — TMap<FName, FInstancedStruct> |
| DamageCondition条件树 | 复合节点（AND/OR）+叶子节点（DamageConditionNode子类） |
| Produces | UDamageRuleDefinition::Produces — UScriptStruct* |
| Consumes | DamageConditionNode子类通过DamageEffect类型反查DamageRule → GetConsumedRules() |
| 拓扑排序 | UDamagePipeline::Build() — 稳定Kahn算法 |
| R3缺失容错 | DamageEffect不存在 → DamageConditionNode评估返回false |
| DamagePipeline | UDamagePipeline — 持有DamageRule列表 + Build() + Execute()，自洽执行 |
| DamageOperation | UDamageOperationBase子类 — 读DC、返回DamageEffect |
