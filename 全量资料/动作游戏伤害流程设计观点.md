# 动作游戏伤害流程设计观点

## 对话

### 动作游戏的伤害流程，不是以数值计算为核心，而是以「服务核心玩法机制」为核心；伤害是机制的表达，不是目的。

- 忍龙：伤害流程 = 维持高速战斗节奏的工具

- 鬼泣：伤害流程 = 连招与控制体系的载体

- 只狼：伤害流程 = 拼刀机制与世界观的统一表达

### 动作游戏的伤害流程，是以数值计算为基础、以多机制整合为核心、以清晰即时反馈为表现的整体战斗表达

- 1. 以数值计算为基础 —— 不是不算数，只是不唯数值

- 2. 以多机制整合为核心 —— 服务动作核心机制、为玩法服务

- 3. 以清晰即时反馈为表现 —— 本质就是战斗反馈

### 机制优先，数值兜底，反馈即时，博弈可感知

### 动作游戏的伤害流程，本质是核心玩法机制（攻击 / 防御 / 资源管理 / 即时反馈）的整合与具象表达，而非以数值计算为核心的独立系统；数值仅作为机制平衡的底层基础（而非战斗决策的核心依据），机制整合的流畅性与反馈的即时性才是核心，最终服务于动作游戏 “即时反馈 - 策略博弈 - 操作爽感” 的核心战斗体验。

### 机制优先，数值兜底，反馈清晰可感

### 反馈清晰可感  避免信息过载

### 玩法修饰

- 恐怖机制是为战斗添加挑战限制（要求完美弹刀）避免恐怖槽累计

## 从设计视角构建动作游戏模型

### 总结

- 数值游戏：伤害流程是连续、线性、可预测的，玩家决策只影响“数值走在这条线的哪个点”

	- 数值游戏 = 同一条流程上的数值微调。

- 动作游戏：伤害流程是离散、分支、可选择的，玩家决策直接决定走哪一条伤害链路，每条链路反馈完全不同。

	- 动作游戏 = 伤害流程的拓扑结构选择

		- 动作游戏的伤害流程，是一个以玩家实时操作为输入、以离散可读的多模态反馈为输出的实时决策状态机；数值计算存在但被压到感知阈值以下，玩家的战斗策略本质上是在这个状态机中选择和组合伤害链路以达成胜利条件。

### 动作游戏的“伤害流”不是数学，是状态机

- 战斗 = 状态机跳转

- 每一条流，都是完全独立的子系统。

- 不是线性数值，是分支路径。

### “明确反馈”，是动作游戏成立的必要条件

- 反馈 = 告诉玩家现在走哪条伤害流

### 数值游戏是“解方程”，动作游戏是“选方程”

- 数值游戏

	- 玩家做的是把系数调到最大

- 动作游戏

	- 伤害 = 你选哪个函数

- 数值游戏是刻度不同；动作游戏是尺子本身就不一样

### 3 个经典游戏直接印证你的理论

- 《只狼》——最完美印证你的模型

	- 选择用什么行为逼对方进入哪条崩溃流程。

	- 不同策略 → 走不同伤害流 → 不同反馈 → 不同结局。

- 《怪物猎人》——部位破坏就是“多流伤害”

	- 每种部位破坏都是独立伤害流，不是单纯扣血。

	- 玩家决策 = 我今天要走哪条破坏流。

- 《黑暗之魂》

	- 精力空 → 走崩防流

	- 盾击 → 走硬直流

	- 背刺/弹反 → 走处决流

### 可对外讲的专业模型

- 动作游戏伤害流程 = 多分支离散状态机

	- 玩家通过操作/策略，选择进入不同伤害子流程（破甲、破躯干、破防、处决、反击等），每条流拥有独立判定、独立反馈、独立结局。

- 数值游戏伤害流程 = 单分支线性公式

	- 玩家只调整这条线上的数值系数，不改变路径本身。

## 解析动作游戏战斗架构的核心挑战

### 架构师视角 + 工程现实

- 用「解耦的架构」，去实现「高度耦合的玩法」，这是一个天然矛盾，也是顶级引擎架构师和战斗系统架构师的核心价值

### 矛盾的本质：玩法要耦合，代码必须解耦。

- 玩法层面 ： 全都要互相影响、互相咬合，形成一张网。

- 解耦架构 ：模块化、可扩展、可维护、可测试、多人并行开发

- 矛盾呈现：伤害、弹反、硬直、状态、AI、玩家策略 相互影响

### 真正的架构解法：你说的「管线 + 模块化 + 解耦」

- 中央战斗事件总线 / 消息机制

	- 发事件 + 监听事件。

		- 各模块 订阅这个事件，各自处理

	- 代码解耦，但玩法天然耦合

- 伤害管线（Damage Pipeline）

	- 伤害流程拆成一串可插拔节点

		- 每一步都是独立模块

		- 串起来就是你说的不同伤害流

		- 架构上解耦，玩法上耦合。

- 状态机 + 上下文（Context）

	- 所有行为依赖一个只读的战斗上下文，而不是互相依赖。

	- 只读Context，不改别人的内部

- 数据驱动，而非逻辑驱动

	- 策划可以拼玩法耦合

	- 程序只需要维护模块稳定

### 一句话总结

- 动作游戏战斗架构的终极目标：让代码保持树状、分层、解耦；让玩法长成网状、联动、高耦合。

	- 代码：干净、稳定、可扩展

	- 玩法：复杂、联动、有深度

- 这就是最大的挑战。也是区分普通游戏和神作动作游戏的工程底线

### 逻辑链闭环

- 玩家体验 → 伤害流 → 反馈易用性 → 网状机制 → 架构挑战

- 体验上：动作游戏是离散多流伤害 + 明确反馈

- 设计上：是主机制为骨架 + 副机制织网 = 高耦合玩法

- 工程上：必须用解耦、模块化、管线化架构去承载

- 矛盾和挑战

	- 玩法要耦合，代码必须解耦

		- 这就是架构师真正要解决的问题

## 深度解析：Hades 的动态伤害流程与工程杰作

### 架构师 + 战斗策划

- Hades 确实是：用极致解耦的架构，做出了极致耦合的玩法，并且把伤害流程做成了「动态可拼装流水线」

### 「动态拼装伤害流程」，就是 Hades 最牛的地方

- Hades 的伤害流程不是写死的，而是 runtime 动态拼出来的：

- 每一局都是全新的伤害管线。

- 每一个机制、每一段伤害逻辑、每一种效果，都必须是独立可插拔模块。

- 解耦到极致，才能在运行时随便拼耦合玩法。

### 动态拼装伤害流程。

- 伤害流程 = 事件链条 + 修饰器链

- 任何祝福都能插在伤害流程的任意节点

- 任何状态都能修改任意一段行为

- 所有系统不互相依赖，只依赖 总线 + 上下文

- 每一个“可插入”，都是一个独立模块

- 玩家每一局拿到的祝福，就是在往这条流水线上插零件。

- 解决顶级矛盾：代码解耦到底 → 玩法耦合到炸

### 动作游戏伤害流程 = 多流、离散、明确反馈

- 每一条流反馈极强、清晰、易用。

- Usability + 明确反馈。

### 主机制为骨架，副机制织成网，1+1+1>10

- 主机制：武器 + 闪避 + 技能

- 副机制：诸神祝福、状态、连击、信物、房间

- 没有一个机制是孤立的。

- 网状有机整体、耦合、化学反应

### 用解耦架构，实现高耦合玩法

- Roguelike + 动作战斗 + 海量Build

## 极简内核·强耦合·化学反应：一套动作游戏的顶级审美

### 极其干净、极其高级、可量化、可评判的动作游戏审美标准

- 极简内核

	- 只存在一个真正的核心机制，其余系统都为它而生

- 强耦合

	- 动作、资源、敌人、场景、招式，不是拼凑，而是咬合

	- 你不能只玩一部分，必须把整套逻辑用满。

	- 没有“可选玩法”，只有“唯一最优解”。

- 化学反应

	- 机制之间不是加法，是化合。

	- 1+1 > 2，两种机制联动，效果质变。

	- 1+1+1 > 10，多机制合一，出现终极爆发。

	- 玩法深度，来自组合的可能性，而不是系统的数量。

### 用这套审美重新评价NGZ

-  内核判断：它有极简内核，而且非常纯粹

	- 三元素化合反应系统

		- 一切动作，服务于“获取元素”

		- 一切战斗，服务于“触发反应”

		- 一切敌人，服务于“让反应有意义”

- 耦合判断：它想做强耦合，但没做到“死锁”

	- 好的强耦合：不用核心机制，你就打不下去。

	- 结果：内核在那里，但玩家可以绕开它。

- 化学反应：设计上做到了，但体验上没炸

### 少而精、一核生万法、咬合紧密、化合爆炸

- 只狼、忍龙2、鬼泣、魔女：神作，因为三项全满

- NGZ：有魂无体，内核对，成品弱

- 新战神、仁王2、浪人崛起：系统多、耦合弱、大杂烩
